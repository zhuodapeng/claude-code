# 11-2 权限规则的数据结构

> 用户在 settings.json 里写的 `"Bash(npm install)"`，代码里长什么样？它存在哪里？怎么被查询？本节追踪从配置字符串到运行时对象的完整过程。

---

## 问题引入：一条规则要表达什么

一条权限规则至少要回答三个问题：
1. **针对哪个工具？**（Bash？Write？某个 MCP 工具？）
2. **效果是什么？**（允许、拒绝，还是强制弹框？）
3. **来自哪里？**（用户的个人设置、项目设置、企业策略……）

第三个问题很容易被忽略，但它决定了优先级——企业管理员的规则应该比用户个人规则的优先级更高。

---

## 朴素方案：一个字符串不够

最简单的方案是用字符串数组存规则，比如：

```json
{ "allow": ["Bash", "Read"], "deny": ["Write"] }
```

这看起来够用了。但有几个问题：
- `"Bash"` 是允许所有 Bash 命令，还是允许某个特定命令？
- 我怎么知道这条规则是用户手动加的，还是命令行参数带进来的？
- 规则之间有优先级时，如何判断谁赢？

所以需要一个更结构化的类型。

---

## PermissionRule：三字段类型

所有权限规则的核心类型定义在 [src/types/permissions.ts](../src/types/permissions.ts#L75-L79)：

**文件：[src/types/permissions.ts](../src/types/permissions.ts#L67-L79)**

```typescript
/**
 * The value of a permission rule - specifies which tool and optional content
 */
export type PermissionRuleValue = {
  toolName: string       // 工具名称（必填）
  ruleContent?: string   // 匹配内容（可选）
}

/**
 * A permission rule with its source and behavior
 */
export type PermissionRule = {
  source: PermissionRuleSource    // 规则来自哪里
  ruleBehavior: PermissionBehavior // 效果：allow / deny / ask
  ruleValue: PermissionRuleValue  // 匹配什么
}
```

三个字段，三个问题，一一对应：
- `ruleValue`：匹配什么（工具 + 可选内容）
- `ruleBehavior`：效果是什么
- `source`：来自哪里（决定优先级）

---

## PermissionBehavior：三值穷举

**文件：[src/types/permissions.ts](../src/types/permissions.ts#L44)**

```typescript
export type PermissionBehavior = 'allow' | 'deny' | 'ask'
```

这三个值中，`'ask'` 最容易被误解。

- `'allow'`：工具直接执行，不问用户
- `'deny'`：工具被阻止，不问用户
- `'ask'`：**强制弹出确认框，哪怕其他条件允许也要弹**

`'ask'` 规则和"没有规则时默认要问"不同：没有规则是系统的缺省行为；`'ask'` 规则是用户主动配置"**我要每次都看到这个确认框**"——常用于生产环境的高风险操作。

---

## PermissionRuleSource：八种来源

**文件：[src/types/permissions.ts](../src/types/permissions.ts#L54-L62)**

```typescript
export type PermissionRuleSource =
  | 'userSettings'    // ~/.claude/settings.json（跨项目个人设置）
  | 'projectSettings' // .claude/settings.json（项目级，可提交 git）
  | 'localSettings'   // .claude/settings.local.json（项目级，不提交 git）
  | 'flagSettings'    // 命令行 flag 指定的设置文件
  | 'policySettings'  // 企业托管策略（只读，优先级最高）
  | 'cliArg'          // --allowedTools / --disallowedTools 参数
  | 'command'         // /permissions 命令运行时修改的
  | 'session'         // 当前会话临时添加（不持久化）
```

优先级关系（高 → 低）：

```
policySettings  → 企业管理员，最高，只读
flagSettings    → 本次启动的 --config flag
cliArg          → 本次启动的 --allowedTools / --disallowedTools
localSettings   → 项目本地个人覆盖
projectSettings → 项目共享规则
userSettings    → 个人全局设置
command         → /permissions 命令临时修改
session         → 最临时，仅此次对话
```

这个分层设计解决了一个实际问题：**企业可以禁止某些工具，但用户可以在个人层面追加允许的项目级命令**——各层相互独立，互不覆盖，而是叠加生效。

## 字符串序列化：配置文件里的样子

`PermissionRule` 是运行时对象，但用户写的是字符串。序列化/反序列化由 [src/utils/permissions/permissionRuleParser.ts](../src/utils/permissions/permissionRuleParser.ts) 完成。

格式规则很简单：`ToolName` 或 `ToolName(content)`

**文件：[src/utils/permissions/permissionRuleParser.ts](../src/utils/permissions/permissionRuleParser.ts#L93-L133)**

```typescript
export function permissionRuleValueFromString(
  ruleString: string,
): PermissionRuleValue {
  // 找到第一个未被转义的 '('
  const openParenIndex = findFirstUnescapedChar(ruleString, '(')
  if (openParenIndex === -1) {
    // 没有括号 → 纯工具名
    return { toolName: normalizeLegacyToolName(ruleString) }
  }

  const toolName = ruleString.substring(0, openParenIndex)
  const rawContent = ruleString.substring(openParenIndex + 1, closeParenIndex)

  // 空内容 "Bash()" 或通配符 "Bash(*)" 等效于工具级规则
  if (rawContent === '' || rawContent === '*') {
    return { toolName: normalizeLegacyToolName(toolName) }
  }

  // 反转义内容（\( → (, \) → )）
  const ruleContent = unescapeRuleContent(rawContent)
  return { toolName: normalizeLegacyToolName(toolName), ruleContent }
}
```

实际转换示例：

```
"Bash"              → { toolName: 'Bash' }                    // 工具级规则
"Bash(npm install)" → { toolName: 'Bash', ruleContent: 'npm install' }
"Bash(*)"           → { toolName: 'Bash' }                    // 等效工具级
"Bash()"            → { toolName: 'Bash' }                    // 等效工具级
```

---

## 括号转义：为什么需要它

规则内容本身可能含括号。例如 Bash 命令 `python -c "print(1)"` 里有括号，存成字符串会是 `Bash(python -c "print(1)")`——解析器会把内层的 `(1)` 误解成格式结构。

**文件：[src/utils/permissions/permissionRuleParser.ts](../src/utils/permissions/permissionRuleParser.ts#L55-L60)**

```typescript
export function escapeRuleContent(content: string): string {
  return content
    .replace(/\\/g, '\\\\')  // 先转义反斜杠（顺序很重要！）
    .replace(/\(/g, '\\(')   // 再转义左括号
    .replace(/\)/g, '\\)')   // 再转义右括号
}
```

存储后的样子：`Bash(python -c "print\(1\)")`

反转义顺序必须相反：先还原括号，再还原反斜杠（[第 74 行](../src/utils/permissions/permissionRuleParser.ts#L74-L79)）。如果顺序搞反，`\\(` 会先被还原成 `\(`，再被错误地还原成 `(`。这是个容易踩的坑。

## 旧工具名的迁移

注意 `permissionRuleValueFromString` 里调用了 `normalizeLegacyToolName`：

**文件：[src/utils/permissions/permissionRuleParser.ts](../src/utils/permissions/permissionRuleParser.ts#L21-L29)**

```typescript
const LEGACY_TOOL_NAME_ALIASES: Record<string, string> = {
  Task: AGENT_TOOL_NAME,              // "Task" → "Agent"
  KillShell: TASK_STOP_TOOL_NAME,
  AgentOutputTool: TASK_OUTPUT_TOOL_NAME,
  BashOutputTool: TASK_OUTPUT_TOOL_NAME,
  // ...
}

export function normalizeLegacyToolName(name: string): string {
  return LEGACY_TOOL_NAME_ALIASES[name] ?? name
}
```

工具改名时不需要迁移用户的配置文件——解析时自动转换。这是向后兼容的实现方式：新代码读老配置不会出错。

## ToolPermissionContext：运行时的规则容器

规则加载完后，被放入 `ToolPermissionContext`——这是权限判断的运行时快照：

**文件：[src/types/permissions.ts](../src/types/permissions.ts#L427-L441)**

```typescript
export type ToolPermissionContext = {
  readonly mode: PermissionMode                    // 当前权限模式
  readonly alwaysAllowRules: ToolPermissionRulesBySource   // allow 规则集
  readonly alwaysDenyRules: ToolPermissionRulesBySource    // deny 规则集
  readonly alwaysAskRules: ToolPermissionRulesBySource     // ask 规则集
  readonly additionalWorkingDirectories: ReadonlyMap<...>
  readonly isBypassPermissionsModeAvailable: boolean       // YOLO 模式可用？
  readonly shouldAvoidPermissionPrompts?: boolean          // 无头模式？
  readonly awaitAutomatedChecksBeforeDialog?: boolean
  readonly prePlanMode?: PermissionMode
}
```

其中规则按来源分组存储：

**文件：[src/types/permissions.ts](../src/types/permissions.ts#L419-L421)**

```typescript
export type ToolPermissionRulesBySource = {
  [T in PermissionRuleSource]?: string[]  // 存的是字符串，不是对象
}
```

注意：**存的是字符串数组，不是 `PermissionRule` 对象**。在查询时按需调用 `permissionRuleValueFromString()` 还原。这是一个延迟解析的优化——每次工具调用只解析当前相关的规则，而不是在初始化时解析全部。

## 查询规则：三个工具函数

权限判断函数通过三个辅助函数查询规则集：

**文件：[src/utils/permissions/permissions.ts](../src/utils/permissions/permissions.ts#L122-L231)**

```typescript
// 获取所有 allow 规则（展平所有来源）
export function getAllowRules(context: ToolPermissionContext): PermissionRule[] {
  return PERMISSION_RULE_SOURCES.flatMap(source =>
    (context.alwaysAllowRules[source] || []).map(ruleString => ({
      source,
      ruleBehavior: 'allow',
      ruleValue: permissionRuleValueFromString(ruleString),
    })),
  )
}

// deny 规则同理
export function getDenyRules(context: ToolPermissionContext): PermissionRule[] { ... }

// ask 规则同理
export function getAskRules(context: ToolPermissionContext): PermissionRule[] { ... }
```

每次调用都会重新解析字符串。代价是每次权限判断要做字符串解析，但规则数量通常很少（十几条），性能影响可以忽略。

还有两个专门用于匹配整个工具的函数（不匹配内容级规则）：

**文件：[src/utils/permissions/permissions.ts](../src/utils/permissions/permissions.ts#L275-L301)**

```typescript
// 找到整个工具级别的 allow 规则（没有 ruleContent 的那种）
export function toolAlwaysAllowedRule(
  context: ToolPermissionContext,
  tool: Pick<Tool, 'name' | 'mcpInfo'>,
): PermissionRule | null {
  return getAllowRules(context).find(rule => toolMatchesRule(tool, rule)) || null
}

// 找到整个工具级别的 deny 规则
export function getDenyRuleForTool(context, tool): PermissionRule | null { ... }

// 找到整个工具级别的 ask 规则
export function getAskRuleForTool(context, tool): PermissionRule | null { ... }
```

`toolMatchesRule` 函数（[第 238 行](../src/utils/permissions/permissions.ts#L238-L268)）只在没有 `ruleContent` 时才匹配——`Bash(npm install)` 不会被 `getDenyRuleForTool` 命中，它要在工具的 `checkPermissions()` 里处理。

## 本节小结

`PermissionRule` 是三字段结构（source + ruleBehavior + ruleValue），配置文件里存字符串，运行时按需解析。`ToolPermissionContext` 是所有规则按来源组织的快照，是权限判断时的唯一信息来源。规则的来源决定优先级，`policySettings` 最高，`session` 最低。
