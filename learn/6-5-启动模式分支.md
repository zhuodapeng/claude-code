# 6-5 启动模式分支：TUI / Print / SDK 三条路

> **本节目标**：Claude Code 不只有一种运行方式。`claude-haha` 交互启动是一种，`claude-haha -p "..." ` 管道输出是另一种，SDK 调用又是一种。这三条路在哪里分叉？它们各自的初始化有什么不同？这些差异背后的工程权衡是什么？

---

## 为什么需要多种运行模式

当一个工具既要给人用（交互界面），又要给程序用（被脚本调用），它面临一个经典矛盾：

- **交互模式**需要 TUI——颜色、光标、全屏刷新
- **管道模式**不能有 TUI——颜色代码会破坏 `grep`，光标控制会乱码，全屏刷新根本没意义

bash 里的 `ls` 命令就是这样解决的：有终端就加颜色，被管道重定向就输出纯文本。

Claude Code 更复杂——它不只有"有没有颜色"的区别，而是整个运行架构都不同：
- 交互模式：React 组件树 + Ink 渲染 + 状态管理
- Print 模式：无 UI + 直接 stdout 输出 + 简化状态

---

## 分叉点

分叉在 `main.tsx` 的 action handler 里，具体是根据 `getIsNonInteractiveSession()` 的返回值：

**文件：[src/main.tsx](../src/main.tsx#L1334)**

```typescript
const isNonInteractiveSession = getIsNonInteractiveSession()
```

什么情况下是 `NonInteractiveSession`？

- 用户传了 `-p` 或 `--print`
- stdout 不是终端（被重定向到文件或管道）
- 某些 SDK 调用模式

这个判断在初始化阶段就完成了，之后整个启动流程分成两条截然不同的路。

---

## 路径 A：交互模式（TUI 路径）

当 `isNonInteractiveSession === false` 时：

**文件：[src/main.tsx](../src/main.tsx#L2224-L2248)**

```typescript
if (!isNonInteractiveSession) {
  // 1. 创建渲染上下文（FPS 计数器、stats 收集器）
  const ctx = getRenderContext(false)
  getFpsMetrics = ctx.getFpsMetrics
  stats = ctx.stats
  
  // 2. 创建 Ink 根节点（这才是 TUI 的"画布"）
  const { createRoot } = await import('./ink.js')
  root = await createRoot(ctx.renderOptions)
  
  // 3. 记录启动时间（在信任对话框弹出之前）
  logEvent('tengu_timer', { event: 'startup', durationMs: ... })
  
  // 4. 显示设置屏幕（信任对话框、OAuth、向导）
  const onboardingShown = await showSetupScreens(root, permissionMode, ...)
}
```

这里有个值得注意的细节：**启动时间在 `showSetupScreens()` 之前记录**。

原因是注释里写的：之前在 REPL 第一次渲染时记录，导致 p99 启动时间是 ~70 秒——因为用户可能在信任对话框上停留几十秒。把记录点挪到对话框之前，才能得到真实的代码路径性能数据。

### AppState：交互模式的状态容器

进入 TUI 路径后，状态通过 `AppStateProvider` 管理（第 4 章讲的外部 store 模式）。初始状态在 `main.tsx` 里构建，传给 `<REPL>` 组件的 `initialState` 参数。

---

## 路径 B：Print 模式（无 UI 路径）

当 `isNonInteractiveSession === true` 时：

**文件：[src/main.tsx](../src/main.tsx#L2568-L2659)**

```typescript
if (print) {
  // 1. 应用完整的环境变量（信任是隐式的）
  applyConfigEnvironmentVariables()
  
  // 2. 提前启动昂贵的并行预取
  void getSystemContext()
  void getUserContext()
  void ensureModelStringsInitialized()
  
  // 3. 创建"无头"状态容器（不需要 React 组件，直接是 JS 对象）
  const headlessInitialState: AppState = {
    ...getDefaultAppState(),
    mcp: { clients: mcpClients, tools: mcpTools, commands: mcpCommands },
    toolPermissionContext,
    // ...
  }
  const headlessStore = createStore(headlessInitialState, onChangeAppState)
  
  // 4. 没有 createRoot()！没有 Ink！
  // 5. 走 print.ts 的 loadInitialMessages() 执行对话
}
```

关键差异：

| 交互模式 | Print 模式 |
|---------|-----------|
| `createRoot()` → Ink TUI | 直接 `createStore()` |
| React 组件树渲染 | 纯 JS 函数调用链 |
| 用户信任对话框 | 隐式信任，跳过 |
| `root.waitUntilExit()` 阻塞 | `loadInitialMessages()` 运行到完成 |
| 颜色 + 格式化输出 | 文本/JSON 到 stdout |

Print 模式为什么不需要 React？

因为 React（+Ink）本质上是一个**状态驱动的渲染引擎**——它响应状态变化，把状态映射成终端字符。

在 Print 模式里，不需要"渲染"，只需要"执行"——调用 API，输出结果，结束。状态变化不需要触发屏幕刷新，只需要影响最终输出内容。

---

## 路径 C：SDK 模式

这是通过 `--output-format stream-json` 或 `--sdk-url` 触发的路径，主要用于程序化调用。

```bash
# 通过 Agent SDK 调用 Claude Code
claude-haha -p "帮我分析这个函数" --output-format stream-json
```

**文件：[src/main.tsx](../src/main.tsx#L1235-L1252)**

```typescript
// 如果提供了 SDK URL，自动切换格式
if (sdkUrl) {
  if (!inputFormat) inputFormat = 'stream-json'
  if (!outputFormat) outputFormat = 'stream-json'
  if (options.verbose === undefined) verbose = true
  if (!options.print) print = true
}
```

SDK 模式本质上是 Print 模式的超集：
- 走和 Print 模式相同的无 UI 路径
- 但输出格式是结构化的 JSON 流
- 每个事件（工具调用开始、文本块、工具结果）都作为 JSON 行输出

这让 Agent SDK 可以消费这个输出流，知道 Claude 在做什么。

---

## 三条路的最终汇聚

这三条路**在主循环（queryLoop）里汇聚**。

无论是交互模式还是 Print 模式，最终都会调用同一个 `queryLoop`——它是 Claude 发 API 请求、处理工具调用、管理对话历史的核心循环。

区别只在于：
- 交互模式：`queryLoop` 的输出通过 React 状态更新传递给 TUI 渲染
- Print 模式：`queryLoop` 的输出直接写到 stdout

```
                    ┌─── 交互模式 ───┐
                    │  React + Ink   │
用户输入            │  UI 渲染        │
    ↓               │                │
┌─────────┐         └────────────────┘
│ query   │
│  Loop   │─────── queryLoop (共用)
└─────────┘
    ↑               ┌────────────────┐
Claude API          │  Print 模式    │
                    │  直接 stdout   │
                    │  流式输出       │
                    └────────────────┘
```

---

## 路径选择的工程权衡

你可能会想：为什么不只做一个路径，Print 模式就用"无 UI 的 TUI"？

React 是有开销的：
1. **内存**：React 虚拟 DOM + Ink 的 Yoga 布局引擎常驻内存
2. **CPU**：每次状态变化都触发 reconciliation（协调算法）
3. **依赖**：Ink 依赖终端是 TTY，非 TTY 环境会报错

Print 模式里用 React 是"把杀鸡刀用来杀蚊子"——不必要的复杂性。

反过来，Print 模式里的"无头 store"方案更轻量：直接是一个 JavaScript 对象 + 一些监听函数，没有 VDOM，没有 layout engine，只有业务逻辑。

---

## 本节小结

- `isNonInteractiveSession` 是分叉判断条件，决定走 TUI 路径还是 Print 路径
- TUI 路径：`createRoot()` → `showSetupScreens()` → `renderAndRun(<REPL>)`
- Print 路径：跳过 UI，直接 `createStore()` → `loadInitialMessages()`，无头运行
- SDK 模式是 Print 路径 + stream-json 输出格式的组合
- 三条路最终都汇聚到同一个 `queryLoop`
