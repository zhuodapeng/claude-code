# 5-3 工具调用（Tool Use）的完整协议

> **本节目标**：彻底理解 LLM 工具调用的完整协议——工具如何向 API 注册、Claude 如何决定调用工具、调用结果如何发回给 Claude、以及多工具调用的处理。这是整个 Tool System（第 7 章）的理论基础。

---

## 问题的起点：LLM 怎么"做事"？

前面说过，LLM 本质上是一个**文本转文本**的函数：给它文字，它输出文字。

但 Claude Code 需要 Claude 能**读文件、执行命令、搜索代码**——这些都是真实的操作，不是文字。

**解法**：设计一个协议，让 Claude 在回复里"请求"执行某个操作，由程序（Claude Code）真正执行，再把结果告诉 Claude。

这就是**工具调用（Tool Use / Function Calling）协议**。

---

## 工具调用的完整往返流程

```
轮次 1：
  
  你 → API：
    "帮我列出当前目录的文件"
    （附带工具定义：BashTool 能执行命令）
  
  API → 你（Claude 的回复）：
    "我来执行 ls 命令来查看目录内容。"
    [工具调用请求] tool_use: BashTool, input: {"command": "ls -la"}
    （stop_reason: "tool_use"）

轮次 2：

  你 → API：
    [之前的历史] + [工具执行结果]
    tool_result: BashTool, content: "README.md\npackage.json\nsrc/\n..."
  
  API → 你（Claude 的回复）：
    "当前目录包含以下文件：
    - README.md：项目说明
    - package.json：依赖配置
    - src/：源码目录
    ..."
    （stop_reason: "end_turn"，正常结束）
```

**整个协议本质上是：在对话里插入"工具调用请求"和"工具调用结果"两种特殊消息。**

---

## 第一步：注册工具（向 API 描述工具）

在 API 请求里，你需要告诉 Claude 有哪些工具可用：

```json
{
  "model": "claude-sonnet-4-6",
  "messages": [...],
  "tools": [
    {
      "name": "bash",
      "description": "执行 bash 命令，返回 stdout 和 stderr",
      "input_schema": {
        "type": "object",
        "properties": {
          "command": {
            "type": "string",
            "description": "要执行的命令"
          },
          "timeout": {
            "type": "number",
            "description": "超时时间（毫秒），可选"
          }
        },
        "required": ["command"]
      }
    },
    {
      "name": "read_file",
      "description": "读取文件内容",
      "input_schema": {
        "type": "object",
        "properties": {
          "path": { "type": "string", "description": "文件路径" }
        },
        "required": ["path"]
      }
    }
  ]
}
```

`input_schema` 是 **JSON Schema** 格式，这正是为什么 Claude Code 用 Zod 定义工具的输入——Zod schema 可以自动转换为 JSON Schema。

**`description` 字段极其重要**：Claude 通过它理解工具是干什么的、什么时候该用。写得清楚，Claude 才能正确选择和使用工具。

---

## 第二步：Claude 决定调用工具

Claude 在生成回复时，如果判断需要工具，会在回复里包含 `tool_use` 内容块：

```json
{
  "id": "msg_abc",
  "type": "message",
  "role": "assistant",
  "content": [
    {
      "type": "text",
      "text": "让我执行 ls 命令来查看目录内容。"
    },
    {
      "type": "tool_use",
      "id": "toolu_01234",
      "name": "bash",
      "input": {
        "command": "ls -la",
        "timeout": 5000
      }
    }
  ],
  "stop_reason": "tool_use"
}
```

注意 `stop_reason: "tool_use"`——这告诉你"Claude 停下来是因为它想用工具，而不是因为回答完了"。

一个关键细节：**`tool_use` 块有一个唯一的 `id`**（`toolu_01234`）。这个 id 在后面返回工具结果时要用到，让 Claude 知道这个结果对应哪个工具调用。

---

## 第三步：执行工具，返回结果

程序收到 `tool_use` 请求后，执行实际操作，然后发起新的 API 请求，在消息里包含工具结果：

```json
{
  "model": "claude-sonnet-4-6",
  "messages": [
    {
      "role": "user",
      "content": "帮我列出当前目录的文件"
    },
    {
      "role": "assistant",
      "content": [
        { "type": "text", "text": "让我执行 ls 命令..." },
        {
          "type": "tool_use",
          "id": "toolu_01234",
          "name": "bash",
          "input": { "command": "ls -la" }
        }
      ]
    },
    {
      "role": "user",
      "content": [
        {
          "type": "tool_result",
          "tool_use_id": "toolu_01234",
          "content": "total 48\ndrwxr-xr-x  10 user  staff   320 Apr  8 10:00 .\n-rw-r--r--   1 user  staff  1234 Apr  8 09:00 README.md\n..."
        }
      ]
    }
  ]
}
```

关键：
1. 工具结果放在 **user 消息**里（role: "user"），不是 assistant 消息
2. `tool_use_id` 必须和之前的 `tool_use` id 匹配
3. **整个历史都要发送**——Claude 需要上下文

---

## 多工具调用：Claude 可以一次调用多个工具

Claude 可以在一个回复里请求多个工具调用：

```json
{
  "role": "assistant",
  "content": [
    { "type": "text", "text": "我来同时查看几个文件..." },
    {
      "type": "tool_use", "id": "toolu_001",
      "name": "read_file", "input": { "path": "src/query.ts" }
    },
    {
      "type": "tool_use", "id": "toolu_002",
      "name": "read_file", "input": { "path": "src/Tool.ts" }
    },
    {
      "type": "tool_use", "id": "toolu_003",
      "name": "bash", "input": { "command": "wc -l src/*.ts" }
    }
  ],
  "stop_reason": "tool_use"
}
```

程序需要执行所有三个工具，然后把所有结果一起发回：

```json
{
  "role": "user",
  "content": [
    { "type": "tool_result", "tool_use_id": "toolu_001", "content": "..." },
    { "type": "tool_result", "tool_use_id": "toolu_002", "content": "..." },
    { "type": "tool_result", "tool_use_id": "toolu_003", "content": "..." }
  ]
}
```

**Claude Code 会并行执行这些工具**（对于只读、不互相依赖的工具），然后等所有结果都拿到，一起发回给 Claude。这就是 `runTools()` 函数的核心逻辑，第 9-6 节会详细讲。

---

## 工具调用的错误处理

如果工具执行失败，可以在 `tool_result` 里标记错误：

```json
{
  "type": "tool_result",
  "tool_use_id": "toolu_001",
  "content": "Error: Permission denied",
  "is_error": true
}
```

Claude 收到带 `is_error: true` 的结果后，会意识到操作失败，尝试其他方法或告知用户。

---

## `tool_choice`：控制 Claude 的工具使用行为

API 支持 `tool_choice` 参数控制 Claude 是否使用工具：

```json
{ "tool_choice": { "type": "auto" } }      // 默认：Claude 自己决定是否用工具
{ "tool_choice": { "type": "any" } }        // 必须使用至少一个工具
{ "tool_choice": { "type": "none" } }       // 不能使用工具（纯文字回复）
{ "tool_choice": { "type": "tool", "name": "bash" } }  // 必须使用 bash 工具
```

Claude Code 在不同场景下使用不同的 `tool_choice`，比如在某些查询模式下禁用工具调用。

---

## 这个协议如何映射到 Claude Code 的代码

```
API 的"工具定义（JSON Schema）"
↑
由 Zod schema → JSON Schema 转换（toolToAPISchema 函数）
  └── 在 src/utils/api.ts 里

API 的"tool_use 事件"（Claude 想用工具）
↓
在 queryLoop 里被收集到 toolUseBlocks 数组
↓
传给 runTools()（src/services/tools/toolOrchestration.ts）
↓
实际执行对应的工具（src/tools/*/）
↓
结果包装成"tool_result"消息
↓
加入 messages 数组，下一轮发给 API
```

---

## 本节小结

- 工具调用是一个三步协议：注册工具 → Claude 请求工具（tool_use） → 程序执行并返回结果（tool_result）
- 工具定义需要名称、描述、JSON Schema（input_schema）
- tool_use 包含唯一 id，tool_result 用这个 id 关联
- stop_reason 是 "tool_use" 表示 Claude 等待工具结果，"end_turn" 表示正常结束
- Claude 可以一次请求多个工具调用，Claude Code 并行执行它们
- 整个历史（包括 tool_use 和 tool_result）每次都要发给 API
