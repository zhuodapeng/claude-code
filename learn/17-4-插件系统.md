# 17-4 插件系统

## 问题：谁来管理 MCP 服务器的配置？

我们已经知道 MCP 服务器怎么连接（17-2），工具怎么转换（17-3）。但还有一个问题没解决：

**这些 MCP 服务器的配置从哪里来？**

用户可以手动在 `~/.claude.json` 里配置，企业可以通过策略文件下发，插件可以自动注册……同一个服务器名字可能出现在多个配置来源里，该用哪个？来自插件的服务器和手动配置的服务器发生冲突时谁赢？

这就是插件系统要解决的问题。

## LoadedPlugin：插件的运行时表示

**文件：[src/types/plugin.ts](../src/types/plugin.ts#L1)**

```typescript
export interface LoadedPlugin {
  name: string               // 插件名称

  manifestPath: string       // manifest.json 路径

  // 以下是插件提供的各种扩展路径
  commandsPath?: string      // 自定义命令的目录
  agentsPath?: string        // Agent 定义目录
  skillsPath?: string        // Skills 目录（见第 16 章）

```

插件是一个**功能包**——它不只提供 MCP 服务器，还可以提供 Skills、Hooks、Agent 定义等。一个插件就是一个扩展点的集合。

## 插件缓存路径与版本安全

用户安装的插件被缓存到本地磁盘：

**文件：[src/utils/plugins/pluginLoader.ts](../src/utils/plugins/pluginLoader.ts#L1)**

```typescript
// 缓存路径：~/.claude/plugins/cache/{marketplace}/{plugin}/{version}/
function getPluginCachePath(
  marketplace: string,
  pluginName: string,
  version: string,
): string {
  // 版本号净化：防止路径穿越攻击
  // 比如 version = "../../etc/passwd" → "..-..-etc-passwd"
  const safeVersion = version.replace(/[^a-zA-Z0-9\-_.]/g, '-')

  return path.join(
    os.homedir(),
    '.claude', 'plugins', 'cache',
    marketplace,
    pluginName,
    safeVersion,
  )
}
```

`version.replace(/[^a-zA-Z0-9\-_.]/g, '-')` 是一个简单但有效的安全措施。版本号通常是 `1.2.3` 这样的格式，只有字母数字、连字符和点。如果攻击者能控制版本字符串并注入 `../`，就能让插件加载到任意路径。这一行替换彻底消除了这种可能。

## probeSeedCache()：企业预装插件

企业部署场景里，IT 部门可能在用户机器上预装了一些插件（比如企业内部的数据库 MCP 服务器），这些插件不需要从网络下载——它们已经在磁盘上。

**文件：[src/utils/plugins/pluginLoader.ts](../src/utils/plugins/pluginLoader.ts#L1)**

```typescript
async function probeSeedCache(
  pluginName: string,
  version: string,
): Promise<string | null> {
  // 检查系统级 seed 目录（企业部署时由 IT 预置）
  const seedDirs = getSeedCacheDirs()

  for (const seedDir of seedDirs) {
    const pluginPath = path.join(seedDir, pluginName, version)
    if (await exists(pluginPath)) {
      return pluginPath
    }
  }
  return null
}

async function probeSeedCacheAnyVersion(
  pluginName: string,
): Promise<string | null> {
  // 版本未知时（第一次启动，还没同步到版本信息）用此函数
  // 只要找到任意版本就返回，解决鸡与蛋问题：
  // "必须知道版本才能找到插件" vs "必须找到插件才能知道版本"
  const seedDirs = getSeedCacheDirs()
  for (const seedDir of seedDirs) {
    const pluginDir = path.join(seedDir, pluginName)
    const versions = await readdir(pluginDir).catch(() => [])
    if (versions.length > 0) {
      return path.join(pluginDir, versions[0])
    }
  }
  return null
}
```

`probeSeedCacheAnyVersion` 解决的是一个有趣的鸡与蛋问题：

- 要找到正确版本的插件，你需要知道版本号
- 要知道版本号，你需要先连接到插件服务拉取元数据
- 但是插件服务本身可能就是个 MCP 服务器，需要插件才能启动

`probeSeedCacheAnyVersion` 的解法是：如果不知道版本，就用任意已安装的版本启动，之后再同步到正确版本。

## PluginError：20+ 种失败类型

插件系统的错误处理是整个项目里最细致的。`PluginError` 是一个 discriminated union，覆盖了 20+ 种不同的失败场景：

**文件：[src/types/plugin.ts](../src/types/plugin.ts#L1)**

```typescript
export type PluginError =
  // 路径相关
  | { type: 'path-not-found'; path: string }
  | { type: 'path-not-directory'; path: string }

  // Git 相关
  | { type: 'git-auth-failed'; url: string; message: string }
  | { type: 'git-clone-failed'; url: string; message: string }
  | { type: 'git-checkout-failed'; ref: string; message: string }

  // Manifest 验证
  | { type: 'manifest-not-found'; path: string }
  | { type: 'manifest-parse-error'; path: string; message: string }
  | { type: 'manifest-validation-error'; path: string; errors: string[] }

  // 依赖
  | { type: 'dependency-unsatisfied'; dep: string; required: string; found: string }
  | { type: 'dependency-missing'; dep: string }

  // 下载
  | { type: 'mcpb-download-failed'; url: string; statusCode: number }
  | { type: 'mcpb-checksum-mismatch'; expected: string; actual: string }

  // 运行时
  | { type: 'lsp-server-crashed'; serverName: string; exitCode: number }
  | { type: 'plugin-cache-miss'; pluginName: string; version: string }
  | { type: 'plugin-init-failed'; pluginName: string; message: string }

  // 其他...
```

每种错误类型都有不同的字段，携带了排查问题所需的精确信息。

`getPluginErrorMessage()` 函数把这些错误转为用户友好的提示：

**文件：[src/types/plugin.ts](../src/types/plugin.ts#L1)**

```typescript
export function getPluginErrorMessage(error: PluginError): string {
  switch (error.type) {
    case 'path-not-found':
      return `插件路径不存在: ${error.path}`
    case 'git-auth-failed':
      return `无法访问插件仓库 ${error.url}，请检查 Git 认证: ${error.message}`
    case 'manifest-validation-error':
      return `插件 manifest 校验失败: ${error.errors.join(', ')}`
    case 'dependency-unsatisfied':
      return `插件依赖 ${error.dep} 版本不满足: 需要 ${error.required}，找到 ${error.found}`
    case 'mcpb-checksum-mismatch':
      return `插件文件校验和不匹配，可能已损坏: 期望 ${error.expected}，实际 ${error.actual}`
    // ...
    default:
      // TypeScript 的穷举检查：如果这里有编译错误，说明你忘了处理某个新的错误类型
      const _exhaustive: never = error
      return `未知错误: ${JSON.stringify(error)}`
  }
}
```

最后一行 `const _exhaustive: never = error` 是 TypeScript 的穷举检查技巧——如果以后添加了新的 error 类型但忘记在 switch 里处理，这里会产生编译错误，不会悄悄变成 `default` 走了一个不精确的错误消息。

## MCP 服务器去重：谁赢？

插件系统最复杂的逻辑之一是**去重（deduplication）**。

场景：用户手动配置了一个叫 `my-db` 的 MCP 服务器，同时有一个插件也提供了一个叫 `my-db` 的 MCP 服务器——该用哪个？

**文件：[src/services/mcp/config.ts](../src/services/mcp/config.ts#L1)**

```typescript
export function dedupPluginMcpServers(
  manualServers: Record<string, ScopedMcpServerConfig>,
  pluginServers: Array<{ pluginName: string; servers: Record<string, ScopedMcpServerConfig> }>,
): Record<string, ScopedMcpServerConfig> {
  const result = { ...manualServers }  // 手动配置优先，先放进来

  for (const { pluginName, servers } of pluginServers) {
    for (const [name, config] of Object.entries(servers)) {
      // 如果手动配置里已有同名，插件的被跳过
      if (name in result) continue

      // 检查签名去重：即使名字不同，底层命令相同的服务器也算重复
      const signature = getMcpServerSignature(config)
      const existingSignatures = Object.values(result).map(getMcpServerSignature)

      if (!existingSignatures.includes(signature)) {
        result[name] = { ...config, pluginSource: pluginName }
      }
    }
  }

  return result
}
```

规则是：
1. **手动配置胜过插件**：用户手动写的 config 不会被插件覆盖
2. **先加载的插件胜过后加载的**：如果两个插件提供同名服务器，先加载的赢
3. **签名去重**：即使名字不同，如果底层命令相同也视为重复

## getMcpServerSignature()：什么算"同一个服务器"

**文件：[src/services/mcp/config.ts](../src/services/mcp/config.ts#L1)**

```typescript
export function getMcpServerSignature(config: McpServerConfig): string {
  if (config.type === 'stdio') {
    // stdio 服务器：用命令+参数的 JSON 作为签名
    return `stdio:${JSON.stringify([config.command, ...(config.args ?? [])])}`
  }

  if (config.type === 'sse' || config.type === 'http' || config.type === 'ws') {
    // URL 类型：用 unwrapCcrProxyUrl() 后的 URL 作为签名
    // 这处理了通过 CCR 代理的情况：同一个服务器可能有不同的代理 URL
    const url = config.type === 'sse' ? config.url
               : config.type === 'http' ? config.url
               : config.url
    return `url:${unwrapCcrProxyUrl(url)}`
  }

  return `other:${JSON.stringify(config)}`
}
```

**unwrapCcrProxyUrl()** 是这里的关键：

```typescript
// CCR = Claude Code Registry（Claude Code 的插件注册中心）
// 通过 CCR 代理的 URL 格式：https://ccr.example.com/proxy?mcp_url=https://actual-server.com
// 我们想用 actual-server.com 作为签名，而不是代理 URL
const CCR_PROXY_PATH_MARKERS = ['/proxy', '/mcp-proxy']

export function unwrapCcrProxyUrl(url: string): string {
  if (!CCR_PROXY_PATH_MARKERS.some(m => url.includes(m))) return url

  const parsed = new URL(url)
  // 从代理 URL 的 query string 里提取原始 URL
  return parsed.searchParams.get('mcp_url') || url
}
```

假设同一个 MCP 服务器：
- 用户直接配置了 `https://my-db-server.com/mcp`
- 插件通过代理访问 `https://ccr.example.com/proxy?mcp_url=https://my-db-server.com/mcp`

如果不 unwrap，这两个看起来是不同的服务器。Unwrap 之后都变成 `https://my-db-server.com/mcp`，可以正确识别为重复。

## isMcpServerAllowedByPolicy()：企业策略管控

企业环境下，管理员可以通过策略文件限制哪些 MCP 服务器可以被使用：

**文件：[src/services/mcp/config.ts](../src/services/mcp/config.ts#L1)**

```typescript
export function isMcpServerAllowedByPolicy(
  serverName: string,
  config: McpServerConfig,
  policy: McpPolicy,
): boolean {
  // 1. 检查 denylist：被明确拒绝的优先级最高
  if (isInDenylist(serverName, config, policy.denylist)) {
    return false   // 绝对拒绝，allowlist 救不了
  }

  // 2. 如果没有 allowlist，默认允许所有
  if (!policy.allowlist || policy.allowlist.length === 0) {
    return true
  }

  // 3. 有 allowlist 时，必须在 allowlist 里才允许
  return isInAllowlist(serverName, config, policy.allowlist)
}

function isInDenylist(name: string, config: McpServerConfig, denylist: PolicyEntry[]): boolean {
  return denylist.some(entry => matchesPolicyEntry(name, config, entry))
}

function matchesPolicyEntry(name: string, config: McpServerConfig, entry: PolicyEntry): boolean {
  // 可以通过服务器名匹配
  if (entry.name && minimatch(name, entry.name)) return true

  // 可以通过命令匹配（stdio 类型）
  if (entry.command && config.type === 'stdio' &&
      minimatch(config.command, entry.command)) return true

  // 可以通过 URL 模式匹配（网络类型）
  if (entry.urlPattern && 'url' in config &&
      minimatch(config.url, entry.urlPattern)) return true

  return false
}
```

策略设计遵循**安全默认原则**：
- denylist 是绝对的，无法被绕过
- 没有 allowlist 时，默认放行（方便普通用户）
- 有 allowlist 时，不在列表里的全部拒绝（企业严格管控场景）

## writeMcpjsonFile()：原子写入

当用户添加/修改 MCP 服务器配置时，需要写入配置文件。这个写入必须是**原子的**——不能出现"写了一半的配置文件"。

**文件：[src/services/mcp/config.ts](../src/services/mcp/config.ts#L1)**

```typescript
export async function writeMcpjsonFile(
  filePath: string,
  content: McpJsonContent,
): Promise<void> {
  const tempPath = filePath + '.tmp.' + Date.now()

  try {
    // 1. 写入临时文件
    const fd = await fs.open(tempPath, 'w')
    await fd.writeFile(JSON.stringify(content, null, 2))
    // 2. 强制刷盘（确保数据真正写到磁盘，而不是停留在 OS buffer）
    await fd.datasync()
    // 3. 设置权限（只有当前用户可读写）
    await fd.chmod(0o600)
    await fd.close()
    // 4. 原子重命名：在同一文件系统上，rename 是原子操作
    await fs.rename(tempPath, filePath)
  } catch (error) {
    // 清理临时文件
    await fs.unlink(tempPath).catch(() => {})
    throw error
  }
}
```

为什么这么复杂？

**问题**：如果直接 `writeFile(filePath, content)`，在写入过程中程序崩溃，会留下一个损坏的配置文件。下次启动时读到损坏的 JSON，整个配置读取失败。

**解法**：write-to-temp + rename 模式。先写到临时文件，写完后 `rename`。`rename` 在同一文件系统上是原子操作（由操作系统保证）——要么旧文件还在，要么新文件完整替换，不存在中间状态。

`datasync()` 确保数据真正持久化到磁盘，而不仅仅在 OS 的 page cache 里——这对于配置文件这种重要数据特别必要。

## 插件系统的整体流程

```
Claude Code 启动
        │
        ▼
读取所有配置来源（按优先级）
  ├── enterprise policy
  ├── managed config
  ├── ~/.claude.json (user)
  ├── .mcp.json (project)
  └── dynamic (runtime)
        │
        ▼
发现插件
  ├── probeSeedCache()       — 企业预装，最高优先级
  ├── ~/.claude/plugins/     — 用户已安装
  └── 内置插件               — 随 Claude Code 发布
        │
        ▼
dedupPluginMcpServers()      — 手动配置胜插件，先加载胜后加载
        │
        ▼
isMcpServerAllowedByPolicy() — 企业策略过滤
        │
        ▼
最终 MCP 服务器列表
        │
        ▼
connectToServer() × N        — 并发连接所有允许的服务器
```

## 本章小结

插件系统解决了 MCP 服务器配置的治理问题：

1. **分层配置**：enterprise > managed > user > project > dynamic，高优先级覆盖低优先级
2. **去重策略**：手动配置胜插件，先加载胜后加载，签名去重防止同一服务器重复注册
3. **企业策略**：denylist 绝对优先，allowlist 可选，安全默认
4. **原子写入**：write-to-temp + rename，防止配置损坏
5. **签名识别**：`unwrapCcrProxyUrl()` 确保代理和直连的同一服务器被正确去重
