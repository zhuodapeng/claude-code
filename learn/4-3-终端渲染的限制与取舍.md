# 4-3 终端渲染的限制与取舍

> **本节目标**：理解终端环境的根本限制——为什么某些 UI 行为在浏览器里理所当然，在终端里却不可能或很困难。这些限制直接影响了 Claude Code 很多设计决策。

---

## 终端高度和滚动：最大的限制

在浏览器里，你可以渲染一个 100 万行的列表（配合虚拟化），用户可以无限滚动。

在终端里：**Ink 渲染的内容高度受终端窗口高度限制。**

具体来说：Ink 使用一种叫 **"alternate screen"（交替屏幕）** 的机制（可选），或者更常见的是，它在终端当前位置往下输出，输出内容超出屏幕后上滚。

**Claude Code 的选择**：不使用 alternate screen。原因是：
1. Alternate screen 模式退出后屏幕内容消失，用户看不到历史
2. Claude Code 的对话历史需要可滚动查看
3. 对话内容应该留在终端缓冲区里，方便用户复制

这个选择带来了一个取舍：**Claude Code 没有"全屏"的 UI 框架**，所有渲染都是往下追加的。工具执行状态、流式 token 输出，都在屏幕上往下"生长"，而不是在固定区域更新。

---

## ANSI 支持差异：不同终端不一样

你在 macOS Terminal 上看到的颜色效果，在 Windows cmd.exe 里可能完全是乱码（`[32mHello[0m`）。

在 Claude Code 里你会看到这类处理代码：

```typescript
// 检测终端支持的颜色深度
import supportsColor from 'supports-color'

// 根据终端能力降级渲染
const hasColor = supportsColor.stdout !== false
const has256Colors = supportsColor.stdout && supportsColor.stdout.has256
const hasTrueColor = supportsColor.stdout && supportsColor.stdout.hasTrueColor
```

这就是为什么 Claude Code 在不同终端里显示效果可能略有不同——它在能力范围内尽量美观，在不支持颜色的终端里退化为纯文字。

---

## 宽字符问题：中文、Emoji 占几列？

终端的字符网格是等宽的——每个格子放一个字符。但有一个问题：

- ASCII 字符（英文字母、数字）：占 **1 列**
- CJK 字符（中文、日文、韩文）：占 **2 列**
- Emoji：可能占 1 列、2 列，甚至更多

如果你不处理宽字符，中文文本会"撑破"布局计算：

```
计算宽度为 10：Hello     → 实际显示：Hello
计算宽度为 10：你好      → 实际显示：你好    （5个中文=10列，正好）
计算宽度为 10：Hello你   → 实际显示：Hello你 （5+2=7列，没问题）
计算宽度为 10：你好Hello → 实际显示：你好Hello（4+5=9列，但以为是7，布局错）
```

**Claude Code 的解法**：

```typescript
// package.json 里
"get-east-asian-width": "^1.5.0",  // 获取 CJK 字符的视觉宽度
"strip-ansi": "^7.2.0",            // 计算宽度前先去掉 ANSI 码
"wrap-ansi": "^10.0.0",            // 正确地在 ANSI 文字里换行
```

这三个库合作，确保中文文本在终端里正确换行和对齐。

## 输入处理：没有 `<input>` 标签

在 Web 里，获取用户输入很简单：

```html
<input type="text" onInput={handleInput} />
```

终端里没有这个。用户按键时，原始键码（raw keycode）会流入 `process.stdin`。Ink 监听这个流，解析键码，提供 `useInput` Hook：

```typescript
import { useInput } from 'ink'

function InputBox() {
  const [text, setText] = useState('')
  
  useInput((input, key) => {
    if (key.return) {
      // 用户按了 Enter
      submitText(text)
    } else if (key.backspace || key.delete) {
      setText(t => t.slice(0, -1))
    } else if (input) {
      // 普通字符输入
      setText(t => t + input)
    }
  })
  
  return <Text>{text}█</Text>  // █ 是光标
}
```

但这比 HTML input 原始得多——没有光标位置管理、没有选区、没有复制粘贴 API。Claude Code 的输入框实现了相当多的手工逻辑来模拟这些功能。

## 渲染频率限制

Ink 有一个内部的渲染节流：不是每次状态变化都立即重绘，而是在事件循环的下一个 tick 批量更新。

这对流式 token 渲染很重要：

```
情景：每 50ms 来一个 token（正常情况）

没有节流：
  token1 → 立即渲染 → token2 → 立即渲染 → ...（每个 token 一次重绘）

有节流（批量）：
  token1 → 排队
  token2 → 排队（同一 tick）
  ─── 渲染 tick ───
  → 一次性渲染 token1 + token2
```

这减少了渲染次数，让 CPU 不会被无意义的重绘占满。

## 取舍总结表

| 需求 | Web 怎么做 | 终端/Ink 怎么做 | 代价 |
|------|-----------|--------------|------|
| 文字颜色 | CSS color | ANSI 转义码 | 需要检测终端支持 |
| 滚动 | overflow: auto | 终端原生滚动 | 没有"虚拟滚动"，无法精确控制 |
| 文字输入 | `<input>` | 手动解析 stdin | 功能有限，需大量手工实现 |
| 图片显示 | `<img>` | 基本不可能 | 某些终端支持特殊协议，但不通用 |
| 动画 | CSS animation | 定时器 + 状态更新 | 帧率受渲染节流限制 |
| 布局宽度 | 像素级 | 字符数（宽字符要特殊处理） | 中文等宽字符需额外处理 |
| 鼠标交互 | 丰富的鼠标事件 | 有限，依赖终端支持 | 基本用键盘代替鼠标 |

---

## 这些限制如何影响 Claude Code 的设计

理解了这些限制，很多 Claude Code 的设计决策就能理解了：

1. **为什么工具执行状态用 `●` 标记而不是进度条？**
   → 进度条宽度依赖终端宽度，跨平台行为复杂；字符标记简单可靠。

2. **为什么流式 token 是追加显示而不是在原地更新？**
   → 实时在原地更新需要追踪光标位置，复杂且容易出 bug；追加更简单。

3. **为什么有 Recovery CLI（降级模式）？**
   → 当 Ink TUI 在某些终端环境出问题时，降级到简单的 `readline` REPL（[bin/claude-haha](../bin/claude-haha#L11-L13)）。

4. **为什么输入框要自己实现光标移动、历史记录？**
   → 终端没有内置这些能力，必须手工实现。

---

## 本节小结

- 终端渲染没有 alternate screen，内容追加显示，不是固定区域更新
- ANSI 颜色码支持因终端而异，需要检测和降级
- CJK 字符和 Emoji 占多列，布局计算需要特殊处理
- 用户输入是原始键码流，没有 `<input>` 标签的便利
- Ink 有渲染节流，批量处理更新，避免 CPU 浪费
- 这些限制直接解释了 Claude Code UI 的很多设计决策
