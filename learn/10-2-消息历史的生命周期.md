# 10-2 消息历史（mutableMessages）的生命周期

## 本节要解决的问题

`QueryEngine.mutableMessages` 是整个对话的"记忆"——Claude 看到的所有历史消息都来自这里。

但它不是一个只会增长的数组。它有自己的**生命周期**：什么时候增长，什么时候被修剪，什么时候被完全替换？

如果你不理解这个生命周期，你就无法理解 Claude Code 是如何支持"长对话不崩溃"的。

---

## mutableMessages 的两个视角

先分清两个变量：

| 变量 | 定义位置 | 生命周期 | 用途 |
|---|---|---|---|
| `this.mutableMessages` | QueryEngine 实例字段 | 整个会话（多次 submitMessage） | 下一轮的上下文基础 |
| `messages` | submitMessage() 内的局部变量 | 单次 submitMessage | 传给 query() 的快照，用于 transcript 写入 |

```typescript
// submitMessage() 开始时：
const messages = [...this.mutableMessages]  // ← 快照
```

`messages` 是 `mutableMessages` 的快照，传给 `query()` 作为本轮的消息历史输入。之后，两者都会同步增长（新消息被 push 进两者），但修剪操作只作用于 `this.mutableMessages`（影响下一轮），不影响本轮 query() 内部的运行。

---

## 阶段一：初始化

**文件：[src/QueryEngine.ts](../src/QueryEngine.ts#L200-L207)**

```typescript
constructor(config: QueryEngineConfig) {
  this.mutableMessages = config.initialMessages ?? []
  ...
}
```

`initialMessages` 可以是空数组（全新对话），也可以是从 transcript 恢复的历史（`--resume` 模式）。

---

## 阶段二：每轮对话的增长

在 `submitMessage()` 的主循环里，所有"有意义的"消息都被推入 `mutableMessages`：

**文件：[src/QueryEngine.ts](../src/QueryEngine.ts#L768-L786)**

```typescript
case 'assistant':
  this.mutableMessages.push(message)  // ← Claude 的回复
  yield* normalizeMessage(message)
  break
case 'progress':
  this.mutableMessages.push(message)  // ← 工具执行进度
  break
case 'user':
  this.mutableMessages.push(message)  // ← 工具结果 / 用户消息
  yield* normalizeMessage(message)
  break
case 'attachment':
  this.mutableMessages.push(message)  // ← 附件（排队命令、记忆注入）
  break
case 'system':
  this.mutableMessages.push(message)  // ← 系统消息（compact_boundary 等）
  break
```

注意：`stream_event` 和 `tombstone` **不进入 mutableMessages**。流式事件只是 UI 更新，不需要被持久化；tombstone 是"撤销标记"，本身不需要留在历史里。

### 典型的一轮增长

```
submitMessage() 前：
  mutableMessages = [
    user: "帮我读 main.ts"
  ]

submitMessage("修改第 42 行") 执行后：
  mutableMessages = [
    user: "帮我读 main.ts",
    assistant: "[tool_use: Read('main.ts')]",
    user: "[tool_result: '文件内容...']",
    assistant: "文件内容如下：...",
    user: "修改第 42 行",          ← 新增（来自本次 submitMessage）
    assistant: "[tool_use: Edit('main.ts', ...)]",
    user: "[tool_result: '修改成功']",
    assistant: "已完成修改。",
  ]
```

每一轮都在追加，消息历史越来越长。

---

## 阶段三：Compact 之后的修剪

当上下文太长时，Claude Code 会触发 compact（压缩），把旧历史替换成摘要。此时，mutableMessages 会被**截断**。

**文件：[src/QueryEngine.ts](../src/QueryEngine.ts#L921-L934)**

```typescript
if (
  message.subtype === 'compact_boundary' &&
  message.compactMetadata
) {
  // 释放 compact 之前的消息，让 GC 回收内存
  const mutableBoundaryIdx = this.mutableMessages.length - 1
  if (mutableBoundaryIdx > 0) {
    this.mutableMessages.splice(0, mutableBoundaryIdx)  // ← 删除 boundary 之前的所有消息
  }
  // 同步截断 messages（本轮快照）
  const localBoundaryIdx = messages.length - 1
  if (localBoundaryIdx > 0) {
    messages.splice(0, localBoundaryIdx)
  }
}
```

Compact 后的 `mutableMessages`：

```
compact 之前：
  [user1, assistant1, user2, assistant2, ..., user50, assistant50, compact_boundary]

compact 之后（splice）：
  [compact_boundary]

再下一轮追加：
  [compact_boundary, user51, assistant51, ...]
```

`compact_boundary` 消息包含了压缩摘要——Claude 在下一轮看到的不是 50 轮的完整历史，而是 compact_boundary 里的摘要加上之后的新消息。这样内存用量被显著控制了。

---

## 阶段四：Snip 截断（HISTORY_SNIP 特性）

**文件：[src/QueryEngine.ts](../src/QueryEngine.ts#L905-L915)**

```typescript
const snipResult = this.config.snipReplay?.(
  message,
  this.mutableMessages,
)
if (snipResult !== undefined) {
  if (snipResult.executed) {
    this.mutableMessages.length = 0         // ← 清空！
    this.mutableMessages.push(...snipResult.messages)  // ← 替换为 snip 后的版本
  }
  break
}
```

Snip 是比 Compact 更激进的截断方式——它**完全替换**整个 `mutableMessages`。这通常用于 SDK 长会话里，防止消息数组无限增长导致内存泄漏。

---

## 阶段五：会话结束

`QueryEngine` 实例销毁时，`mutableMessages` 被 GC 回收。但在此之前，所有消息都已通过 `recordTranscript()` 写入磁盘，可以用 `--resume` 恢复。

---

## mutableMessages 的两条生长轴

```
时间轴（submitMessage 调用次序）:
                                       
  初始化 ─── submitMessage1 ─── submitMessage2 ─── submitMessage3
                │                    │                    │
                ▼                    ▼                    ▼
           [user1, …]          […, user2, …]         […, user3, …]
                                                           ↑
                                                  如果触发 compact：
                                                  splice 旧消息 →
                                                  [compact_boundary, user3, …]


轮内轴（一次 submitMessage 内部）:
  
  push(用户消息)
      │
      ▼
  push(assistant 消息1)    ← Claude 流式输出完成
      │
      ▼
  push(tool_result 消息)   ← 工具执行完毕
      │
      ▼
  push(assistant 消息2)    ← Claude 继续回答
      │
      ▼
  submitMessage 返回
```

---

## 为什么 mutableMessages 比 messages（快照）更权威？

在 submitMessage() 里，`messages`（快照）和 `this.mutableMessages` 会同步增长。但修剪操作只更新 `this.mutableMessages`，而 `messages` 在本轮 query() 调用期间**不会被修剪**——query() 需要完整的历史才能继续运行。

修剪后，下一次 `submitMessage()` 调用时：

```typescript
const messages = [...this.mutableMessages]  // ← 已修剪的版本成为新一轮的起点
```

这是一个"延迟生效"的设计：compact/snip 的效果在当前轮的结束时固化到 `mutableMessages`，在**下一轮**才作为新的起点生效。

---

## 本节小结

- `mutableMessages` 是 QueryEngine 持有的跨轮消息历史，不是每次 submitMessage 重新创建的
- 每轮 submitMessage 会追加 assistant/user/attachment/system 消息（stream_event 和 tombstone 不追加）
- Compact 触发时，`splice(0, boundaryIdx)` 删除 boundary 之前的旧消息，释放内存
- Snip 截断时，`length = 0; push(...newMessages)` 完全替换消息数组
- 修剪在当前轮结束时固化到 `mutableMessages`，下一轮调用才生效
