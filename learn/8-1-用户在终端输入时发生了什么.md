# 8-1 用户在终端输入时发生了什么

> **本节目标**：用户在 Claude Code 终端里打了一段话，按下回车。这条消息从键盘输入到最终变成 Anthropic API 请求，要经过多少个步骤？每一步做了什么？这节课从 `onSubmit` 开始，追踪完整的"用户输入处理链"。

---

## 从按下回车开始

整个输入处理链的起点是 `REPL.tsx` 里的 `onSubmit` 回调：

**文件：[src/screens/REPL.tsx](../src/screens/REPL.tsx#L3139)**

```typescript
const onSubmit = useCallback(async (
  input: string,          // 用户输入的文本
  helpers: PromptInputHelpers,  // 输入框操作（清空、重置历史）
  ...
) => {
  // 0. 先做几件"立刻要做的事"
  repinScroll()  // 把滚动位置固定到最底部
  
  // 1. 处理"立即命令"（不需要等 Claude 的命令）
  if (input.trim().startsWith('/')) {
    const immediateCommand = findImmediateCommand(input, commands)
    if (immediateCommand && queryGuard.isActive) {
      // 执行立即命令（比如 /config 打开配置界面）
      // 立即命令在 Claude 忙的时候也能执行
      return
    }
  }
  
  // 2. 如果 Claude 正在处理中，把输入加入队列
  if (queryGuard.isActive) {
    enqueue({ value: input, mode, pastedContents })
    onInputChange('')  // 清空输入框
    return
  }
  
  // 3. Claude 空闲，直接走 handlePromptSubmit
  await handlePromptSubmit({ input, helpers, ... })
}, [...])
```

这里有一个关键的设计决策：用户输入时，Claude 可能正在忙（处理上一条消息）。怎么处理这种情况？

**朴素方案**：禁用输入框，等 Claude 处理完再允许输入。
**实际方案**：维护一个消息队列——用户可以继续输入，这些消息会在 Claude 空闲后自动发出。

这就是 `queryGuard.isActive` 检查的意义，和 `enqueue()` 的作用。

---

## 两条路径：立即执行 vs 排队

```
用户按下回车
        ↓
input.startsWith('/') ?
  ├─ 是，是立即命令？
  │   ├─ 是 → 不管 Claude 是否忙，立刻执行
  │   │       例如：/config 打开配置 UI
  │   └─ 否 → 走普通流程
  └─ 否（普通文本）↓

queryGuard.isActive ?（Claude 是否正在处理？）
  ├─ 是 → enqueue(input)，清空输入框，返回
  │        Claude 空闲后会自动取出并执行
  └─ 否 → 立即调用 handlePromptSubmit(input)
```

为什么"立即命令"要特殊处理？

`/config`、`/doctors`、`/permissions` 这些命令显示的是 UI 界面，不需要发 API 请求，跟 Claude 的处理状态无关。如果这些命令也被排队，用户在 Claude 忙的时候就无法打开配置——体验很差。

## handlePromptSubmit()：真正的预处理

**文件：[src/utils/handlePromptSubmit.ts](../src/utils/handlePromptSubmit.ts#L120)**

`handlePromptSubmit()` 负责多件事：

```typescript
export async function handlePromptSubmit(params: HandlePromptSubmitParams): Promise<void> {
  const { input, mode, pastedContents, helpers, ... } = params
  
  // 1. 处理粘贴内容（图片、长文本）
  //    [Image #1] 这样的占位符被替换成实际内容
  const finalInput = expandPastedTextRefs(input, pastedContents)
  
  // 2. 空输入检查
  if (input.trim() === '') return
  
  // 3. 处理 exit/quit/:q 等退出命令
  if (['exit', 'quit', ':q', ...].includes(input.trim())) {
    handleExit()
    return
  }
  
  // 4. 如果 Claude 仍在处理（race condition），再次尝试排队
  if (queryGuard.isActive) {
    enqueue({ value: finalInput, mode, pastedContents })
    onInputChange('')
    return
  }
  
  // 5. 构造 QueuedCommand 对象，统一路径
  const cmd: QueuedCommand = {
    value: finalInput,
    preExpansionValue: input,
    mode,
    pastedContents: hasImages ? pastedContents : undefined,
  }
  
  // 6. 进入核心执行路径
  await executeUserInput({ queuedCommands: [cmd], ... })
}
```

注意步骤 5：无论是直接执行还是从队列中取出，都会先构造一个 `QueuedCommand` 对象，然后走同一个 `executeUserInput()` 函数。这是"统一路径"设计——避免两条代码路径出现不一致的行为。

## executeUserInput()：保护锁 + 消息处理循环

**文件：[src/utils/handlePromptSubmit.ts](../src/utils/handlePromptSubmit.ts#L396)**

```typescript
async function executeUserInput(params: ExecuteUserInputParams): Promise<void> {
  // 1. 申请"保护锁"
  //    queryGuard.reserve() 标记"查询即将开始"
  //    后续并发的 handlePromptSubmit 会排队
  queryGuard.reserve()
  
  // 2. 创建新的 AbortController（每次查询独立）
  const abortController = createAbortController()
  setAbortController(abortController)
  
  // 3. 遍历所有排队的命令（通常只有 1 条，但批量队列可能多条）
  const newMessages: Message[] = []
  for (const cmd of queuedCommands) {
    const result = await processUserInput({
      input: cmd.value,
      mode: cmd.mode,
      pastedContents: cmd.pastedContents,
      context: makeContext(),
      ...
    })
    newMessages.push(...result.messages)
  }
  
  // 4. 调用 onQuery（真正发起 API 请求）
  if (shouldQuery) {
    await onQuery(newMessages, abortController, ...)
  }
}
```

`queryGuard` 是一个状态机：
- `idle`：可以接受新查询
- `dispatching`：`queryGuard.reserve()` 后，正在准备查询
- `running`：`onQuery()` 开始后，Claude 正在处理
- 回到 `idle`：查询完成

这个状态机确保在任何时刻最多只有一个活跃的 Claude 对话——不会出现两个 API 请求同时进行的情况。

## processUserInput()：识别输入类型

**文件：[src/utils/processUserInput/processUserInput.ts](../src/utils/processUserInput/processUserInput.ts#L85)**

`processUserInput()` 负责识别输入是哪种类型，然后走不同的处理路径：

```
输入类型：
  ├─ Bash 模式（`!command`）→ processBashCommand()
  ├─ Slash 命令（`/xxx`）→ processSlashCommand()
  └─ 普通文本提示 → processTextPrompt()
```

```typescript
export async function processUserInput({ input, mode, ... }) {
  
  // 判断输入类型
  if (mode === 'bash' || (typeof input === 'string' && input.startsWith('!'))) {
    return processBashCommand(...)  // 直接执行 bash 命令
  }
  
  if (!skipSlashCommands && typeof input === 'string' && input.trim().startsWith('/')) {
    const slashCmd = parseSlashCommand(input)
    return processSlashCommand(slashCmd, ...)  // 执行 skill/命令
  }
  
  // 普通文本 → 直接进入 processTextPrompt
  return processUserInputBase(input, mode, ...)
}
```

### Bash 模式（`!command`）

这不是发给 Claude 的——直接在本地执行 bash 命令并显示输出。用户可以在 Claude Code 里直接运行命令，不需要 Claude 的参与。

### Slash 命令（`/xxx`）

`/commit`、`/compact`、`/diff` 这些是 Skills（参见第 16 章）。它们被展开成一段提示词，然后以"普通文本"的形式发给 Claude。Skills 在这一步被解析和展开。

### 普通文本提示

最常见的路径，走 `processUserInputBase()` → `processTextPrompt()`。

## processTextPrompt()：构造 UserMessage

**文件：[src/utils/processUserInput/processTextPrompt.ts](../src/utils/processUserInput/processTextPrompt.ts#L19)**

这是把用户文本转换成 API 消息格式的地方：

```typescript
export function processTextPrompt(
  input: string | ContentBlockParam[],
  imageContentBlocks: ContentBlockParam[],
  ...
): { messages: (UserMessage | AttachmentMessage)[], shouldQuery: boolean } {
  
  // 1. 检查"负面关键词"（如用户说"不对"、"停下"）
  //    这些关键词可能触发特殊行为（记录到 analytics）
  const isNegative = matchesNegativeKeyword(userPromptText)
  
  // 2. 如果有图片，构建包含图片的消息
  if (imageContentBlocks.length > 0) {
    const userMessage = createUserMessage({
      content: [...textContent, ...imageContentBlocks],
      uuid: uuid,
      imagePasteIds: ...,
    })
    return { messages: [userMessage, ...attachmentMessages], shouldQuery: true }
  }
  
  // 3. 普通文本消息
  const userMessage = createUserMessage({
    content: input,
    uuid,
    permissionMode,
  })
  return { messages: [userMessage, ...attachmentMessages], shouldQuery: true }
}
```

注意返回值里有 `shouldQuery: boolean`。为什么需要这个？

因为某些 slash 命令不需要发 API 请求（比如 `/clear` 只是清空对话历史，`/help` 直接显示帮助信息）。`shouldQuery: false` 告诉上层"这条输入处理完了，不需要调 API"。

## UserPromptSubmit Hooks

在 `processTextPrompt()` 之后，还有一步：

**文件：[src/utils/processUserInput/processUserInput.ts](../src/utils/processUserInput/processUserInput.ts#L182)**

```typescript
// 执行 UserPromptSubmit hooks
for await (const hookResult of executeUserPromptSubmitHooks(
  inputMessage,
  appState.toolPermissionContext.mode,
  context,
)) {
  if (hookResult.blockingError) {
    // Hook 阻止了这次提交
    return { messages: [errorMessage], shouldQuery: false }
  }
}
```

`UserPromptSubmit` hooks 是用户可以配置的钩子，在每次用户提交消息时运行。用途例如：

- 自动给某些关键词的消息添加上下文
- 在敏感词触发时拒绝提交
- 记录用户的提问历史

## 完整的输入处理链

```
用户按下回车
        ↓
REPL.tsx: onSubmit(input)
  ├─ 立即命令（/config）→ 直接执行 UI 操作
  └─ 普通输入 ↓
  
handlePromptSubmit()
  ├─ expandPastedTextRefs()  扩展粘贴内容
  ├─ 空输入/退出命令检查
  └─ 排队 or 直接执行 ↓
  
executeUserInput()
  ├─ queryGuard.reserve()  申请保护锁
  ├─ createAbortController()  创建取消控制器
  └─ processUserInput() 循环 ↓
  
processUserInput()
  ├─ Bash 模式 → processBashCommand() → 返回
  ├─ Slash 命令 → processSlashCommand() → 展开 Skill
  └─ 普通文本 → processUserInputBase() ↓
  
processUserInputBase()
  ├─ 处理图片（resize）
  ├─ 处理 IDE selection（选中的代码）
  └─ processTextPrompt() ↓
  
processTextPrompt()
  ├─ createUserMessage(input)
  └─ 返回 { messages: [UserMessage], shouldQuery: true } ↓
  
executeUserInput() 继续
  ├─ UserPromptSubmit hooks
  └─ onQuery(newMessages, abortController)  → 下一章
```

---

## 为什么要这么多层？

看完这个流程，你可能觉得"层数太多了"。为什么不直接 `onSubmit → onQuery`？

每一层都有它的职责：

| 层 | 职责 | 为什么需要 |
|---|---|---|
| `onSubmit` | UI 响应，立即命令，滚动重置 | 这些是 UI 关注点，不应该和业务逻辑混在一起 |
| `handlePromptSubmit` | 队列判断，粘贴展开，统一路径 | 队列处理和直接处理要走同一条路 |
| `executeUserInput` | 保护锁，AbortController，批量处理 | 确保并发安全，支持多命令批处理 |
| `processUserInput` | 类型识别（bash/slash/text） | 三种输入类型的逻辑完全不同，分开处理 |
| `processTextPrompt` | 消息构造 | 纯函数，易测试 |

这是典型的"关注点分离"——每一层只做自己的事，层与层之间通过清晰的接口通信。

## 本节小结

- `onSubmit` 是入口，分两条路：立即命令（不管 Claude 状态）和普通输入（可能排队）
- `queryGuard` 是核心并发控制机制——任何时刻最多只有一个活跃查询
- `handlePromptSubmit` 展开粘贴内容，构造 `QueuedCommand`，统一直接执行和队列执行的路径
- `processUserInput` 根据输入类型（bash/slash/text）分流处理
- `processTextPrompt` 是最终构造 `UserMessage` 的地方，返回 `{ messages, shouldQuery }`
- `UserPromptSubmit` hooks 可以在消息发出前阻止提交
