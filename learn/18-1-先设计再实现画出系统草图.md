## 第一步：明确我们要构建什么

我们要构建的是：**一个能够调用工具的 LLM 命令行工具（LLM CLI）**。

最终它应该支持：
1. 用户在终端输入问题
2. LLM 流式输出回答
3. LLM 可以调用工具（读文件、执行命令）
4. 有 TUI 界面（基于 Ink/React）
5. 执行危险操作前询问用户
6. 跨会话记忆

我们分 7 个步骤实现，每步增加一个能力。这一节先把草图画出来。

## 系统草图：从外到内

在写一行代码之前，先在脑子里走一遍整个系统。

### 最外层：用户的视角

```
用户打开终端，运行 ./my-llm-cli

> 你好，请帮我列出当前目录下的文件

我来帮你查看...
[调用 list_files 工具]
当前目录包含以下文件：
- README.md
- src/
- package.json
...

> _
```

用户看到的是：一个像聊天窗口一样的界面，问题-回答轮流出现，LLM 的回答实时流式输出。

### 核心循环：系统的骨架

```
while (用户还在) {
  接收用户输入
  构建消息历史（加入用户消息）
  发送给 LLM（流式）
    while (LLM 还在生成) {
      接收 token，打印到屏幕
      if (LLM 要调用工具) {
        执行工具
        把工具结果加入消息历史
        再次发送给 LLM（工具调用后继续生成）
      }
    }
  构建消息历史（加入 LLM 回答）
}
```

这就是 `queryLoop` 的骨架。越简单的骨架越好——后面每一步都是在这个骨架上加细节。

### 数据流向

```
用户输入 (string)
    │
    ▼
messages: Message[]          ← 消息历史，核心状态
    │
    ▼
Anthropic API (stream)
    │
    ├──── text token ──────── 打印到屏幕
    │
    └──── tool_use ──────────► 工具执行
                                    │
                                    ▼
                              tool_result
                                    │
                                    ▼ 加入 messages
                              再次调用 API
```

整个系统里最重要的状态是 `messages` 数组。其他一切都是围绕它的操作：往里加内容、读它来构建 API 请求。

## 核心抽象：提前识别出来

在开始写代码之前，识别出系统里的核心抽象。这些抽象是设计决策的结果——好的抽象边界，后面扩展代码会很顺畅；坏的抽象，每次扩展都是痛苦的重构。

### 1. Message（消息）

消息是 Anthropic API 的核心货币。我们需要能表示：
- 用户消息（`role: 'user'`）
- 助手消息（`role: 'assistant'`，可以包含文本和工具调用）
- 工具结果消息（`role: 'user'`，包含 `tool_result` 块）

```typescript
// 来自 Anthropic SDK 的类型，我们直接用
type Message = UserMessage | AssistantMessage
```

### 2. Tool（工具）

工具是可执行的能力单元。一个工具有：
- 元数据（名字、描述、参数 schema）——供 LLM 选择工具使用
- 执行函数（接收参数，返回结果）——供 Claude Code 执行

```typescript
interface Tool {
  name: string
  description: string
  inputSchema: z.ZodSchema   // 或 JSON Schema
  call(input: unknown): Promise<string>
}
```

### 3. PermissionGate（权限门控）

工具执行前的决策点：
- 自动批准（读文件、只读操作）
- 需要确认（写文件、执行命令）
- 自动拒绝（危险操作）

```typescript
type PermissionDecision = 'allow' | 'deny' | 'ask-user'

interface PermissionGate {
  check(toolName: string, input: unknown): PermissionDecision
}
```

### 4. Memory（记忆）

会话之间持久化的信息：
- 读取：启动时注入 system prompt
- 写入：会话结束时保存重要信息

```typescript
interface Memory {
  load(): Promise<string>           // 返回注入 system prompt 的文本
  save(content: string): Promise<void>
}
```

## 技术选型

我们要做几个基础技术决定，这些决定影响后面所有的代码：

### 运行时：Bun

- 内置 TypeScript 支持，不需要编译步骤
- 比 Node.js 快
- 直接运行 `.ts` 文件：`bun src/index.ts`

### API 客户端：@anthropic-ai/sdk

```typescript
import Anthropic from '@anthropic-ai/sdk'
const client = new Anthropic({ apiKey: process.env.ANTHROPIC_API_KEY })
```

### 终端 UI：Ink（阶段性引入）

前几步用 `process.stdout.write()` 直接输出，后面再替换成 Ink。

### 类型校验：Zod

工具的输入参数校验，防止 LLM 传入错误格式的参数。

### 项目结构（最终形态）

```
my-llm-cli/
├── src/
│   ├── index.ts           ← 入口
│   ├── core/
│   │   ├── loop.ts        ← queryLoop
│   │   └── stream.ts      ← 流式响应处理
│   ├── tools/
│   │   ├── index.ts       ← 工具注册
│   │   ├── readFile.ts
│   │   ├── writeFile.ts
│   │   └── bash.ts
│   ├── permissions/
│   │   └── index.ts       ← 权限系统
│   ├── memory/
│   │   └── index.ts       ← 记忆系统
│   ├── ui/
│   │   └── App.tsx        ← Ink TUI

```

但我们不会一开始就建这么多目录。我们从单个文件开始，随着功能增加，自然地拆分。

## 关键设计决定

在开始写代码之前，我们还有三个设计决定要做。回顾 Claude Code 怎么做的，然后决定自己的方案：

### 决定 1：消息历史放在哪里？

**Claude Code 的做法**：AppState（外部状态存储），通过 `useSyncExternalStore` 连接 React。

**我们的做法**：先用模块级的简单数组，后面需要时再换。复杂度分级——不要一开始就做最复杂的方案。

```typescript
// 第一版：简单数组，够用就好
let messages: Message[] = []
```

### 决定 2：工具执行和 UI 更新怎么解耦？

**Claude Code 的做法**：queryLoop 发出事件，UI 订阅；工具执行后，更新 AppState，Ink 重新渲染。

**我们的做法**：先直接 `console.log`，引入 Ink 后换成 React 状态更新。接口先定义，实现可以换。

### 决定 3：权限检查同步还是异步？

**Claude Code 的做法**：异步，`checkPermissions()` 返回 Promise，UI 展示对话框等用户点击。

**我们的做法**：readline 同步问，简单直接。

## 七步计划

我们按以下顺序构建：

```
Step 1: 50 行实现最简 LLM CLI
         ↓ 加入工具
Step 2: 工具调用（读文件 + Bash）
         ↓ 优化输出
Step 3: 流式输出（打字机效果）
         ↓ 换成 Ink
Step 4: Ink TUI（交互界面）
         ↓ 加入安全性
Step 5: 权限系统
         ↓ 加入持久化
Step 6: 记忆系统
         ↓ 加入并发
Step 7: 子 Agent（并行任务分发）
```

每一步都是可以独立运行的完整程序，不是"差一步就能跑"的半成品。

## 认知准备：什么是好的从零实现

从零实现最容易犯的错误是：一开始就想把所有边界情况都处理好。

这是错的。

好的从零实现遵循这个原则：**让程序在任何时刻都是完整的，哪怕功能很少**。

- Step 1 的程序：50 行，只能对话，不能工具调用，但可以运行
- Step 2 的程序：100 行，加了工具调用，但还是可以运行
- Step 7 的程序：500+ 行，支持子 Agent，仍然可以运行

每一步都是有意义的里程碑，不是通往"最终版本"的过渡状态。

Claude Code 本身就是这样构建的——它的早期版本肯定远比现在简单，但在那个阶段它是完整的、可用的。
