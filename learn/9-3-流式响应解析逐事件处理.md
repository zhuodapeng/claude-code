# 9-3 流式响应解析：逐事件处理

## 本节要解决的问题

Anthropic API 的流式响应不是一个大 JSON——它是一连串 Server-Sent Events（SSE），每个事件是一个独立的 JSON 对象。这意味着接收端必须自己**拼装**完整的响应。

工程问题很具体：**如何把碎片化的流式事件，拼装成完整的 AssistantMessage，同时不阻塞 UI 的实时渲染？**

**文件：[src/services/api/claude.ts](../src/services/api/claude.ts#L1940-L2304)**

---

## 流式协议的结构

Anthropic 的流式 API 遵循这样的事件序列：

```
message_start          ← 响应开始，包含 requestId、初始 usage
  content_block_start  ← 一个 content block 开始（文本/工具/thinking）
    content_block_delta ← 内容的增量（文字片段/JSON 片段/thinking 片段）
    content_block_delta
    ...
  content_block_stop   ← content block 完成
  content_block_start  ← 下一个 block
    ...
  content_block_stop
message_delta          ← 响应结束信号，包含 stop_reason 和完整 usage
message_stop           ← 流彻底结束
```

**一次 Claude 响应可以包含多个 content block**，比如：
- 一个 `text` block（Claude 说"我来帮你读这个文件"）
- 一个 `tool_use` block（实际调用 Read 工具）
- 另一个 `text` block（"读取结果如下"）

---

## 核心数据结构：contentBlocks 数组

**文件：[src/services/api/claude.ts](../src/services/api/claude.ts#L1764)**

```typescript
const contentBlocks: (BetaContentBlock | ConnectorTextBlock)[] = []
let partialMessage: BetaMessage | undefined = undefined
const newMessages: AssistantMessage[] = []
```

`contentBlocks` 是一个**按 index 随机写入**的数组。每个事件携带 `index` 字段，指示对应的是哪个 content block。这允许多个 block 的事件交错到来（虽然实际上 API 是顺序发送的，但 index 设计保留了并行的可能性）。

---

## 六种事件类型的处理

### message_start

**文件：[src/services/api/claude.ts](../src/services/api/claude.ts#L1980-L1993)**

```typescript
case 'message_start': {
  partialMessage = part.message  // BetaMessage，usage/stop_reason 都是初始值
  ttftMs = Date.now() - start    // 计算 TTFB（Time to First Byte）
  usage = updateUsage(usage, part.message?.usage)
  break
}
```

这是流的第一个事件，到达时意味着服务器已开始处理请求（headers 已收到）。`ttftMs` 是从发出请求到收到第一个 content 的延迟，这是关键的性能指标。

### content_block_start

**文件：[src/services/api/claude.ts](../src/services/api/claude.ts#L1995-L2051)**

```typescript
case 'content_block_start':
  switch (part.content_block.type) {
    case 'text':
      contentBlocks[part.index] = {
        ...part.content_block,
        text: '',  // 初始化为空，后续 delta 来追加
      }
      break
    case 'tool_use':
      contentBlocks[part.index] = {
        ...part.content_block,
        input: '',  // 工具输入从空字符串开始累积
      }
      break
    case 'thinking':
      contentBlocks[part.index] = {
        ...part.content_block,
        thinking: '',
        signature: '',  // 初始化 signature，防止 signature_delta 未到达
      }
      break
    default:
      contentBlocks[part.index] = { ...part.content_block }
  }
  break
```

注意：**工具的 `input` 被初始化为字符串 `''`**，而不是对象 `{}`。这是因为工具输入在流式传输时是 JSON 字符串的形式——一个个 JSON 片段拼接起来，最后才解析。用字符串累积比实时解析 JSON 要高效得多。

### content_block_delta（核心）

**文件：[src/services/api/claude.ts](../src/services/api/claude.ts#L2053-L2169)**

```typescript
case 'content_block_delta': {
  const contentBlock = contentBlocks[part.index]
  const delta = part.delta
  
  switch (delta.type) {
    case 'text_delta':
      contentBlock.text += delta.text  // 文字追加
      break
    case 'input_json_delta':
      contentBlock.input += delta.partial_json  // JSON 片段追加（字符串拼接）
      break
    case 'thinking_delta':
      contentBlock.thinking += delta.thinking  // thinking 追加
      break
    case 'signature_delta':
      contentBlock.signature = delta.signature  // thinking 签名（原子赋值）
      break
  }
  break
}
```

这里的字符串拼接是故意的。大模型输出 JSON 工具参数时，会一个字符一个字符地流式输出：

```
partial_json: '{"f'
partial_json: 'ile_'
partial_json: 'path'
partial_json: '": "'
partial_json: 'src/'
...
```

每个 `partial_json` 都 `+= delta.partial_json`，最后在 `content_block_stop` 时一次性 `JSON.parse()`。这比在每个 delta 上做 partial JSON parsing 快得多——如代码注释所说，用 `BetaMessageStream` 那个方案是 O(n²) 的。

### content_block_stop（yield AssistantMessage）

**文件：[src/services/api/claude.ts](../src/services/api/claude.ts#L2171-L2212)**

```typescript
case 'content_block_stop': {
  const contentBlock = contentBlocks[part.index]
  
  const m: AssistantMessage = {
    message: {
      ...partialMessage,
      content: normalizeContentFromAPI(
        [contentBlock] as BetaContentBlock[],
        tools,
        options.agentId,
      ),  // ← 这里做 JSON.parse() 和类型检查
    },
    requestId: streamRequestId ?? undefined,
    type: 'assistant',
    uuid: randomUUID(),
    timestamp: new Date().toISOString(),
  }
  newMessages.push(m)
  yield m  // ← 立即 yield 给上层
  break
}
```

每个 content block 完成时，立即构建一个 `AssistantMessage` 并 yield 出去。`normalizeContentFromAPI()` 在这里做 JSON 解析：把 `input` 字符串变成对象 `{ file_path: 'src/...' }`。

**关键：每个 block 完成就立即 yield，而不是等全部 block 完成。** 这意味着 Claude 说完第一句话（第一个 text block），UI 就能立即显示——不需要等工具调用结束。

### message_delta（更新 usage 和 stop_reason）

**文件：[src/services/api/claude.ts](../src/services/api/claude.ts#L2213-L2293)**

```typescript
case 'message_delta': {
  usage = updateUsage(usage, part.usage)
  stopReason = part.delta.stop_reason  // 'end_turn' | 'tool_use' | 'max_tokens' | ...
  
  // 直接 mutation（不 clone）：写回已 yield 的 message
  const lastMsg = newMessages.at(-1)
  if (lastMsg) {
    lastMsg.message.usage = usage
    lastMsg.message.stop_reason = stopReason
  }
  
  // max_tokens 错误
  if (stopReason === 'max_tokens') {
    yield createAssistantAPIErrorMessage({
      content: `Claude's response exceeded the ${maxOutputTokens} output token maximum...`,
      apiError: 'max_output_tokens',
    })
  }
  
  break
}
```

这里有一个非常有趣的设计：**直接 mutation 已经 yield 出去的消息**。

为什么？因为 `message_delta` 携带的 `usage`（最终 token 计数）和 `stop_reason` 是在所有 content blocks 完成**之后**才到达的。但 `AssistantMessage` 在 `content_block_stop` 时就已经 yield 出去了，上层代码持有这个对象的引用。

如果用 `{ ...lastMsg.message, usage }` 替换整个对象，上层持有的旧引用不会更新。但**直接 mutation** 会：上层持有的引用 `m.message` 和这里的 `lastMsg.message` 是同一个对象，mutation 对双方都可见。

代码注释里明确说明了这个设计：

```typescript
// IMPORTANT: Use direct property mutation, not object replacement.
// The transcript write queue holds a reference to message.message and
// serializes it lazily (100ms flush interval). Object replacement would
// disconnect the queued reference; direct mutation ensures the transcript
// captures the final values.
```

transcript 写入队列是异步的（100ms 延迟 flush），如果在 flush 之前用对象替换，队列里的引用还是旧对象，usage 和 stop_reason 就丢了。

### message_stop

这个事件什么都不做，只是 break。它的到来标志着流彻底结束。

---

## 双 yield 机制

注意到了吗？`queryModel()` 有**两种 yield**：

```typescript
// 1. 在 content_block_stop 时：yield 一个完整的 AssistantMessage
yield m  // AssistantMessage

// 2. 每个事件都额外 yield 一次：原始事件
yield {
  type: 'stream_event',
  event: part,
  ...(part.type === 'message_start' ? { ttftMs } : undefined),
}
```

第一种 yield 是给上层的**业务逻辑**用的：`queryLoop()` 收到 `AssistantMessage`，检查里面是否有 `tool_use` block，决定是否继续循环。

第二种 yield（`stream_event`）是给 **UI 和 SDK 调用者**用的：TUI 渲染器可以用 `text_delta` 事件实时更新终端输出，SDK 调用者可以监听 `input_json_delta` 看到工具参数的实时形成过程。

两种 yield 的职责完全不同，一个是数据流，一个是事件流。

## 流事件处理的整体时序

```
API stream
   │
   ├── message_start ─────────────────────────────→ yield stream_event
   │   partialMessage = {...}
   │   ttftMs = 计算
   │
   ├── content_block_start(index=0, type='text') → yield stream_event
   │   contentBlocks[0] = { text: '' }
   │
   ├── content_block_delta(index=0, text_delta='我来') → yield stream_event
   │   contentBlocks[0].text += '我来'
   │
   ├── content_block_delta(index=0, text_delta='帮你') → yield stream_event
   │   contentBlocks[0].text += '帮你'
   │
   ├── content_block_stop(index=0) ──────────────→ yield stream_event
   │   m = AssistantMessage { text: '我来帮你' }
   │                        ──────────────────────→ yield m  ← 第一个 AssistantMessage
   │
   ├── content_block_start(index=1, type='tool_use') → yield stream_event
   │   contentBlocks[1] = { input: '' }
   │
   ├── content_block_delta(index=1, input_json_delta='{"f') → yield stream_event
   │   contentBlocks[1].input += '{"f'
   │
   ├── ... 更多 json delta ... ──────────────────→ yield stream_event (每个)
   │
   ├── content_block_stop(index=1) ──────────────→ yield stream_event
   │   input = JSON.parse('{"file_path": "..."}')
   │   m = AssistantMessage { tool_use: { name: 'Read', input: {...} } }
   │                        ──────────────────────→ yield m  ← 第二个 AssistantMessage
   │
   ├── message_delta(stop_reason='tool_use', usage={...}) → yield stream_event
   │   lastMsg.message.usage = usage    ← mutation
   │   lastMsg.message.stop_reason = 'tool_use'  ← mutation
   │
   └── message_stop ─────────────────────────────→ yield stream_event
```

## 本节小结

- 流式 API 的事件序列是：`message_start → (block 开始/增量/结束)* → message_delta → message_stop`
- 工具输入在流式传输时是 JSON 字符串片段，通过字符串拼接累积，在 `content_block_stop` 时一次性 parse
- 每个 block 完成后立即 yield `AssistantMessage`，UI 可以实时渲染，无需等待全部 block 完成
- `message_delta` 到达时做 direct mutation 更新已 yield 的消息，保证 transcript 写入队列持有的引用能看到最终 usage
- 存在两条并行的 yield 流：业务级 `AssistantMessage` 和原始 `stream_event`，分别服务于循环控制和 UI/SDK
