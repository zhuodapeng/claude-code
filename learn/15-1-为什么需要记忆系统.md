# 15-1 为什么需要记忆系统？

在所有关于 Claude Code 的功能特性中，记忆系统可能是最容易被忽视、却又最体现工程哲学的设计。它试图回答一个根本性的问题：**一个基于 LLM 的工具，如何跨越会话的边界，积累和保持对用户的了解？**

---

## 问题：LLM 是无状态的

每一次你启动 Claude Code，对 Claude 来说都是全新的开始。Claude 不记得你上周告诉它的"这个项目不要用 eslint --fix 来自动修复"，不记得你更喜欢先写测试再写实现，不记得这个代码库有一条特殊的 commit 规则，也不记得你是一个有 10 年 Java 经验、刚开始学 TypeScript 的后端工程师。

这不是 Claude 的 bug，这是 LLM 架构的根本特性。模型的知识来自训练数据，而你的个人偏好、项目背景、协作习惯，这些都不在训练数据里。每次 API 调用都是一个独立的上下文窗口，会话结束，上下文消失。

对于一个临时的问答工具来说，这无所谓。但 Claude Code 的定位是**长期的工程协作伙伴**。如果它每次都需要用户重新解释背景，每次都犯同样被纠正过的错误，它本质上就退化成了一个聪明但健忘的助手。

---

## 直觉方案为什么不够

最直觉的答案是：把历史对话都发给 Claude。这样它就"记得"了。

这个方案在短期内是可行的（这就是 Claude Code 的 `--continue` 模式做的事）。但它有三个问题：

**问题一：Token 窗口有限。** 你和 Claude 每天有数十次对话，几周下来历史消息可以轻易累积到几十万 token。而 Claude 的上下文窗口是有限的。你不可能把所有历史都塞进去。

**问题二：历史消息的信噪比差。** 对话历史里大量是一次性的任务细节（"帮我改这个文件的第 42 行"），真正需要在未来对话中起作用的只是少量的元信息（"这个用户偏好函数式风格"）。把所有历史消息都带上，是在用大量噪音淹没真正有价值的信号。

**问题三：跨项目无法共享。** 你在项目 A 里告诉 Claude 你是一个 Go 老手，在项目 B 里同样需要 Claude 知道这件事。历史消息是项目级别的，无法跨项目传播。

---

## Claude Code 的答案：文件系统作为记忆载体

Claude Code 的解法是一个相当务实的架构决策：**用文件系统来存储记忆**。

记忆以 Markdown 文件的形式存储在 `~/.claude/projects/<项目slug>/memory/` 目录下。每个记忆是一个独立的 `.md` 文件，有 YAML Frontmatter 标注元信息，内容是自然语言描述。

这个设计有几个非常自然的好处：

1. **无需数据库**：Markdown 文件是人类可读的，用户可以直接编辑、删除记忆内容。
2. **持久化不需要特殊机制**：文件系统天然持久化。
3. **Claude 本身就知道怎么操作文件**：通过 Write/Edit/Read 工具，Claude 可以自主读写自己的记忆，不需要额外的 API。
4. **可以被版本控制**：如果你愿意，可以把 `memory/` 目录放进 git 管理。

---

## 四类记忆的分类学设计

文件往哪放不是关键，关键是"什么应该被记住，什么不应该"。这个分类学设计是记忆系统的核心，体现在 [src/memdir/memoryTypes.ts](../src/memdir/memoryTypes.ts#L14-L21) 里：

```typescript
export const MEMORY_TYPES = [
  'user',
  'feedback',
  'project',
  'reference',
] as const
```

四种类型，每种类型回答一个不同的问题：

**`user` 类型**：这个用户是谁？
- 用户的技术背景（深度 Go 经验、刚学 React）
- 用户的角色和职责（数据科学家、Tech Lead）
- 用户的沟通偏好（喜欢简洁答案、偏好有完整上下文的解释）
- 用途：帮助 Claude 调整解释的深度和角度，不让老手等待入门科普，也不让新手被术语淹没

**`feedback` 类型**：这个用户希望 Claude 怎么工作？
- "不要在回复末尾总结你做了什么，我能看 diff"
- "不要 mock 数据库，我们上季度因为这个踩坑了"
- "这种重构用一个大 PR 而不是拆开，之前的经验验证了这个做法"
- 用途：让 Claude 在未来不重复已经被纠正的行为，以及复现已经被验证有效的行为。源码注释里有一句很关键：`Record from failure AND success` ——只记录纠正会导致 Claude 变得过于谨慎；也要记录被确认的正确做法

**`project` 类型**：这个项目目前发生了什么？
- "这周五之后不能合并非关键 PR，移动团队在切 release 分支"
- "重写认证中间件的原因是 legal 合规要求，不是技术债"
- 用途：帮助 Claude 理解工作背景，做出更符合实际情况的建议。项目记忆衰减快，所以 `body_structure` 要求包含 `Why:` 和 `How to apply:` 行，帮助未来的 Claude 判断这条记忆是否还有效

**`reference` 类型**：哪里能找到什么信息？
- "pipeline bug 在 Linear 的 INGEST 项目里追踪"
- "oncall 盯的 Grafana 看板在 grafana.internal/d/api-latency"
- 用途：作为指针，告诉 Claude 去哪里找最新的权威信息

## "不应该记忆什么"同样重要

[src/memdir/memoryTypes.ts:183-195](../src/memdir/memoryTypes.ts#L183-L195) 里 `WHAT_NOT_TO_SAVE_SECTION` 明确定义了禁止范围：

```
- Code patterns, conventions, architecture, file paths, or project structure — these can be derived by reading the current project state.
- Git history, recent changes, or who-changed-what — `git log` / `git blame` are authoritative.
- Debugging solutions or fix recipes — the fix is in the code; the commit message has the context.
- Anything already documented in CLAUDE.md files.
- Ephemeral task details: in-progress work, temporary state, current conversation context.
```

这个"禁止列表"的设计原则是：**可以从当前项目状态派生的内容，不需要记忆**。代码规范可以从代码里读出来，git 历史可以用 git log 查，bug 修复可以从 commit 看。记忆系统应该只存储无法从任何现有信息源派生的内容。

注释里还有一句语气很强的话：
> These exclusions apply even when the user explicitly asks you to save. If they ask you to save a PR list or activity summary, ask what was *surprising* or *non-obvious* about it — that is the part worth keeping.

这说明这条规则是评估（eval）验证过的——没有这条规定，Claude 会把大量噪音塞进记忆（比如"这周合并了 3 个 PR"这种无意义内容），而不是真正有价值的洞察。

## 记忆系统在架构中的位置

```
用户输入
    │
    ├── System Prompt 构建阶段
    │       │
    │       ├── getSystemContext()        → gitStatus
    │       └── getUserContext()          → CLAUDE.md + MEMORY.md 内容
    │                                         ↑
    │                                    记忆作为上下文注入
    │
    ├── Query Loop 执行阶段
    │       │
    │       └── 对话中产生的洞察（用户偏好、项目背景）
    │
    └── 每轮结束时 (handleStopHooks)
            │
            └── executeExtractMemories()
                    │
                    └── 后台提取 → 写入 memory/ 目录
```

注意这个架构的时序：记忆的**读取**发生在会话开始时（注入 System Prompt），记忆的**写入**发生在每轮对话结束时（后台提取）。两个方向是异步解耦的。

## 关于记忆系统的开关

[src/memdir/paths.ts:30-55](../src/memdir/paths.ts#L30-L55) 里 `isAutoMemoryEnabled()` 定义了记忆系统的优先级开关链：

```typescript
export function isAutoMemoryEnabled(): boolean {
  const envVal = process.env.CLAUDE_CODE_DISABLE_AUTO_MEMORY
  if (isEnvTruthy(envVal)) {
    return false   // 环境变量明确关闭
  }
  if (isEnvDefinedFalsy(envVal)) {
    return true    // 环境变量明确开启
  }
  if (isEnvTruthy(process.env.CLAUDE_CODE_SIMPLE)) {
    return false   // --bare 模式：精简模式，关闭
  }
  if (isEnvTruthy(process.env.CLAUDE_CODE_REMOTE) &&
      !process.env.CLAUDE_CODE_REMOTE_MEMORY_DIR) {
    return false   // CCR 远程模式但没有持久存储，关闭
  }
  const settings = getInitialSettings()
  if (settings.autoMemoryEnabled !== undefined) {
    return settings.autoMemoryEnabled  // 用户设置
  }
  return true  // 默认开启
}
```

优先级从高到低：环境变量 > 运行模式（bare/remote）> settings.json > 默认（开启）。这个优先级链是典型的"明确配置胜于隐式推断"模式。

---

## 本章小结

记忆系统解决的是 LLM 天然无状态的根本缺陷。Claude Code 的选择是：
1. 用文件系统作为记忆载体（人类可读、Claude 天然会操作）
2. 只记忆无法从项目状态派生的内容（4 种类型：用户/反馈/项目/引用）
3. 明确禁止记忆可派生内容（代码结构、git 历史、临时任务）
4. 读取（注入 System Prompt）和写入（后台提取）异步解耦
