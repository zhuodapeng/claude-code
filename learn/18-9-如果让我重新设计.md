# 18-9 如果让我重新设计

## 这一节做什么

我们花了整个课程理解 Claude Code 的设计。现在，站在"已经完全理解这个系统"的视角，问自己一个问题：

**如果从零重新设计，什么会不同？什么会保留？**

这不是批评 Claude Code——每一个设计决定都有它的历史背景和合理理由。但通过主动思考"我会做什么"，我们能更深刻地理解为什么现在是这样。

---

## 会保留的设计

### 1. React + Ink for TUI

这是 Claude Code 最出人意料的选择，也是最正确的选择。

用 React 的声明式模型渲染终端，让 UI 代码变成了普通的 React 代码。团队不需要学"终端渲染"——任何会 React 的工程师都能贡献 TUI 代码。

同时，React 的组件化让各部分的 UI 完全解耦：权限对话框、消息列表、工具输出——它们各自管好自己。

如果重新设计，我会做完全相同的选择。

### 2. queryLoop 作为状态机

把整个 AI 对话流程建模为状态机（idle → streaming → tool-executing → idle），而不是复杂的事件驱动系统，是一个非常清晰的设计。

状态机容易测试：你可以直接检查"处于什么状态"。状态机容易调试：出错时知道卡在哪个状态。状态机容易扩展：新增一个状态，对应一个新的 case。

### 3. Tool 的接口设计

```typescript
interface Tool {
  name: string
  description: string
  inputSchema: z.ZodSchema
  call(input: unknown): Promise<ToolResult>
  checkPermissions(input: unknown): Promise<PermissionResult>
}
```

把"执行"和"权限检查"放在同一个对象上，是正确的。工具知道自己有多危险——`readFile` 工具知道自己是只读的，`bash` 工具知道自己需要确认。把权限逻辑外置反而会造成信息不对称。

### 4. MCP 作为扩展机制

不把所有工具都内置，而是提供标准协议让外部服务器提供工具，这个方向是完全正确的。这是 Claude Code 未来可扩展性的基础。

## 会做不同的设计

### 1. 消息历史的管理

**现状**：AppState 是一个庞大的单一状态对象，消息历史存在里面，通过 `useSyncExternalStore` 连接 React。

**问题**：消息历史既有"用于 API 调用的格式"，又有"用于 UI 渲染的格式"，这两者的边界在代码里不清晰。你经常需要在 API MessageParam 和 UI DisplayMessage 之间转换。

**我的设计**：把这两者更明确地分开，用两个不同的数据层：
```typescript
// API 层（严格符合 Anthropic API 格式）
class ConversationHistory {
  getApiMessages(): MessageParam[]
  appendUser(content: string): void
  appendAssistant(response: Message): void
  appendToolResults(results: ToolResult[]): void
}

// UI 层（展示用，可以有额外字段）
class DisplayHistory {
  getDisplayMessages(): DisplayMessage[]
  // 由 ConversationHistory 的变化自动更新
}
```

这样两层各自有清晰的职责，不相互污染。

### 2. 权限系统的规则定义

**现状**：权限规则用字符串模式（`"bash:/usr/bin/*"`），解析逻辑分散在多个文件里。

**问题**：字符串解析容易出错，难以静态分析，IDE 无法给你补全。

**我的设计**：用 TypeScript 对象定义规则，而不是字符串：
```typescript
const rules: PermissionRule[] = [
  { tool: 'bash', pattern: /^ls\s/, decision: 'auto-approve' },
  { tool: 'bash', pattern: /rm\s+-rf/, decision: 'ask-user', alwaysAsk: true },
  { tool: 'read_file', decision: 'auto-approve' },
]
```

TypeScript 类型检查 + IDE 补全 + 可以在运行前验证所有规则，比字符串模式健壮得多。

### 3. 子 Agent 的上下文传递

**现状**：子 Agent 通过 `system prompt` 接收来自主 Agent 的上下文，格式是自由文本。

**问题**：没有结构化的约定——主 Agent 随意写，子 Agent 随意解析，容易出现信息丢失或格式不兼容。

**我的设计**：定义结构化的上下文传递格式：
```typescript
interface AgentContext {
  task: string                    // 必须：任务描述
  workingDirectory?: string       // 可选：工作目录
  relevantFiles?: string[]        // 可选：相关文件列表
  constraints?: string[]          // 可选：约束条件
  outputFormat?: 'text' | 'json' | 'markdown'  // 可选：期望的输出格式
}
```

子 Agent 知道父 Agent 传了什么，父 Agent 知道子 Agent 期望什么。

### 4. 配置系统

**现状**：配置读取逻辑分散在多个文件里，环境变量、`.claude.json`、`.mcp.json`、命令行参数——每种来源的读取代码在不同的地方。

**问题**：很难追踪"一个配置值最终是多少"，因为覆盖逻辑分散在各处。

**我的设计**：集中式配置层，在程序启动时一次性解析所有来源，合并成统一的配置对象：

```typescript
// config.ts - 程序启动时运行一次
const config = buildConfig({
  sources: [
    loadEnvVars(),           // 环境变量（最高优先级）
    loadCliArgs(),           // 命令行参数
    loadProjectConfig(),     // .claude.json（项目级）
    loadUserConfig(),        // ~/.claude.json（用户级）
    loadDefaults(),          // 默认值（最低优先级）
  ],
})

// 之后任何地方都从这个统一对象读
config.model          // 'claude-opus-4-6'
config.maxTokens      // 8096
config.mcpServers     // [...]
```

这样配置的优先级逻辑一目了然，不需要追踪多个文件。

## 历史包袱与不可避免的复杂性

有些设计看起来复杂，但复杂是有原因的：

### React Compiler 的编译输出

Claude Code 的源码经过 React Compiler 处理，原本清晰的代码变成了带 `$[0]`、`$[1]` 的缓存魔法：

```typescript
// 编译前（你写的代码）
function MCPConnectionManager({ children, dynamicMcpConfig }) {
  const { reconnectMcpServer, toggleMcpServer } = useManageMCPConnections(...)
  const value = useMemo(() => ({ reconnectMcpServer, toggleMcpServer }), [...])
  return <Context.Provider value={value}>{children}</Context.Provider>
}

// 编译后（源码里实际看到的）
export function MCPConnectionManager(t0) {
  const $ = _c(6)
  const { children, dynamicMcpConfig, isStrictMcpConfig } = t0
  // ...
  if ($[0] !== reconnectMcpServer || $[1] !== toggleMcpServer) {
    t1 = { reconnectMcpServer, toggleMcpServer }
    $[0] = reconnectMcpServer; $[1] = toggleMcpServer; $[2] = t1
  } else { t1 = $[2] }
  // ...
}
```

这不是坏设计——React Compiler 是正确的优化手段，减少不必要的渲染。但它让源码变得难以直接阅读。

如果重新设计，我会把 React Compiler 的处理放在构建步骤而不是源码层，让开发者看到和写到的是同一份代码。

### 2000 行的 client.ts

[src/services/mcp/client.ts](../src/services/mcp/client.ts) 是整个项目最长的文件，约 2000 行，处理了 MCP 的所有方面：连接、工具转换、资源列举、认证……

这是真实世界里的"边界腐蚀"——随着功能一点一点加入，单个文件越来越大，重构的代价越来越高，于是就这样了。

如果重新设计，我会把它拆成：
- `connection.ts`：建立/维护连接
- `toolAdapter.ts`：工具转换和 LRU 缓存
- `resourceAdapter.ts`：资源相关
- `auth.ts`：OAuth 和认证

每个文件职责单一，好找也好测试。

## 最重要的一个设计原则

纵观整个 Claude Code 的架构，有一个原则贯穿始终：

**在正确的地方写正确的代码，不要过早泛化。**

Claude Code 没有在"第 1 版"就把权限系统做得超级灵活——它先有了基本的 allow/deny，随着需求增加才有了更复杂的规则。

Claude Code 没有在"第 1 版"就支持 6 种 MCP 传输类型——可能先支持了 stdio，然后随着用户需求才加了 SSE、WebSocket。

**系统是生长出来的，不是设计出来的。**

你可以有一个好的初始架构（这很重要），但不要试图在最开始就预见所有需求。好的架构是容易修改的架构——当需求变化时，改动是局部的，不需要重写整个系统。

## 学完这门课，你现在能做什么

如果你从头认真读完了这 18 章，你现在应该能：

1. **读懂 Claude Code 的任何一段源码**——不是每一行都能立刻看懂，但你知道它在哪个模块、这个模块负责什么、它和其他模块怎么交互。

2. **在 Claude Code 里添加新功能**——你知道一个新工具应该怎么注册，一个新的 MCP 服务器应该怎么配置，一个新的 UI 组件应该怎么插入。

3. **设计一个类似的系统**——不是复制 Claude Code，而是用同样的设计思维——状态机、消息驱动、权限分层——构建你自己的 LLM 工具。

4. **理解 Claude Code 做的权衡**——每个设计决定都有代价和收益。你现在能看出来 Claude Code 在哪些地方选择了简单、在哪些地方选择了复杂，以及为什么。

## 最后的话

Claude Code 是一个真实的、生产级的系统。它有优雅的设计，也有历史包袱。有高明的抽象，也有"先凑合用"的权宜之计。

这才是真实的软件工程——不是教科书上完美的示例代码，而是在现实约束下做出的一个个具体决策。

读完这门课，你不是学会了"怎么用 Claude Code"——你学会了**怎么思考这类系统**。这种思维方式，在你下一个项目、下下个项目里，都会继续发挥作用。
