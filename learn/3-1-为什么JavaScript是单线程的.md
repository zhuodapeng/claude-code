# 3-1 为什么 JavaScript 是单线程的

> **本节目标**：从根本上理解 JavaScript 的执行模型——为什么是单线程、事件循环是什么、这对我们写代码意味着什么。这是后续理解 async/await、Promise、AsyncGenerator 的地基。不理解这一节，后面的所有异步内容都是空中楼阁。

---

## 从一个你可能没想过的问题开始

假设你在做一个网页，用户点了一个按钮，按钮要：
1. 发一个网络请求（需要 300ms）
2. 处理返回数据
3. 更新页面显示

如果 JavaScript 像 Java 一样，你可能会写：

```java
// 假设 Java 风格
String response = httpClient.get("https://api.example.com/data");  // 阻塞 300ms
Data data = parseResponse(response);
updateUI(data);
```

**问题来了**：在等待 300ms 的过程中，用户能滚动页面吗？能点击其他按钮吗？

答案是：**不能**。因为 JavaScript 是单线程的——只有一个线程在跑你的代码，如果这个线程在等网络请求，它就什么都做不了。

这就引出了 JavaScript 设计中最根本的问题：**如何在单线程里做"同时"做多件事？**

---

## JavaScript 为什么是单线程的？

这是一个历史和设计决策的问题。

### 起源：浏览器里的脚本语言

JavaScript 最初（1995年）被设计为**浏览器里的轻量脚本语言**，主要职责是：
- 响应用户点击
- 做简单的表单验证
- 偶尔修改一下页面内容

这个场景下，多线程的复杂性（锁、竞争条件、死锁）完全没必要。**单线程更简单、更安全**。

### 关键约束：DOM 不是线程安全的

浏览器里有一个更深层的原因：**DOM（页面结构）不是线程安全的**。

如果允许多线程同时修改 DOM，想象这个场景：
- 线程 A：删除一个 `<div>`
- 线程 B：同时在这个 `<div>` 里插入一个 `<span>`

谁先执行？结果不可预测，页面会崩溃。为了避免这类问题，浏览器选择了"只允许一个线程操作 DOM"。

### Node.js/Bun 继承了这个模型

后来的 Node.js（以及 Claude Code 用的 Bun）运行在服务端，没有 DOM，但继承了 JavaScript 的单线程模型——因为这个模型配合**事件循环**已经被证明可以高效处理大量并发 I/O。

---

## 单线程 + 事件循环 = 并发幻觉

现在来讲核心概念：**事件循环（Event Loop）**。

这是 JavaScript 解决"单线程也能处理多件事"的机制。

### 先理解两个区域

```
┌──────────────────────────────────────────────────────────┐
│                   JavaScript 运行时                       │
│                                                          │
│  ┌─────────────────┐         ┌───────────────────────┐   │
│  │    调用栈         │         │       任务队列          │   │
│  │  (Call Stack)   │         │    (Task Queue)        │   │
│  │                 │         │                        │   │
│  │  ← 当前在执行的  │         │  ← 等待被执行的回调     │   │
│  │    函数          │         │                        │   │
│  └─────────────────┘         └───────────────────────┘   │
│                                                          │
│  ┌────────────────────────────────────────────────────┐  │
│  │                  Web APIs / Node APIs               │  │
│  │  (计时器、网络请求、文件读写——真正的"等待"在这里发生)   │  │
│  └────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

**调用栈（Call Stack）**：JavaScript 线程当前正在执行的代码。就像 Java 里的方法调用栈——函数调用时压栈，函数返回时出栈。

**Web APIs / Node APIs**：浏览器或 Node/Bun 提供的异步操作能力，比如：
- `setTimeout`（计时器）
- `fetch`（网络请求）
- 文件系统操作（`fs.readFile`）

这些操作**不由 JavaScript 线程执行**，而是交给底层系统（C++ 层）去做。JavaScript 线程把任务"委托"出去，然后继续干其他事。

**任务队列（Task Queue）**：当异步操作完成后，其回调函数会放进这里等待执行。

**事件循环（Event Loop）**：一个无限循环，不断检查"调用栈是否为空？任务队列里有没有任务？如果有，把任务放进调用栈执行。"

## 用一个具体例子追踪执行过程

```javascript
console.log("1. 开始")

setTimeout(() => {
  console.log("3. 定时器回调")
}, 0)   // 注意：即使 delay 是 0ms，也不会立即执行！

console.log("2. 结束")
```

输出：
```
1. 开始
2. 结束
3. 定时器回调
```

为什么 "3. 定时器回调" 最后才输出，即使 delay 是 0？

让我们逐步追踪：

```
步骤 1：执行 console.log("1. 开始")
调用栈: [console.log]     任务队列: []
输出: "1. 开始"

步骤 2：执行 setTimeout(callback, 0)
调用栈: [setTimeout]      任务队列: []
→ 把 callback 委托给计时器 API
→ 即使 0ms，计时器完成后 callback 进入任务队列

步骤 3：执行 console.log("2. 结束")
调用栈: [console.log]     任务队列: [callback]
输出: "2. 结束"

步骤 4：调用栈清空，事件循环发现任务队列有任务
调用栈: []                任务队列: [callback]
→ 把 callback 移入调用栈

步骤 5：执行 callback
调用栈: [callback]        任务队列: []
输出: "3. 定时器回调"
```

**关键规律**：**异步回调永远不会"插队"到当前代码的中间执行**。它必须等到调用栈清空后，才能从任务队列里被取出执行。

## 同步代码阻塞会带来什么问题

理解了上面的机制，现在来理解"阻塞"的真正危害：

```javascript
// 这段代码会把 JavaScript 线程卡住 5 秒
function blockFor5Seconds() {
  const start = Date.now()
  while (Date.now() - start < 5000) {
    // 什么都不做，就是占着线程
  }
}

console.log("开始")
blockFor5Seconds()
console.log("5秒后才到这里")
```

这 5 秒里：
- 用户无法点击任何按钮
- 定时器的回调被堆在任务队列里，但没机会执行
- 网络请求的响应到了，但处理代码没机会跑
- 对于服务端（Node/Bun）：**新来的请求全部堵在外面**

这就是为什么 Node.js 开发里有一条铁律：**永远不要在主线程上做同步的 I/O 或长时间计算**。

## 那网络请求是怎么"不阻塞"的？

这是关键：当你调用 `fetch()` 时，发生了什么？

```javascript
console.log("1. 发请求前")

fetch("https://api.example.com/data")
  .then(response => response.json())
  .then(data => {
    console.log("3. 收到数据：", data)
  })

console.log("2. 发请求后，代码继续执行")
```

输出顺序：
```
1. 发请求前
2. 发请求后，代码继续执行
3. 收到数据：...（300ms 后）
```

流程：

```
JavaScript 线程                      底层系统（C++ 层）
      │
      │ 调用 fetch()
      │─────────────────────────────→  TCP 连接、发 HTTP 请求
      │                               （JavaScript 线程已经返回，继续执行下面的代码）
      │ 执行 console.log("2. ...")
      │
      │ 调用栈清空
      │                               ← 300ms 后，HTTP 响应到了
      │                               ← 把 .then 的回调放进任务队列
      │
事件循环检测到任务队列有任务
      │
      │ 执行 .then(response => ...)
      │ 执行 .then(data => ...)
      │ 输出 "3. 收到数据"
```

**JavaScript 线程在 300ms 等待期间根本没有等待——它在干别的事（或者空闲）。** 真正的等待发生在底层系统层面，跟 JavaScript 线程无关。

这就是为什么 Node.js/Bun 能用单线程处理大量并发请求——每个请求的"等待网络"时间里，线程可以处理其他请求。

## 微任务队列：比普通任务更高优先级

事件循环其实有两个队列，不是一个：

```
┌──────────────────────────────────────────┐
│              事件循环一次迭代              │
│                                          │
│  1. 执行调用栈上的代码（同步代码）          │
│                                          │
│  2. 清空微任务队列（Microtask Queue）      │
│     ← Promise 的 .then/.catch/.finally    │
│     ← queueMicrotask()                   │
│     ← await 后面的代码                    │
│                                          │
│  3. 执行一个宏任务（Macrotask）            │
│     ← setTimeout/setInterval 回调        │
│     ← I/O 回调                           │
│     ← setImmediate (Node.js)             │
│                                          │
│  4. 回到步骤 2                            │
└──────────────────────────────────────────┘
```

微任务队列的优先级高于宏任务队列。每次处理完一个宏任务，都要先把微任务队列清空。

这意味着：

```javascript
setTimeout(() => console.log("宏任务"), 0)

Promise.resolve().then(() => console.log("微任务 1"))
Promise.resolve().then(() => console.log("微任务 2"))

console.log("同步代码")
```

输出：
```
同步代码        ← 当前调用栈
微任务 1        ← 微任务队列（优先于宏任务）
微任务 2        ← 微任务队列继续清空
宏任务          ← 最后才是 setTimeout
```

**为什么要知道这个？** 因为 `await` 后面的代码以微任务的形式执行，理解这个能帮你调试一些奇怪的执行顺序问题。

## 这一切和 Claude Code 有什么关系？

Claude Code 是一个高度异步的系统。它需要同时：
- 等待 Anthropic API 返回流式 token
- 处理用户输入
- 执行工具调用（可能需要等待文件系统、网络）
- 刷新终端 UI

**所有这些都在单线程里"同时"发生**，靠的就是事件循环机制。

当你看到 Claude Code 的 `queryLoop` 里大量的 `await`：

```typescript
// src/query.ts
while (true) {
  const response = await callAPI(messages)   // 等待 API，但不阻塞线程
  
  for await (const event of response) {      // 处理流式事件，不阻塞
    yield event                              // 传递给上层，不阻塞
  }
  
  const toolResults = await runTools(...)    // 执行工具，不阻塞
}
```

每个 `await` 都是在说："我需要等这个结果，但请先去处理其他任务，等结果好了再回来。"这是事件循环机制在语言层面的体现，下一节会讲 `await` 背后的 Promise。

## 小结：建立正确的心智模型

一个清晰的类比：

> 把 JavaScript 想象成一个**餐厅里只有一个服务员**。
>
> - 服务员一次只能做一件事（单线程）
> - 客人点餐后，服务员把订单传给厨房，然后去招待下一桌（非阻塞 I/O）
> - 厨房做好了，服务员再回来上菜（回调/事件）
> - 服务员不会站在厨房门口干等（不阻塞）
>
> 这个服务员效率极高，能服务很多桌——只要他不傻站着等（不写同步阻塞代码）。

---

## 本节小结

- JavaScript 是单线程的，历史原因来自浏览器的 DOM 设计
- 单线程不等于"只能做一件事"——事件循环让 JavaScript 能处理大量并发 I/O
- 事件循环机制：调用栈清空后，从任务队列取出回调执行
- 异步操作（网络、文件、定时器）委托给底层系统，JavaScript 线程不等待
- 同步阻塞代码会卡住整个事件循环，是性能大忌
- 微任务队列（Promise）比宏任务队列（setTimeout）优先级高
