# 13-4 压缩后的消息结构

## 问题引入：压缩完成之后，消息列表变成什么样了？

前三节讲了压缩的触发和执行。这一节回答最后一个问题：**压缩完成之后，原来那几十条消息历史去哪了？新的消息列表长什么样？query 循环是怎么切换到新消息的？**

---

## CompactionResult：压缩操作的全部输出

[`compactConversation()`](../src/services/compact/compact.ts#L387) 的返回值就是 [`CompactionResult`](../src/services/compact/compact.ts#L299)：

**文件：[src/services/compact/compact.ts](../src/services/compact/compact.ts#L299-L310)**

```typescript
export interface CompactionResult {
  boundaryMarker: SystemMessage         // 标记"压缩发生在这里"的系统消息
  summaryMessages: UserMessage[]        // 摘要内容（包裹在用户消息里）
  attachments: AttachmentMessage[]      // 重新注入的文件/工具附件
  hookResults: HookResultMessage[]      // session_start hook 的输出
  messagesToKeep?: Message[]            // 部分压缩时保留的最近消息
  userDisplayMessage?: string           // 用户界面里显示的提示文字
  preCompactTokenCount?: number         // 压缩前的 token 数（遥测用）
  postCompactTokenCount?: number        // 压缩调用的总 token 数（遥测用，命名略有误导）
  truePostCompactTokenCount?: number    // 压缩后新消息序列的真实大小
  compactionUsage?: ...                 // 压缩 API 调用的 token 用量明细
}
```

注意 `postCompactTokenCount` 和 `truePostCompactTokenCount` 的区别（代码注释在 compact.ts:626-636 有解释）：
- `postCompactTokenCount`：实际上是压缩 API 调用的总用量，因为 API 调用的输入 ≈ 原始消息大小，所以它近似于压缩前的 token 数。字段名有些误导，但为了遥测数据的历史连续性，没有重命名。
- `truePostCompactTokenCount`：才是压缩后新消息序列的实际大小，是 `shouldAutoCompact()` 下一轮会用到的真正数字。

## buildPostCompactMessages：组装顺序有讲究

**文件：[src/services/compact/compact.ts](../src/services/compact/compact.ts#L330-L338)**

```typescript
export function buildPostCompactMessages(result: CompactionResult): Message[] {
  return [
    result.boundaryMarker,            // ① 压缩边界标记（最早）
    ...result.summaryMessages,        // ② 摘要文本
    ...(result.messagesToKeep ?? []), // ③ 保留的最近消息（部分压缩时）
    ...result.attachments,            // ④ 文件/工具附件
    ...result.hookResults,            // ⑤ hook 结果（最晚）
  ]
}
```

顺序是精确约定的，不能随意调换：

```
压缩前的消息历史（全部丢弃）：
  user: "帮我重构这个文件"          ← 被摘要替换
  assistant: "好的，我先读一下..."   ← 被摘要替换
  user: [tool_result: 文件内容]     ← 被摘要替换
  ...（几十条消息）

压缩后的新消息序列：

  [SystemMessage] boundaryMarker    ← "这里发生了压缩，之前有 N 个 token"
         ↓
  [UserMessage] summaryMessages[0]  ← "这是上下文摘要：这次会话目标是X，
                                       读了文件A/B/C，当前正在做Y..."
         ↓
  [AttachmentMessage] fileAttachment  ← src/foo.ts 的最新内容（最多 5 个文件）
  [AttachmentMessage] toolsAttachment ← 所有工具的声明（重新注入）
  ...
         ↓
  [HookResultMessage] hookResult    ← session_start hook 的输出
```

`boundaryMarker` 排在最前面，让 `getMessagesAfterCompactBoundary()` 能快速找到"这轮 API 调用应该从哪里开始"——它总是找最后一个 boundaryMarker，然后取后面的消息。

## query.ts 如何接收和切换消息

**文件：[src/query.ts](../src/query.ts#L528-L535)**

```typescript
// 1. 组装新消息序列
const postCompactMessages = buildPostCompactMessages(compactionResult)

// 2. 将每条新消息 yield 出去
//    REPL.tsx 收到这些消息后，会把它们插入到 messagesRef 里
//    并更新用户界面（显示"已压缩上下文"提示）
for (const message of postCompactMessages) {
  yield message
}

// 3. 同时更新本地的 messagesForQuery，让当前轮次直接使用新消息
messagesForQuery = postCompactMessages
```

`yield message` 的作用是双重的：
- 把新消息送给 REPL 组件，让界面更新
- 通过 generator 机制，让调用者（REPL 的 `submitMessage` 循环）能感知到"发生了压缩"

压缩完成后，**当前轮 API 调用继续执行**——使用的是 `postCompactMessages` 而不是原来的历史。这个转换对用户是透明的：他们发了一条消息，中间触发了压缩，然后 Claude 正常回复，整个过程就像没发生过一样。

## 用户看到的是什么？

在 REPL 界面里，用户会看到一条特殊的摘要消息，来自 [`getCompactUserSummaryMessage()`](../src/services/compact/prompt.ts#L337)：

```
This session is being continued from a previous conversation that ran out of context.
The summary below covers the earlier portion of the conversation.

Summary:
1. Primary Request and Intent:
   [用户的目标]
2. Key Technical Concepts:
   - [概念1]
   ...
8. Current Work:
   [压缩发生时正在做的事]
9. Optional Next Step:
   [下一步建议]

If you need specific details from before compaction (like exact code snippets, error messages,
or content you generated), read the full transcript at: /path/to/transcript.jsonl
```

这条消息的 `isCompactSummary: true` 和 `isVisibleInTranscriptOnly: true` 标志让它在 API 请求时包含（Claude 需要读摘要），但在 UI 里可以特殊显示（而不是普通的用户消息气泡）。

## 压缩后的追踪状态重置

成功压缩后，`AutoCompactTrackingState` 被重置：

**文件：[src/query.ts](../src/query.ts#L521-L526)**

```typescript
tracking = {
  compacted: true,          // 标记"本 session 发生过压缩"
  turnId: deps.uuid(),      // 新的唯一 ID，用于追踪后续轮次
  turnCounter: 0,           // 重置计数器——从压缩点重新开始计轮次
  consecutiveFailures: 0,   // 重置熔断器
}
```

`turnCounter` 从 0 重新计数，配合 `tengu_post_autocompact_turn` 事件，让 Anthropic 能分析"压缩后第 N 轮再次触发压缩"的分布——这是优化压缩时机（那个 13K buffer 常量是否合理）的核心数据。

## 完整的 Auto Compact 数据流

把四节内容串起来：

```
query 循环
  │
  ├─ microcompact (工具结果压缩)
  │
  ├─ deps.autocompact()
  │    │
  │    ├─ shouldAutoCompact()
  │    │    ├─ querySource 递归守卫
  │    │    ├─ isAutoCompactEnabled()
  │    │    └─ tokenCount >= autoCompactThreshold?
  │    │
  │    ├─ [熔断器] consecutiveFailures >= 3?
  │    │
  │    └─ compactConversation()
  │         ├─ stripImagesFromMessages()
  │         ├─ getCompactPrompt() → NO_TOOLS_PREAMBLE + 9节模板
  │         ├─ runForkedAgent(querySource='compact') → Claude 生成摘要
  │         ├─ formatCompactSummary() → 剥离 <analysis>
  │         ├─ createPostCompactFileAttachments() → 重注入读过的文件
  │         └─ 返回 CompactionResult
  │
  ├─ buildPostCompactMessages(compactionResult)
  │    → [boundaryMarker, summaryMessages, attachments, hookResults]
  │
  ├─ for message in postCompactMessages: yield message
  │    → REPL 更新界面
  │
  ├─ messagesForQuery = postCompactMessages
  │    → 当前轮继续，使用新消息
  │
  └─ 调用 Anthropic API（使用压缩后的消息）
```

---

## 本节小结

`CompactionResult` 是压缩操作的全部输出，包含五类消息。`buildPostCompactMessages()` 按固定顺序（边界标记 → 摘要 → 保留消息 → 附件 → hook结果）组装新消息序列。query.ts 通过 `yield` 把新消息送给 REPL 界面，同时用 `messagesForQuery = postCompactMessages` 让当前 API 调用切换到新历史。整个过程对用户透明：发一条消息，中间压缩，Claude 正常回复。
