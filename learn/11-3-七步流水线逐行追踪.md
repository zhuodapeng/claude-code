# 11-3 七步流水线逐行追踪

> `hasPermissionsToUseToolInner` 是权限系统的大脑。它接受一次工具调用，按顺序走七步，遇到明确结论就返回。本节逐行追踪这七步，理解为什么是这个顺序、为什么某几步无法被绕过。

---

## 问题引入：判断逻辑为什么要分步

最直觉的权限判断是 if-else 树：

```
if 有 deny 规则 → deny
else if 有 allow 规则 → allow
else → ask 用户
```

但真实场景比这复杂得多：
- 用户开了 YOLO 模式（--dangerously-skip-permissions），但某个工具明确说"我必须弹框"
- 用户配了 `ask` 规则，但同时开了 YOLO 模式——谁赢？
- 工具的 `checkPermissions()` 说 `deny`，但有全局 `allow` 规则——谁赢？

如果用 if-else 树，逻辑会互相缠绕、难以理解。流水线的好处是：**每一步要么直接返回，要么继续往下走**——逻辑是线性的，谁先谁后一目了然。

## 进入函数：定位起点

**文件：[src/utils/permissions/permissions.ts](../src/utils/permissions/permissions.ts#L1158-L1165)**

```typescript
async function hasPermissionsToUseToolInner(
  tool: Tool,
  input: { [key: string]: unknown },
  context: ToolUseContext,
): Promise<PermissionDecision> {
  if (context.abortController.signal.aborted) {
    throw new AbortError()   // 会话已中止，直接抛错
  }

  let appState = context.getAppState()  // 获取当前应用状态快照
```

`appState` 里包含了 `toolPermissionContext`，所有规则集就在这里。

七步从第 1171 行开始，按顺序执行。

---

## Step 1a：全局 deny 规则（第 1170 行）

**文件：[src/utils/permissions/permissions.ts](../src/utils/permissions/permissions.ts#L1170-L1181)**

```typescript
// 1a. Entire tool is denied
const denyRule = getDenyRuleForTool(appState.toolPermissionContext, tool)
if (denyRule) {
  return {
    behavior: 'deny',
    decisionReason: { type: 'rule', rule: denyRule },
    message: `Permission to use ${tool.name} has been denied.`,
  }
}
```

`getDenyRuleForTool` 只找**没有 `ruleContent` 的 deny 规则**——即工具级别的拒绝（`"Bash"` 这种，而不是 `"Bash(rm -rf)"` 这种）。

**为什么 deny 放第一步？**

Fail-closed 原则。明确被禁止的操作应该最先拦截，任何后续步骤（包括 bypass 模式）都不应该有机会绕过它。这是安全系统设计的基本准则。

如果 deny 放在 bypass 之后，`--dangerously-skip-permissions` 就能绕过用户的 deny 规则，这显然是错误的。

---

## Step 1b：全局 ask 规则（第 1183 行）

**文件：[src/utils/permissions/permissions.ts](../src/utils/permissions/permissions.ts#L1183-L1206)**

```typescript
// 1b. Check if the entire tool should always ask for permission
const askRule = getAskRuleForTool(appState.toolPermissionContext, tool)
if (askRule) {
  // 沙箱特殊情况：开了沙箱自动允许，且命令会被沙箱化 → 跳过 ask 规则
  const canSandboxAutoAllow =
    tool.name === BASH_TOOL_NAME &&
    SandboxManager.isSandboxingEnabled() &&
    SandboxManager.isAutoAllowBashIfSandboxedEnabled() &&
    shouldUseSandbox(input)

  if (!canSandboxAutoAllow) {
    return {
      behavior: 'ask',
      decisionReason: { type: 'rule', rule: askRule },
      message: createPermissionRequestMessage(tool.name),
    }
  }
  // Fall through to let Bash's checkPermissions handle command-specific rules
}
```

这里有个有趣的沙箱例外：如果 Bash 命令会在沙箱里执行（隔离环境），即使有工具级 ask 规则，也可以跳过弹框——因为沙箱本身提供了足够的隔离保护。

**ask 规则和 bypass 的关系**：这一步的 `ask` 不是 bypass 免疫的（Step 2a 的 bypass 排在它后面）。所以 `--dangerously-skip-permissions` 可以绕过工具级 ask 规则。这是有意设计的：工具级 ask 规则是用户设置"我想每次看到确认框"，但 YOLO 模式的意思正是"我授权跳过所有确认框"。

---

## Step 1c：委托给工具自己的 checkPermissions（第 1208 行）

**文件：[src/utils/permissions/permissions.ts](../src/utils/permissions/permissions.ts#L1208-L1223)**

```typescript
// 1c. Ask the tool implementation for a permission result
let toolPermissionResult: PermissionResult = {
  behavior: 'passthrough',           // 默认：无意见
  message: createPermissionRequestMessage(tool.name),
}
try {
  const parsedInput = tool.inputSchema.parse(input)
  toolPermissionResult = await tool.checkPermissions(parsedInput, context)
} catch (e) {
  if (e instanceof AbortError || e instanceof APIUserAbortError) {
    throw e   // AbortError 必须往上传，不能吞掉
  }
  logError(e) // 其他错误记录但继续，默认 passthrough
}
```

每个工具都可以实现自己的 `checkPermissions()` 方法做**内容级检查**：

```
BashTool.checkPermissions("npm install")
  → 检查 "npm install" 是否在 allow 规则里
  → 检查 "npm install" 是否在 deny 规则里
  → 检查工作目录限制
  → 返回 allow / deny / passthrough

WriteFileTool.checkPermissions("/etc/hosts")
  → 检查路径是否在 allowedDirectories 里
  → 检查是否是敏感路径（.git/, .claude/）
  → 返回 allow / ask(safetyCheck) / passthrough
```

`passthrough` 是工具的"无意见"信号：没有规则命中，让上层判断。

## Steps 1d-1g：bypass 免疫区（第 1225-1260 行）

这四步是权限系统最关键的设计：**任何模式（包括 bypassPermissions）都无法绕过这些检查**。

**文件：[src/utils/permissions/permissions.ts](../src/utils/permissions/permissions.ts#L1225-L1260)**

```typescript
// 1d. Tool implementation denied permission
if (toolPermissionResult?.behavior === 'deny') {
  return toolPermissionResult
  // 工具明确说"不"，bypass 无效
}

// 1e. Tool requires user interaction even in bypass mode
if (
  tool.requiresUserInteraction?.() &&
  toolPermissionResult?.behavior === 'ask'
) {
  return toolPermissionResult
  // AskUserQuestion 这类工具本质上需要用户参与
}

// 1f. Content-specific ask rules from tool.checkPermissions take precedence
// When a user explicitly configures a content-specific ask rule (e.g. Bash(npm publish:*))
// the tool's checkPermissions returns {behavior:'ask', decisionReason:{type:'rule',
// rule:{ruleBehavior:'ask'}}}. This must be respected even in bypass mode.
if (
  toolPermissionResult?.behavior === 'ask' &&
  toolPermissionResult.decisionReason?.type === 'rule' &&
  toolPermissionResult.decisionReason.rule.ruleBehavior === 'ask'
) {
  return toolPermissionResult
  // 用户明确配了内容级 ask 规则，bypass 也要弹框
}

// 1g. Safety checks are bypass-immune
// .git/, .claude/, .vscode/, shell configs — must prompt even in bypassPermissions mode.
if (
  toolPermissionResult?.behavior === 'ask' &&
  toolPermissionResult.decisionReason?.type === 'safetyCheck'
) {
  return toolPermissionResult
  // 安全路径：无论如何都要弹框
}
```

**为什么这四步必须在 bypass 之前？**

看注释里的解释（第 1239-1243 行）：内容级 ask 规则（如 `Bash(npm publish:*)`）是用户明确声明"我要每次确认这个命令"。如果 bypassPermissions 能绕过它，相当于 YOLO 模式暗中违背了用户的显式意图。

Step 1g（安全路径）的理由更直接：`~/.bashrc`、`.claude/settings.json` 这些文件被修改，可能被提示词注入利用来永久改变 Claude 的行为。即使用户开了 YOLO 模式，这类操作也必须确认。

## 四步 bypass 免疫区总结

```
┌─────────────────────────────────────────────────────────────┐
│              bypass 免疫区（Steps 1d-1g）                    │
│  ─────────────────────────────────────────────────────────  │
│  1d. 工具自己说 deny          → 无条件拒绝                    │
│  1e. 工具必须弹框             → 无条件弹框（如 AskUserQuestion）│
│  1f. 用户配了内容级 ask 规则   → 无条件弹框（尊重用户意图）    │
│  1g. 敏感安全路径             → 无条件弹框（防提示词注入）     │
└─────────────────────────────────────────────────────────────┘
            │
            ↓（全部通过后才能到这里）
┌─────────────────────────────────────────────────────────────┐
│              Step 2a：bypassPermissions 检查                 │
└─────────────────────────────────────────────────────────────┘
```

## Step 2a：bypassPermissions 模式绕过（第 1262 行）

**文件：[src/utils/permissions/permissions.ts](../src/utils/permissions/permissions.ts#L1262-L1281)**

```typescript
// 2a. Check if mode allows the tool to run
// IMPORTANT: Call getAppState() to get the latest value
appState = context.getAppState()   // 重新获取最新状态（用户可能切换了模式）

const shouldBypassPermissions =
  appState.toolPermissionContext.mode === 'bypassPermissions' ||
  (appState.toolPermissionContext.mode === 'plan' &&
    appState.toolPermissionContext.isBypassPermissionsModeAvailable)

if (shouldBypassPermissions) {
  return {
    behavior: 'allow',
    updatedInput: getUpdatedInputOrFallback(toolPermissionResult, input),
    decisionReason: {
      type: 'mode',
      mode: appState.toolPermissionContext.mode,
    },
  }
}
```

两个细节值得注意：

**注释"IMPORTANT: Call getAppState() to get the latest value"**：`appState` 是在函数开头读取的。但在进入函数到走到这一步之间，用户可能切换了权限模式（比如从 default 切到 bypassPermissions）。这里重新调用 `getAppState()` 确保拿到最新状态。

**Plan 模式的特殊处理**：Plan 模式通常不执行工具，但如果用户是用 `--dangerously-skip-permissions` 启动的，切换到 Plan 模式后，工具执行依然保持 bypass 行为。这通过 `isBypassPermissionsModeAvailable` 字段实现（第 11-6 节详讲）。

## Step 2b：整体 allow 规则检查（第 1283 行）

**文件：[src/utils/permissions/permissions.ts](../src/utils/permissions/permissions.ts#L1283-L1297)**

```typescript
// 2b. Entire tool is allowed
const alwaysAllowedRule = toolAlwaysAllowedRule(
  appState.toolPermissionContext,
  tool,
)
if (alwaysAllowedRule) {
  return {
    behavior: 'allow',
    updatedInput: getUpdatedInputOrFallback(toolPermissionResult, input),
    decisionReason: {
      type: 'rule',
      rule: alwaysAllowedRule,
    },
  }
}
```

`toolAlwaysAllowedRule` 只找**没有 `ruleContent` 的 allow 规则**——即 `"Read"` 这种整个工具级别的允许。

**为什么 allow 规则在 bypass 之后？**

这乍看很奇怪——bypass 模式的优先级比 allow 规则还高？

其实是为了代码简洁。bypass 模式里 allow 规则的结果是一样的（都允许）。更重要的是：这个顺序确保了 bypass 免疫区（1d-1g）对所有情况都生效——即使有全局 allow 规则，安全检查也不会被跳过（因为 1d-1g 在 2b 之前已经返回了）。

## Step 3：passthrough → ask 转换（第 1299 行）

**文件：[src/utils/permissions/permissions.ts](../src/utils/permissions/permissions.ts#L1299-L1318)**

```typescript
// 3. Convert "passthrough" to "ask"
const result: PermissionDecision =
  toolPermissionResult.behavior === 'passthrough'
    ? {
        ...toolPermissionResult,
        behavior: 'ask' as const,
        message: createPermissionRequestMessage(
          tool.name,
          toolPermissionResult.decisionReason,
        ),
      }
    : toolPermissionResult

return result
```

走到这里说明：没有任何规则命中，工具也没有明确意见（passthrough），也没有 bypass 模式，也没有全局 allow 规则。

**结论：需要询问用户。**

把 `passthrough` 转成 `ask` 是精心设计的语义区分：
- `passthrough`：工具说"我不知道，你来定"——不是允许，不是拒绝
- `ask`：系统的最终结论——需要人类判断

如果工具直接返回 `ask`，意思是"我认为这需要问"（有具体理由）。如果工具返回 `passthrough`，意思是"规则没命中，我没意见"——两者都会弹框，但语义和 `decisionReason` 不同。

## 完整流水线图

```
入口：hasPermissionsToUseToolInner(tool, input, context)
  │
  ├── 1a. getDenyRuleForTool()
  │         ├── 有 deny 规则 → return { behavior: 'deny' }
  │         └── 无 → 继续
  │
  ├── 1b. getAskRuleForTool()
  │         ├── 有 ask 规则 && !沙箱自动允许 → return { behavior: 'ask' }
  │         └── 无/沙箱 → 继续
  │
  ├── 1c. tool.checkPermissions(parsedInput, context)
  │         └── 得到 toolPermissionResult（passthrough / allow / ask / deny）
  │
  ├── ┌─────────────────── bypass 免疫区 ─────────────────────┐
  │   │ 1d. toolPermissionResult.behavior === 'deny'        │
  │   │         └── return toolPermissionResult              │
  │   │                                                      │
  │   │ 1e. tool.requiresUserInteraction() && 结果是 'ask'   │
  │   │         └── return toolPermissionResult              │
  │   │                                                      │
  │   │ 1f. decisionReason.type==='rule' && ruleBehavior==='ask'│
  │   │         └── return toolPermissionResult              │
  │   │                                                      │
  │   │ 1g. decisionReason.type === 'safetyCheck'           │
  │   │         └── return toolPermissionResult              │
  │   └──────────────────────────────────────────────────────┘
  │
  ├── 2a. shouldBypassPermissions？
  │         ├── 是 → return { behavior: 'allow', decisionReason: {type:'mode'} }
  │         └── 否 → 继续
  │
  ├── 2b. toolAlwaysAllowedRule()
  │         ├── 有 allow 规则 → return { behavior: 'allow', decisionReason: {type:'rule'} }
  │         └── 无 → 继续
  │
  └── 3. passthrough → ask 转换
            └── return { behavior: 'ask', message: ... }
```

## 一个具体例子：`git push --force` 走什么路径

假设用户配置了 `"Bash(git:*)"` allow 规则，Claude 要执行 `git push --force`：

```
Step 1a：getDenyRuleForTool("Bash") → null（没有工具级 deny）→ 继续
Step 1b：getAskRuleForTool("Bash") → null（没有工具级 ask）→ 继续

Step 1c：BashTool.checkPermissions("git push --force")
  → 检查 "git push --force" 是否匹配 deny 规则：无
  → 检查 "git push --force" 是否匹配 allow 规则："git:*" → 前缀匹配
  → 返回 { behavior: 'allow', decisionReason: {type:'rule', rule: ...} }

Step 1d：toolPermissionResult.behavior 是 'allow'，不是 'deny' → 继续
Step 1e：BashTool.requiresUserInteraction() 是 false → 继续
Step 1f：decisionReason.rule.ruleBehavior 是 'allow'，不是 'ask' → 继续
Step 1g：decisionReason.type 是 'rule'，不是 'safetyCheck' → 继续

Step 2a：没有 bypass 模式 → 继续

Step 2b：toolAlwaysAllowedRule() → null（Bash(git:*) 有 ruleContent，不是工具级规则）→ 继续

Step 3：toolPermissionResult.behavior 是 'allow'（非 passthrough）
  → return toolPermissionResult（允许执行）
```

命令执行了。因为 `Bash(git:*)` 规则匹配了 `git push --force`。

## 本节小结

七步流水线的顺序不是随意的，而是由安全优先级决定的：deny 最先（fail-closed），bypass 免疫区紧随其后（防绕过），然后才是 bypass 模式和 allow 规则，最后是 ask 兜底。每一步都有返回就不继续——这是线性决策的优雅之处。
