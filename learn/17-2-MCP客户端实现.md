# 17-2 MCP 客户端实现

## 问题：一个 connectToServer() 要处理多少种情况？

上一节我们看了 MCP 的六种传输类型和连接状态机。现在进入真正的实现核心：

- 同一个服务器配置被多次调用时，怎么避免重复连接？
- 连接超时了怎么处理？
- stdio 子进程退出时怎么清理？
- in-process 服务器为什么不启动子进程？

这些都在 [src/services/mcp/client.ts](../src/services/mcp/client.ts) 里，这个文件约 2000 行，是整个 MCP 子系统最重要的文件。

---

## connectToServer() 的签名和 memoize

**文件：[src/services/mcp/client.ts](../src/services/mcp/client.ts#L592)**

```typescript
export const connectToServer = memoize(
  async (
    name: string,
    serverRef: ScopedMcpServerConfig,
    agentId: string | undefined,
    skipPermissionCheck = false,
  ): Promise<MCPServerConnection> => {
    // ...
  },
  getServerCacheKey,   // 决定"什么算同一个服务器"的函数
)
```

`connectToServer` 是用 `memoize` 包裹的**记忆化异步函数**。

这意味着：用相同的参数调用两次 `connectToServer()`，第二次会直接返回第一次的 Promise，不会重新建立连接。

缓存 key 由 `getServerCacheKey()` 决定：

**文件：[src/services/mcp/client.ts](../src/services/mcp/client.ts#L1)**

```typescript
function getServerCacheKey(
  name: string,
  serverRef: ScopedMcpServerConfig,
  agentId: string | undefined,
): string {
  // 用 name + JSON(config) 作为 key
  // 同一个名字但不同配置 = 不同的缓存条目
  return `${name}:${JSON.stringify(serverRef)}:${agentId ?? ''}`
}
```

为什么要 memoize？

MCP 服务器连接是有代价的资源——启动子进程（stdio）、建立 TCP 连接（sse/ws）。Claude Code 在多个地方都可能需要"获取某个 MCP 服务器的连接"，如果每次都重新连接，代价太大。Memoize 确保了同一个服务器只连接一次。

## 连接超时保护

连接一个 MCP 服务器可能卡住——子进程无响应、网络不通……不能无限等待。

**文件：[src/services/mcp/client.ts](../src/services/mcp/client.ts#L1)**

```typescript
function getConnectionTimeoutMs(): number {
  const envVal = process.env.MCP_TIMEOUT
  if (envVal) {
    const parsed = parseInt(envVal, 10)
    if (!isNaN(parsed) && parsed > 0) return parsed
  }
  return 30000   // 默认 30 秒
}
```

连接使用 `Promise.race()` 实现超时：

```typescript
const timeoutPromise = new Promise<never>((_, reject) =>
  setTimeout(
    () => reject(new Error(`MCP server "${name}" connection timed out after ${timeoutMs}ms`)),
    timeoutMs,
  )
)

await Promise.race([connectPromise, timeoutPromise])
```

`Promise.race()` 是这里唯一合理的选择：我们想要"连接成功"或"超时"，哪个先发生就用哪个结果。

## 传输类型的分支创建

连接建立之前，要先根据 `serverRef.type` 创建对应的传输对象。这是一个大的 `switch`（或 `if-else if`）分支：

**文件：[src/services/mcp/client.ts](../src/services/mcp/client.ts#L616)**

```typescript
// stdio 类型：启动子进程
if (serverRef.type === 'stdio') {
  const { command, args, env, cwd } = serverRef
  transport = new StdioClientTransport({
    command,
    args,
    env: { ...process.env, ...env },   // 合并当前进程环境变量
    cwd,
  })
}

// SSE 类型：HTTP Server-Sent Events
else if (serverRef.type === 'sse') {
  const { url, headers } = serverRef
  transport = new SSEClientTransport(new URL(url), {
    eventSourceInit: {
      fetch: createAuthAwareFetch(url, headers),  // 带认证的 fetch
    },
    requestInit: {
      headers,
    },
  })
}

// WebSocket（IDE 扩展）
else if (serverRef.type === 'ws-ide') {
  transport = new WebSocketClientTransport(new URL(serverRef.url))
}

// 标准 WebSocket
else if (serverRef.type === 'ws') {
  transport = new WebSocketClientTransport(new URL(serverRef.url), {
    headers: serverRef.headers,
  })
}

// Streamable HTTP（支持流式的 HTTP）
else if (serverRef.type === 'http') {
  transport = new StreamableHTTPClientTransport(new URL(serverRef.url), {
    requestInit: {
      headers: serverRef.headers,
    },
  })
}
```

每种传输类型的构造方式不同，但它们最终都实现了同一个 Transport 接口——这是 MCP SDK 的设计，传输层对上面的 Client 完全透明。

## in-process 服务器：SDK 类型的特殊处理

对于 `sdk` 类型和两个内置的重量级服务器（Chrome MCP 和 Computer Use），Claude Code 使用 **in-process** 方式运行——不启动子进程，直接在同一个 Node.js/Bun 进程里创建 MCP server 和 client，用内存信道连接它们。

**文件：[src/services/mcp/client.ts](../src/services/mcp/client.ts#L907)**

```typescript
// Chrome MCP Server：进程内运行
if (isChromeMcpServer(serverRef)) {
  // createLinkedTransportPair() 返回两端连通的传输对
  // clientTransport 给 MCP client 用
  // serverTransport 给 MCP server 用
  const [clientTransport, serverTransport] = createLinkedTransportPair()

  // 创建并启动 server（进程内）
  const server = createChromeMcpServer()
  await server.connect(serverTransport)

  transport = clientTransport
}

// Computer Use Server：进程内运行
else if (isComputerUseMcpServer(serverRef)) {
  const [clientTransport, serverTransport] = createLinkedTransportPair()
  const server = createComputerUseMcpServer()
  await server.connect(serverTransport)
  transport = clientTransport
}
```

`createLinkedTransportPair()` 创建的是一对通过内存 channel 相互连接的传输对象——就像管道的两端，向一端写入的数据从另一端读出。

**为什么要进程内？**

注释里写得很直白：这些工具的初始化开销约 **325MB**。如果每次以 stdio 子进程启动，需要额外分配这 325MB 的内存，并等待进程启动时间。进程内运行共享同一个内存空间，这个开销消失了。

## Client 实例创建与能力协商

传输准备好之后，创建 MCP SDK 的 `Client` 实例：

**文件：[src/services/mcp/client.ts](../src/services/mcp/client.ts#L1)**

```typescript
const client = new Client(
  {
    name: 'claude-code',      // 客户端标识
    version: getVersion(),    // Claude Code 的版本号
  },
  {
    capabilities: {
      roots: {},              // 支持 roots（工作目录）能力
      elicitation: {},        // 支持 elicitation（请求用户输入）能力
    },
  },
)
```

然后调用 `client.connect(transport)`，这个调用会触发 MCP 的**初始化握手**：

```
Claude Code (Client)          MCP Server
       │                           │
       │──── initialize ──────────►│   发送客户端版本和能力
       │                           │
       │◄─── initialized ──────────│   服务端返回自己的版本和能力
       │                           │
       │      连接建立完成          │
```

握手完成后，`client.getServerCapabilities()` 返回服务端声明的能力（它支持哪些方法）。

## 连接成功：返回 ConnectedMCPServer

连接成功后，构建返回值：

**文件：[src/services/mcp/client.ts](../src/services/mcp/client.ts#L1)**

```typescript
return {
  name,
  type: 'connected',
  client,
  capabilities: client.getServerCapabilities() ?? {},
  config: serverRef,
  cleanup: async () => {
    // 清理函数：关闭连接，杀死子进程
    await client.close()
    await transport.close()
  },
}
```

注意 `cleanup` 字段——每个连接返回一个 cleanup 函数。调用方（通常是 session 结束时）负责调用它，确保资源被释放。这是 Go 风格的显式资源管理，而不是依赖 GC。

---

## stdio 子进程的清理序列

对于 `stdio` 类型的连接，cleanup 远比 `transport.close()` 复杂。子进程可能不响应关闭信号，需要强制杀死。

**文件：[src/services/mcp/client.ts](../src/services/mcp/client.ts#L1425)**

```typescript
async function cleanupStdioProcess(childProcess: ChildProcess): Promise<void> {
  // 第一步：发送 SIGINT（Ctrl+C），请求进程优雅退出
  childProcess.kill('SIGINT')

  await waitMs(100)   // 等 100ms，给进程机会优雅关闭

  if (!childProcess.killed) {
    // 第二步：进程还在，发送 SIGTERM
    childProcess.kill('SIGTERM')

    await waitMs(400)  // 再等 400ms

    if (!childProcess.killed) {
      // 第三步：进程还在，强制 SIGKILL（不可忽略）
      childProcess.kill('SIGKILL')
    }
  }
}
```

这是 Unix 进程管理的经典三段式：SIGINT → SIGTERM → SIGKILL，给进程逐步升级的信号。SIGKILL 不可被捕获或忽略，是最后手段。

100ms + 400ms 的等待时间是经验值：足够大多数进程完成清理，又不会让用户等太久。

## 连接失败：返回 FailedMCPServer

如果连接过程中抛出异常（超时、拒绝连接、认证失败……），`connectToServer` 捕获错误并返回失败状态：

**文件：[src/services/mcp/client.ts](../src/services/mcp/client.ts#L1)**

```typescript
try {
  await Promise.race([connectPromise, timeoutPromise])
  return { type: 'connected', ... }
} catch (error) {
  // OAuth 认证需要：返回 needs-auth
  if (isOAuthRequiredError(error)) {
    return {
      type: 'needs-auth',
      name,
      config: serverRef,
      authUrl: extractAuthUrl(error),
    }
  }

  // 其他错误：返回 failed
  return {
    type: 'failed',
    name,
    error: error instanceof Error ? error : new Error(String(error)),
    config: serverRef,
    lastAttempt: new Date(),
  }
}
```

注意 memoize 的行为：如果返回了 `failed` 状态，这个结果**也会被缓存**。下次调用同样参数的 `connectToServer()` 会直接返回缓存的 `failed`，不会重试。

这是有意为之的——避免服务器启动失败时的重试风暴。用户需要手动触发重连（通过 `reconnectMcpServer()`），这会清除缓存条目。

## reconnectMcpServer()：清除缓存强制重连

**文件：[src/services/mcp/MCPConnectionManager.tsx](../src/services/mcp/MCPConnectionManager.tsx#L1)**

```typescript
async function reconnectMcpServer(serverName: string) {
  // 清除 memoize 缓存中对应的条目
  connectToServer.cache.delete(
    getServerCacheKey(serverName, config, agentId)
  )

  // 重新调用，这次会真正建立新连接
  return connectToServer(serverName, config, agentId)
}
```

这就是 `MCPConnectionManager` 暴露出来的 `reconnectMcpServer` 函数的核心逻辑——先清缓存，再重连。

## MAX_MCP_DESCRIPTION_LENGTH：限制工具描述长度

**文件：[src/services/mcp/client.ts](../src/services/mcp/client.ts#L218)**

```typescript
const MAX_MCP_DESCRIPTION_LENGTH = 2048
```

MCP 服务器返回工具的 description 字段，但如果是 OpenAPI 自动生成的 MCP 服务器，工具描述可能极其冗长（完整的 OpenAPI 文档）。这 2048 字符的上限确保工具描述不会撑爆 system prompt 的 token 预算。

超出部分会被截断：

```typescript
const description = rawDescription?.length > MAX_MCP_DESCRIPTION_LENGTH
  ? rawDescription.slice(0, MAX_MCP_DESCRIPTION_LENGTH) + '...'
  : rawDescription
```

## 本章小结

`connectToServer()` 的核心设计：

1. **Memoize 避免重复连接**：同一配置只连一次，结果（成功或失败）都缓存
2. **Promise.race 超时保护**：默认 30 秒，可通过 `MCP_TIMEOUT` 环境变量覆盖
3. **Transport 工厂**：根据 `config.type` 分支创建，对上层完全透明
4. **in-process 优化**：重量级服务器（Chrome/Computer Use）在进程内运行，节省 325MB 开销
5. **三段式 cleanup**：SIGINT → SIGTERM → SIGKILL，100ms + 400ms 缓冲
