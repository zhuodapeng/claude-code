# 12-3 消息流的渲染：从 query loop 到终端显示

> 消息列表不在 AppState 里——它有独立的存储机制。本节追踪消息从 query loop 产生，到 REPL.tsx 存储，再到终端渲染的完整路径，重点解释"ref 作为真值源"模式以及为什么消息要单独管理。

---

## 问题引入：为什么消息不在 AppState 里

上一节看到 `AppState` 管理了大量 TUI 状态。直觉上，消息列表（Claude 的回复、用户的输入）也应该在 `AppState` 里。

为什么没有呢？

**消息列表的更新频率极高**：streaming 期间每个 token 都可能触发更新。如果消息在 AppState 里，每次 token 都调用 `store.setState()`，触发 `onChangeAppState` 副作用（权限模式检查、设置持久化……），这些副作用的开销在 streaming 期间完全不必要。

**消息只在 REPL 组件内使用**：消息列表没有被其他组件通过 AppState 读取。把它放在 `AppState` 里会给所有订阅了 `tasks`、`verbose` 等字段的组件增加无谓的重渲染压力。

**解法：消息列表是 REPL 的本地 `useState`**，配合一个 ref 作为同步真值源。

---

## 消息的存储：本地 useState + ref 双轨

**文件：[src/screens/REPL.tsx](../src/screens/REPL.tsx#L1183-L1184)**

```typescript
const [messages, rawSetMessages] = useState<MessageType[]>(initialMessages ?? [])
const messagesRef = useRef(messages)
```

两个变量各有用途：

| 变量 | 类型 | 用途 |
|------|------|------|
| `messages` | `MessageType[]` (React state) | 触发 React 重渲染，驱动 UI |
| `messagesRef.current` | `MessageType[]` (ref) | 同步读取，不触发重渲染 |

问题来了——为什么需要 ref？直接用 `messages` 不行吗？

**React state 有"异步更新"问题**。考虑这个场景：

```typescript
// 连续多次调用 setMessages，React 可能批量处理这些更新
setMessages(prev => [...prev, messageA])   // 还没处理
setMessages(prev => [...prev, messageB])   // 还没处理
// 然后同步读取 messages — 这里 messages 仍然是 "旧值"！
const current = messages  // ← 可能是旧快照
```

`messagesRef` 解决了这个问题：**ref 在每次 setMessages 调用时立即同步更新**，不等 React 批处理。

## setMessages 的自定义包装器

**文件：[src/screens/REPL.tsx](../src/screens/REPL.tsx#L1199-L1223)**

```typescript
const setMessages = useCallback((action: React.SetStateAction<MessageType[]>) => {
  const prev = messagesRef.current
  
  // 立刻更新 ref（同步，不等 React）
  const next = typeof action === 'function' ? action(messagesRef.current) : action
  messagesRef.current = next
  
  // 处理消息数组缩短的情况（compact/rewind/clear）
  if (next.length < userInputBaselineRef.current) {
    userInputBaselineRef.current = 0
  } else if (next.length > prev.length && userMessagePendingRef.current) {
    // 消息数增加时，检查用户消息是否已落地
    const added = next.slice(-delta)
    if (added.some(isHumanTurn)) {
      userMessagePendingRef.current = false  // 用户消息落地了
    } else {
      userInputBaselineRef.current = next.length  // 调整基线
    }
  }
  
  // 通知 React 重渲染（可能被批处理）
  rawSetMessages(next)
}, [])
```

代码注释里明确写着：

> This is the Zustand pattern: ref is source of truth, React state is the render projection.

这是 Zustand 状态库的核心思想：**ref 是真值源，React state 是渲染投影**。ref 立刻反映最新值，React state 则在合适时机触发重渲染。

## 消息的产生：query loop 的 yield 链

消息从哪里来？回到 query loop（第 9 章的主题），它是一个 `AsyncGenerator`，不断 yield 消息：

```
query.ts 的 queryLoop()
        │
        │  yield { type: 'assistant', message: {...} }
        │  yield { type: 'stream_event', event: {...} }
        │  yield { type: 'system', ... }
        ▼
QueryEngine.ts 的 submitMessage()
        │
        │  for await (const message of queryLoop(...))
        │    yield message  (透传给调用者)
        ▼
REPL.tsx 的 onQuery()
        │
        │  for await (const message of queryEngine.submitMessage(...))
        │    onQueryEvent(message)  ← 每个消息都调用这个
        ▼
onQueryEvent → handleMessageFromStream() → setMessages()
```

关键函数在 [src/screens/REPL.tsx](../src/screens/REPL.tsx#L2583-L2659) 的 `onQueryEvent`：

```typescript
const onQueryEvent = useCallback(
  (event: Parameters<typeof handleMessageFromStream>[0]) => {
    handleMessageFromStream(
      event,
      // 回调1：完整消息落地（非流事件）
      newMessage => {
        if (isCompactBoundaryMessage(newMessage)) {
          // 压缩边界：替换整个消息列表
          setMessages(() => [newMessage])
        } else if (newMessage.type === 'progress' && isEphemeralToolProgress(...)) {
          // 临时进度：替换最后一条（不累积）
          setMessages(oldMessages => {
            const last = oldMessages.at(-1)
            if (last?.type === 'progress' && ...) {
              const copy = oldMessages.slice()
              copy[copy.length - 1] = newMessage
              return copy
            }
            return [...oldMessages, newMessage]
          })
        } else {
          // 普通消息：追加
          setMessages(oldMessages => [...oldMessages, newMessage])
        }
      },
      // 回调2：streaming 文本增量（下一节详解）
      newContent => setResponseLength(length => length + newContent.length),
      // 回调3：设置 spinner 状态
      setStreamMode,
      // 回调4：更新 streaming 工具调用列表
      setStreamingToolUses,
      // 回调5：消息被 tombstone（删除）
      tombstonedMessage => {
        setMessages(old => old.filter(m => m !== tombstonedMessage))
      },
      // 回调6：thinking 块更新
      setStreamingThinking,
      // 回调7：streaming 文本更新（字符级）
      onStreamingText,
    )
  },
  [setMessages, ...]
)
```

## handleMessageFromStream 的分发逻辑

**文件：[src/utils/messages.ts](../src/utils/messages.ts#L2930-L2982)**

```typescript
export function handleMessageFromStream(message, onMessage, onUpdateLength, ...) {
  // 非流事件（完整消息）：直接交给 onMessage
  if (message.type !== 'stream_event' && message.type !== 'stream_request_start') {
    onStreamingText?.(() => null)  // ← 先清除 streaming 文本
    onMessage(message)             // ← 再落地完整消息
    return
  }

  // 流事件：根据 event.type 分发
  switch (message.event.type) {
    case 'content_block_delta':
      switch (message.event.delta.type) {
        case 'text_delta':
          // ← 字符级更新，下一节讲
          onUpdateLength(deltaText)
          onStreamingText?.(text => (text ?? '') + deltaText)
          return
        case 'input_json_delta':
          // ← 工具调用参数流
          onStreamingToolUses(...)
          return
      }
    case 'message_stop':
      onSetStreamMode('tool-use')    // spinner 切换到工具执行状态
      onStreamingToolUses(() => [])  // 清空 streaming 工具列表
      return
  }
}
```

关键设计：**清除 streaming 文本先于落地完整消息**（`onStreamingText?.(() => null)` 在 `onMessage(message)` 之前）。这是为了让 React 在同一个批次里：先看到 streaming 文本消失，再看到完整消息出现，实现"无缝切换"，避免两者同时出现的闪烁。

## 消息的渲染：Messages 组件

消息列表从 REPL 传给 `Messages` 组件：

```
REPL.tsx
  messages (useState)
        │
        ▼
<Messages
  messages={messages}
  streamingText={visibleStreamingText}   // 正在 streaming 的文字
  streamingToolUses={streamingToolUses}  // 正在 streaming 的工具调用
  streamingThinking={streamingThinking}  // 正在 streaming 的思考块
/>
```

`visibleStreamingText` 有一个特殊处理：

```typescript
const visibleStreamingText = streamingText && showStreamingText
  ? streamingText.substring(0, streamingText.lastIndexOf('\n') + 1) || null
  : null
```

它只显示**完整行**（`lastIndexOf('\n')` 截断到最后一个换行符前）。为什么？

如果按字符显示，用户会看到文字一个字一个字往右延伸，再折行，再延伸——这种动画效果在高速 streaming 时会剧烈闪烁。**只显示已完成的行**，每次"刷新"都是一整行新增，视觉上更稳定。

## 消息类型系统

消息是一个大 Union 类型（来自 `src/types/message.js`）：

```typescript
type Message =
  | UserMessage          // { type: 'user', message: {...} }
  | AssistantMessage     // { type: 'assistant', message: { content: ContentBlock[] } }
  | SystemMessage        // { type: 'system', level: 'info'|'warning'|'error', content }
  | ProgressMessage      // { type: 'progress', data: { type: '...' }, parentToolUseID }
  | AttachmentMessage    // { type: 'attachment', ... }
  | TombstoneMessage     // { type: 'tombstone', message: Message }
```

渲染时，`Messages` 组件通过 `message.type` 区分并调用不同的子组件：

```
messages.map(msg =>
  msg.type === 'assistant'  → <AssistantMessage />
  msg.type === 'user'       → <UserMessage />
  msg.type === 'system'     → <SystemMessage level={...} />
  msg.type === 'progress'   → <ProgressMessage />
)
```

---

## Tombstone 机制：消息的"撤销"

`TombstoneMessage` 是一个特殊类型——它不是要显示的内容，而是一个"删除指令"：

```typescript
tombstonedMessage => {
  setMessages(old => old.filter(m => m !== tombstonedMessage))
  void removeTranscriptMessage(tombstonedMessage.uuid)
}
```

当 query loop 里某个消息需要被撤销（如中断时的半截 assistant 消息），它 yield 一个 `{ type: 'tombstone', message: originalMessage }`。`handleMessageFromStream` 收到后不走 `onMessage`，而是走 `onTombstone`，把原始消息从数组里过滤掉。

## 完整流程图

```
Claude API 响应流
        │
        │  SSE: content_block_delta / message_stop / ...
        ▼
query.ts 的流解析
  └── yield { type: 'stream_event', event: {...} }  (流中)
  └── yield { type: 'assistant', message: {...} }   (完整消息)
        │
        ▼
REPL.tsx 的 onQueryEvent()
        │
        ▼
handleMessageFromStream() 分发
  ├── stream_event → onStreamingText(f) → setStreamingText
  │                   onStreamingToolUses(f) → setStreamingToolUses
  │                   onSetStreamMode → setStreamMode
  │
  └── 完整消息 → onMessage(msg) → setMessages(prev => [...prev, msg])
                                 messagesRef.current = [...prev, msg]
                                 rawSetMessages(next)  ← React 批处理
        │
        ▼
React 重渲染 REPL → 传给 Messages 组件 → 终端输出
```

## 本节小结

消息列表放在 REPL 的本地 `useState`（而非 AppState），避免高频更新触发全局副作用。`setMessages` 的自定义包装器用"ref 作为真值源"模式解决 React 异步批处理的一致性问题。`handleMessageFromStream` 是分发器：流事件更新临时状态（streaming text、spinner 模式），完整消息落地到数组。`TombstoneMessage` 提供了消息"撤销"机制。
