# 17-1 MCP 协议是什么

## 问题：Claude 怎么连接到外部世界？

Claude 本身只能处理文本。你给它一段代码，它能分析；你让它查天气，它做不到——除非有人把"查天气"这件事包装成一个工具喂给它。

内置工具（Bash、Read、Grep……）解决了本地文件系统的问题。但外部服务呢？数据库？企业内部系统？你自己写的 API？

如果每次都需要修改 Claude Code 的源码才能接入一个新服务，这个系统显然无法扩展。

**MCP（Model Context Protocol）**就是为这个问题而生的——它是一套标准协议，让 Claude 能够通过统一的方式连接到任意外部工具服务器，不需要修改 Claude 自身的代码。

---

## 先来理解类比：MCP 是 AI 世界的 LSP

如果你熟悉 IDE 开发，你一定听说过 **LSP（Language Server Protocol）**。

VSCode 怎么实现对几十种编程语言的智能补全？它没有把每种语言的补全逻辑都写在 VSCode 里——而是定义了一套协议，每种语言自己实现一个"语言服务器"，VSCode 通过 LSP 协议与它们通信。Rust 有 rust-analyzer，Python 有 pylsp，Go 有 gopls……

```
VSCode (客户端)  <──── LSP 协议 ────>  rust-analyzer (服务端)
                                       pylsp (服务端)
                                       gopls (服务端)
```

MCP 的思路完全一样：

```
Claude Code (客户端)  <──── MCP 协议 ────>  天气服务 MCP Server
                                             数据库查询 MCP Server
                                             企业内部 API MCP Server
```

**协议标准化 = 解耦 = 可扩展**。这是工程上最经典的设计模式之一。

---

## MCP 的技术基础：JSON-RPC 2.0

MCP 在传输层之上使用 **JSON-RPC 2.0** 作为消息格式。

JSON-RPC 2.0 非常简单：每条消息是一个 JSON 对象，有三种类型：

**请求（Request）**：
```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/list",
  "params": {}
}
```

**响应（Response）**：
```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "tools": [
      {
        "name": "get_weather",
        "description": "获取指定城市的天气",
        "inputSchema": { ... }
      }
    ]
  }
}
```

**通知（Notification）**——没有 `id`，不需要响应：
```json
{
  "jsonrpc": "2.0",
  "method": "notifications/claude/channel",
  "params": { ... }
}
```

为什么选 JSON-RPC 2.0 而不是 REST？因为 MCP 需要**双向通信**——服务端可以主动向客户端推送通知（比如 Channel 系统里 IM 消息推送进来）。REST 是单向请求-响应模型，做不到这一点。

---

## 传输层：六种连接方式

协议（JSON-RPC 2.0）解决了消息格式，传输层解决消息怎么走的问题。

Claude Code 支持六种传输类型，在 [src/services/mcp/types.ts](../src/services/mcp/types.ts#L1) 中用 Zod schema 定义：

**文件：[src/services/mcp/types.ts](../src/services/mcp/types.ts#L1)**

```typescript
export const TransportSchema = lazySchema(() =>
  z.enum(['stdio', 'sse', 'sse-ide', 'http', 'ws', 'sdk']),
)
```

六种传输类型的对应场景：

| 类型 | 全称 | 适用场景 |
|------|------|----------|
| `stdio` | Standard I/O | 本地子进程，最常见 |
| `sse` | Server-Sent Events | 远程 HTTP 服务器，单向推送 |
| `sse-ide` | SSE (IDE) | 从 IDE 扩展接入 |
| `http` | Streamable HTTP | 支持流式传输的 HTTP 服务器 |
| `ws` | WebSocket | 双向实时通信 |
| `sdk` | In-process SDK | 进程内运行，无需子进程 |

还有一个特殊类型 `claudeai-proxy`——这是 claude.ai 官方连接器专用的代理传输，源码里额外处理：

**文件：[src/services/mcp/types.ts](../src/services/mcp/types.ts#L1)**

```typescript
export const McpClaudeAIProxyServerConfigSchema = lazySchema(() =>
  z.object({
    type: z.literal('claudeai-proxy'),
    claudeAiMcpUrl: z.string(),    // 连接到 claude.ai 的代理 URL
    claudeAiUserId: z.string(),
    // ...
  })
)
```

## 六种 Server Config 的数据结构

每种传输类型对应一个独立的配置结构。这是 MCP 架构的核心约定——**不同传输类型的配置不兼容，靠 `type` 字段区分**。

看 `stdio` 类型（最常见）：

**文件：[src/services/mcp/types.ts](../src/services/mcp/types.ts#L1)**

```typescript
export const McpStdioServerConfigSchema = lazySchema(() =>
  z.object({
    type: z.literal('stdio'),
    command: z.string(),          // 要运行的可执行文件，如 "npx"
    args: z.array(z.string()),    // 参数列表，如 ["@modelcontextprotocol/server-filesystem"]
    env: z.record(z.string()).optional(),  // 额外环境变量
    cwd: z.string().optional(),   // 工作目录
  })
)
```

看 SSE 类型（远程服务器）：

```typescript
export const McpSSEServerConfigSchema = lazySchema(() =>
  z.object({
    type: z.literal('sse'),
    url: z.string(),              // 服务器 URL
    headers: z.record(z.string()).optional(),  // 认证 headers 等
  })
)
```

这六种 config 通过联合类型合并：

```typescript
export type McpServerConfig =
  | McpStdioServerConfig
  | McpSSEServerConfig
  | McpHTTPServerConfig
  | McpWebSocketServerConfig
  | McpSdkServerConfig
  | McpClaudeAIProxyServerConfig
```

当代码需要处理"任意一种 MCP 服务器配置"时，用 `McpServerConfig`；用 TypeScript 的 discriminated union（靠 `type` 字段区分），编译器帮你做穷举检查。

## Server 的生命周期状态机

一个 MCP 服务器连接不是二元的（连接/未连接）——它有多种中间状态。

**文件：[src/services/mcp/types.ts](../src/services/mcp/types.ts#L1)**

```typescript
// 已成功连接
interface ConnectedMCPServer {
  type: 'connected'
  name: string
  client: Client            // MCP SDK 的 Client 实例
  capabilities: ServerCapabilities
  config: ScopedMcpServerConfig
  cleanup: () => Promise<void>  // 清理函数，关闭连接释放资源
  tools?: Tool[]
  resources?: ServerResource[]
}

// 连接失败
interface FailedMCPServer {
  type: 'failed'
  name: string
  error: Error
  config: ScopedMcpServerConfig
  lastAttempt: Date
}

// 需要认证（OAuth 等）
interface NeedsAuthMCPServer {
  type: 'needs-auth'
  name: string
  config: ScopedMcpServerConfig
  authUrl?: string
}

// 等待中（初始状态）
interface PendingMCPServer {
  type: 'pending'
  name: string
  config: ScopedMcpServerConfig
}

// 已被用户禁用
interface DisabledMCPServer {
  type: 'disabled'
  name: string
  config: ScopedMcpServerConfig
}

export type MCPServerConnection =
  | ConnectedMCPServer
  | FailedMCPServer
  | NeedsAuthMCPServer
  | PendingMCPServer
  | DisabledMCPServer
```

状态流转：

```
pending
   │
   ▼ connectToServer() 被调用
connecting (内部过渡状态)
   │
   ├──── 连接成功 ──────► connected
   │
   ├──── 需要 OAuth ────► needs-auth ──── 用户完成授权 ──► connected
   │
   └──── 连接失败 ──────► failed

connected ──── 用户手动禁用 ──► disabled
```

这个设计的精妙之处：UI 层不需要关心"为什么没连上"——只要看 `type` 字段，就知道该展示什么状态给用户（正在连接、需要授权、连接失败……）。

## ScopedMcpServerConfig：加了一个 scope 字段

Claude Code 有多个配置来源——企业部署、用户的 `~/.claude.json`、项目级 `.mcp.json`、命令行参数……同一个 MCP 服务器可能在多个地方都有配置，谁的优先级高？

这就是 `ScopedMcpServerConfig` 存在的意义：

**文件：[src/services/mcp/types.ts](../src/services/mcp/types.ts#L1)**

```typescript
export type ScopedMcpServerConfig = McpServerConfig & {
  scope: ConfigScope       // 来自哪个配置层级
  pluginSource?: string    // 如果来自插件，是哪个插件
}

export type ConfigScope =
  | 'enterprise'   // 企业策略，最高优先级
  | 'managed'      // 托管配置
  | 'user'         // 用户全局配置 (~/.claude.json)
  | 'project'      // 项目配置 (.mcp.json)
  | 'dynamic'      // 运行时动态注册
  | 'claudeai'     // claude.ai 官方连接器
```

优先级从高到低：`enterprise > managed > user > project > dynamic > claudeai`

当两个地方都配置了同名服务器时，高优先级的配置胜出。具体逻辑在 [src/services/mcp/config.ts](../src/services/mcp/config.ts) 的 `dedupPluginMcpServers()` 函数中实现，我们在 17-4 节讲插件时详细看。

## MCP 核心 RPC 方法

当连接建立后，Claude Code 和 MCP 服务器之间主要用以下 RPC 方法通信：

| 方向 | 方法 | 用途 |
|------|------|------|
| 客户端 → 服务端 | `tools/list` | 获取服务器提供的工具列表 |
| 客户端 → 服务端 | `tools/call` | 调用某个工具，传入参数 |
| 客户端 → 服务端 | `resources/list` | 获取资源列表（文件、数据等） |
| 客户端 → 服务端 | `resources/read` | 读取某个资源的内容 |
| 服务端 → 客户端 | `notifications/tools/list_changed` | 工具列表变化了 |
| 服务端 → 客户端 | `notifications/claude/channel` | Channel 系统：IM 消息推入 |
| 双向 | `ping` | 心跳检测 |

Claude Code 的 MCP client 基于 `@modelcontextprotocol/sdk` 这个官方 SDK 构建——它封装了 JSON-RPC 2.0 的底层细节，暴露出来的是 `client.listTools()`、`client.callTool()` 这样的高层 API。

## 为什么有这么多传输类型？

这是个好问题。理论上一种传输类型就够了，为什么需要六种？

**原因 1：不同部署场景的约束不同**

- 本地运行的 MCP 服务器用 `stdio` 最简单——直接 fork 子进程，stdin/stdout 就是信道，不需要网络
- 远程服务器无法用 stdio，需要 SSE 或 WebSocket
- IDE 扩展运行在特殊沙箱里，需要单独的 `sse-ide`

**原因 2：性能权衡**

- `stdio` 比网络快（进程内通信），但只能本地用
- `sdk` 更快（完全进程内，无序列化开销），但 MCP 服务器必须是 JavaScript/TypeScript

**原因 3：某些类型有特殊能力**

- WebSocket 天然支持服务端主动推送，SSE 只支持服务端→客户端的单向流
- Streamable HTTP 支持流式响应，普通 SSE 不支持请求体

Claude Code 里有一个特别的优化——**in-process MCP server**（`sdk` 类型）。Chrome 控制工具和 Computer Use 桌面控制工具都以这种方式运行：不启动子进程，直接在 Claude Code 进程内运行。理由是这些工具很重（约 325MB），用子进程的启动开销太大，所以改成进程内加载。我们在 17-2 节看具体实现。

---

## 本章小结

MCP 是 Claude Code 连接外部世界的标准协议，本质是 JSON-RPC 2.0 over 多种传输层。核心设计：

1. **传输层与协议层分离**：六种传输类型共享同一套 JSON-RPC 2.0 消息格式
2. **配置有层级（scope）**：enterprise > user > project，高优先级覆盖低优先级
3. **连接状态是枚举**：pending / connected / failed / needs-auth / disabled，UI 直接消费
