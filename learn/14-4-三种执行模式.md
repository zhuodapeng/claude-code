# 14-4 三种执行模式：本地 / Worktree / 远程

## 问题引入：子 Agent 在哪里跑？

"在哪里运行"不只是个环境问题，它决定了子 Agent 能做什么、做了什么是否可以回滚、以及失败了影响的范围有多大。

Claude Code 为子 Agent 提供了三种执行环境：

| 模式 | 文件系统 | 触发方式 | 典型用途 |
|------|----------|----------|----------|
| 本地（local） | 与父共享 | 不设 isolation | 只读任务、分析任务 |
| Worktree | 独立 git 工作树 | `isolation: 'worktree'` | 写代码、修改文件 |
| 远程（remote） | 独立 CCR 容器 | `isolation: 'remote'` | 完全隔离、耗时任务 |

本节重点讲本地模式的两个变体（同步和 Fork），以及 Worktree 模式的技术实现。远程模式是 Anthropic 内部（ant）专用功能，仅作简要介绍。

---

## 一、本地同步模式（Local Sync）

这是最简单的模式，也是默认行为。

子 Agent 在父进程内运行，与父 Agent 共享：
- 同一个进程地址空间
- 同一个文件系统（子 Agent 修改文件，父 Agent 立刻看到）
- 同一个 git 仓库状态

差异只在隔离的 `ToolUseContext`（第 14-3 节讲过）：读文件缓存、权限状态、UI 回调都是隔离的。

**关键行为：阻塞主线程**

同步模式下，主线程 Claude 的整个 API 轮次被阻塞，直到子 Agent 完成：

```typescript
// AgentTool.tsx 同步执行路径
for await (const message of runAgent(runAgentParams)) {
  // 每条消息通过 onProgress 更新 UI
  if (onProgress) {
    onProgress({ type: 'agent_progress', agentId, ... })
  }
}
// 子 Agent 结束后，才能返回，主线程才能继续
return { data: { status: 'completed', content: finalText, ... } }
```

这意味着：用户在子 Agent 运行期间无法中断主线程（只能中断子 Agent）。这对于"查询后续需要基于查询结果做决策"的场景是合理的——主线程需要等子 Agent的结果，才能决定下一步。

## 二、本地 Fork 模式（runForkedAgent）

Fork 模式不是 AgentTool 的 `run_in_background` 参数，而是一个完全不同的机制——它直接调用 `runForkedAgent()`，由 Claude Code 内部代码触发（不是 Claude 通过工具调用触发）。

**文件：[src/utils/forkedAgent.ts](../src/utils/forkedAgent.ts#L489-L626)**

```typescript
export async function runForkedAgent({
  promptMessages,
  cacheSafeParams,      // 必须与父 Agent 一致（保证 cache 命中）
  canUseTool,
  querySource,
  forkLabel,            // 用于遥测，如 'compact', 'session_memory'
  maxOutputTokens,
  maxTurns,
  skipTranscript,       // 跳过 sidechain 记录（用于临时性工作）
  skipCacheWrite,
}: ForkedAgentParams): Promise<ForkedAgentResult>
```

Fork 模式的核心设计目标是**prompt cache 复用**。

### 为什么 Fork 需要 CacheSafeParams？

Anthropic API 的 prompt cache 键由以下因素组成：
- 系统提示（完整文本）
- 工具列表（完整 JSON）
- 模型名称
- 消息前缀（到第一条 `cache_control` 标记为止的消息）

`CacheSafeParams` 把前四个打包在一起：

```typescript
export type CacheSafeParams = {
  systemPrompt: SystemPrompt         // 必须与父完全相同
  userContext: { [k: string]: string }  // 影响系统提示的上下文
  systemContext: { [k: string]: string }
  toolUseContext: ToolUseContext      // 包含 tools 和 model
  forkContextMessages: Message[]     // 父会话的消息历史
}
```

如果任何一个字段与父 Agent 上一个 API 请求不同，prompt cache 就会失效，所有那些 token 都要重新付费处理。

### Cache 参数是怎么传递的？

主线程每轮结束后，`handleStopHooks()` 调用 `saveCacheSafeParams(params)` 把这轮的 cache 参数存到全局槽：

**文件：[src/utils/forkedAgent.ts](../src/utils/forkedAgent.ts#L70-L81)**

```typescript
// 全局槽：每轮 stop hooks 更新，post-turn forks 从这里读取
let lastCacheSafeParams: CacheSafeParams | null = null

export function saveCacheSafeParams(params: CacheSafeParams | null): void {
  lastCacheSafeParams = params
}

export function getLastCacheSafeParams(): CacheSafeParams | null {
  return lastCacheSafeParams
}
```

压缩服务 `compactConversation()`、记忆服务 `sessionMemory`、`/btw` 命令等，都调用 `getLastCacheSafeParams()` 来获取这轮的参数，确保它们 fork 出的子 Agent 与主线程共享 prompt cache。

这是一个典型的"数据最近一次生产者写，最近一次消费者读"模式——简单有效，但要求调用者知道自己在使用"上一轮"的参数。

### 执行完成后的内存清理

**文件：[src/utils/forkedAgent.ts](../src/utils/forkedAgent.ts#L599-L604)**

```typescript
} finally {
  // 释放克隆的文件状态缓存（防止内存泄漏）
  isolatedToolUseContext.readFileState.clear()
  // 释放克隆的 fork 上下文消息（可能是整个会话历史）
  initialMessages.length = 0
}
```

`finally` 块在子 Agent 完成或报错时都会执行。`readFileState.clear()` 和 `initialMessages.length = 0` 是手动触发 GC——这些数组可能非常大（整个会话历史），不手动清理会造成明显的内存压力，尤其是在高频调用 Fork（如每轮压缩一次）的场景里。

## 三、Worktree 模式（isolation: 'worktree'）

Worktree 模式为子 Agent 创建一个**独立的 git 工作树**，子 Agent 在那里工作，父 Agent 完全不受影响。

### git worktree 是什么？

`git worktree` 是 git 的内置功能，允许同一个仓库在多个目录里同时有不同的工作树（checked-out 状态）。它们共享同一个 `.git` 目录（或通过 `.git` 文件链接），但各自有独立的工作文件、独立的 HEAD 指向、可以各自 commit。

Claude Code 的 Worktree 隔离正是基于这个机制：

```
主仓库: /project
  .git/ (共享)
  src/foo.ts (父 Agent 正在修改)

Worktree: /project/.claude/worktrees/agent-abc12345
  .git (文件，链接回主 .git)
  src/foo.ts (子 Agent 的独立副本，基于 origin/main)
```

### createAgentWorktree() 的完整流程

**文件：[src/utils/worktree.ts](../src/utils/worktree.ts#L235-L334)** (简化版)

```
createAgentWorktree(slug):
  1. 计算路径：
     .claude/worktrees/{slug}  →  实际目录
     worktree-{slug}           →  git 分支名

  2. 快速检查（fast-resume path）：
     readWorktreeHeadSha(worktreePath)
     如果已存在 → 直接复用（跳过 git fetch）
     注：大型仓库 git fetch 可能需要 6-8s，跳过很重要

  3. 新建流程：
     mkdir -p .claude/worktrees/
     git fetch origin main       ← 获取最新基准
     git worktree add -B worktree-{slug} .claude/worktrees/{slug} origin/main

  4. symlink 大型目录（防止磁盘爆炸）：
     node_modules → 符号链接到主仓库 node_modules
     （否则每个 worktree 都复制 node_modules，磁盘消耗 N 倍）

  5. 返回 { worktreePath, worktreeBranch, headCommit }
```

**注意磁盘空间的考量**（[worktree.ts#L102-L138](../src/utils/worktree.ts#L102-L138)）：

```typescript
async function symlinkDirectories(
  repoRootPath: string,
  worktreePath: string,
  dirsToSymlink: string[],  // 如 ['node_modules', 'dist']
) {
  for (const dir of dirsToSymlink) {
    await symlink(sourcePath, destPath, 'dir')
    // 注释：Symlinked node_modules to avoid disk bloat
  }
}
```

`node_modules` 可能有几百 MB 到几 GB，每个子 Agent 都复制一份会很快耗尽磁盘空间。用符号链接让所有 worktree 共享主仓库的 `node_modules`，只有实际修改了的源文件会被复制。

### 子 Agent 在 Worktree 里的路径感知

Fork 子 Agent 继承了父 Agent 的会话历史，里面充满了父 Agent 工作目录（如 `/project/src/foo.ts`）的路径。但子 Agent 实际在 `/project/.claude/worktrees/agent-abc12345/` 里运行，这两个路径指向不同的文件。

为了解决这个"路径错位"问题，AgentTool 在 Fork + Worktree 组合时注入一条额外的提示：

**文件：[src/tools/AgentTool/forkSubagent.ts](../src/tools/AgentTool/forkSubagent.ts#L205-L210)**

```typescript
export function buildWorktreeNotice(parentCwd: string, worktreeCwd: string): string {
  return `You've inherited the conversation context above from a parent agent working in ${parentCwd}.
You are operating in an isolated git worktree at ${worktreeCwd} — same repository,
same relative file structure, separate working copy.
Paths in the inherited context refer to the parent's working directory;
translate them to your worktree root.
Re-read files before editing if the parent may have modified them since they appear in the context.
Your changes stay in this worktree and will not affect the parent's files.`
}
```

这段文字告诉子 Agent：
1. 你继承的历史是父 Agent 的，路径是父 Agent 的
2. 你现在在独立的 worktree 里，需要把路径翻译过来
3. 如果父 Agent 可能修改过某文件，在编辑前先重新读取一次

### Worktree 清理：自动 vs 保留

**文件：[src/utils/worktree.ts](../src/tools/AgentTool/AgentTool.tsx)** 中的 `hasWorktreeChanges()` 和 `removeAgentWorktree()`:

子 Agent 完成后，AgentTool 检查是否有未提交的修改：

```typescript
// 如果子 Agent 有修改，保留 worktree（用户可能要查看或合并）
// 如果子 Agent 没有修改，自动清理 worktree（节省磁盘空间）
if (await hasWorktreeChanges(worktreePath)) {
  // 保留，通知用户 worktree 路径和分支名
} else {
  await removeAgentWorktree(worktreePath)
}
```

这个"有修改才保留"的策略很务实：大多数只读探索任务不会修改文件，自动清理避免了 worktree 的积累；而真正修改了文件的任务，保留让用户可以审查、合并，或继续在那个 worktree 里工作。

### 路径安全：防止目录穿越

创建 worktree 时，slug（子目录名）经过严格验证：

**文件：[src/utils/worktree.ts](../src/utils/worktree.ts#L66-L87)**

```typescript
const VALID_WORKTREE_SLUG_SEGMENT = /^[a-zA-Z0-9._-]+$/

export function validateWorktreeSlug(slug: string): void {
  for (const segment of slug.split('/')) {
    if (segment === '.' || segment === '..') {
      throw new Error('Invalid worktree name: must not contain "." or ".." path segments')
    }
    if (!VALID_WORKTREE_SLUG_SEGMENT.test(segment)) {
      throw new Error(`Invalid worktree name: each segment must contain only letters, digits, dots, underscores, and dashes`)
    }
  }
}
```

Claude Code 生成 slug 时用 `agent-${agentId.slice(0, 8)}` 格式（[AgentTool.tsx#L591](../src/tools/AgentTool/AgentTool.tsx#L591)），理论上不会触发这个检查。但验证仍然存在，因为用户自定义 worktree slug 的其他代码路径可能传入任意字符串，防御性验证是必要的。

另外注意 slug 扁平化处理：

```typescript
// 嵌套 slug（如 user/feature）→ 扁平化为 user+feature
function flattenSlug(slug: string): string {
  return slug.replaceAll('/', '+')
}
```

注释解释了原因：嵌套会导致 D/F 冲突（git 分支名 `worktree-user` 与 `worktree-user/feature` 冲突），以及目录层级问题（`user` worktree 的删除会连带删除 `user/feature`）。

## 四、远程模式（isolation: 'remote'，ant 内部专用）

远程模式把子 Agent 的整个执行上传到 Anthropic 的 CCR（Cloud Compute Runtime）环境里运行。这是 Anthropic 内部工程师专用的功能，在外部构建里被编译期剔除（dead code elimination）：

**文件：[src/tools/AgentTool/AgentTool.tsx](../src/tools/AgentTool/AgentTool.tsx#L435)**

```typescript
// "external" === 'ant' 永远为 false（外部构建）
// 这整个 if 块在外部构建里会被 tree-shaking 掉
if ("external" === 'ant' && effectiveIsolation === 'remote') {
  const session = await teleportToRemote({
    initialMessage: prompt,
    // ...
  })
  return { status: 'remote_launched', taskId, sessionUrl, ... }
}
```

远程模式的输出是 `remote_launched`，包含一个 `sessionUrl`——用户可以打开这个 URL 查看远程 Agent 的执行过程。远程 Agent 完成后通过 webhook 通知本地进程。

## 三种模式对比

```
                 本地同步            Worktree              远程
─────────────────────────────────────────────────────────────────
文件系统         共享               独立工作树            独立容器
父 Agent 影响    直接受影响         隔离                  完全隔离
启动速度         极快               中（需要 git）         慢（网络）
失败回滚         需要手动回滚       删除 worktree          销毁容器
并发安全         ❌（文件竞争）     ✅                    ✅
磁盘消耗         无额外             少量（共享 node_modules）  无本地
适合任务         只读/分析          写代码                耗时/高风险
```

---

## 本节小结

本地同步模式是最简单、启动最快的，适合只读或不需要隔离的任务；Worktree 模式利用 git 的原生 worktree 功能实现文件系统隔离，适合写代码的任务，通过 symlink 节省磁盘空间，通过 `buildWorktreeNotice()` 帮助子 Agent 正确处理路径；远程模式（ant 专用）提供完全隔离的环境，但不对外开放。`runForkedAgent()` 是独立的轻量级机制，专为 prompt cache 复用优化，用于 Claude Code 内部服务。
