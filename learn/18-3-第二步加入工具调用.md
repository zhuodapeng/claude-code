# 18-3 第二步：加入工具调用

## 目标

让 LLM 能调用工具：读文件、执行 Bash 命令。

这一步是从"聊天机器人"到"编程助手"的跨越。工具调用是 Claude Code 最核心的能力，理解它的实现对理解 Claude Code 至关重要。

---

## 工具调用的协议回顾

我们在第 5 章（5-3 节）学过工具调用协议。这里快速复习，然后看实现。

工具调用的消息流：

```
1. Claude Code → API: 我有这些工具，用户问了 XXX
2. API → Claude Code: 我决定调用 read_file，参数是 { path: "./README.md" }
3. Claude Code 执行 read_file，得到文件内容
4. Claude Code → API: 工具结果是 "# Hello World..."
5. API → Claude Code: 基于文件内容，我的回答是...
```

关键：步骤 2 的 API 响应的 `stop_reason` 是 `'tool_use'`，而不是 `'end_turn'`。这告诉 Claude Code："我还没说完，我需要工具结果才能继续"。

---

## 工具的数据结构

Anthropic API 要求工具按这个格式传入：

```typescript
const tools: Anthropic.Tool[] = [
  {
    name: 'read_file',
    description: '读取文件内容。使用这个工具来查看文件。',
    input_schema: {
      type: 'object',
      properties: {
        path: {
          type: 'string',
          description: '文件路径（相对或绝对路径）',
        },
      },
      required: ['path'],
    },
  },
  {
    name: 'bash',
    description: '执行 Bash 命令。用于运行系统命令、脚本等。',
    input_schema: {
      type: 'object',
      properties: {
        command: {
          type: 'string',
          description: '要执行的 Bash 命令',
        },
      },
      required: ['command'],
    },
  },
]
```

注意 `input_schema` 使用 JSON Schema 格式，不是 Zod。LLM API 使用 JSON Schema 是因为它是与语言无关的标准——无论你用 Python、TypeScript 还是 Go，JSON Schema 都是一样的。

## 完整代码：step2.ts

```typescript
// step2.ts - 加入工具调用
import Anthropic from '@anthropic-ai/sdk'
import * as readline from 'readline'
import * as fs from 'fs/promises'
import { exec } from 'child_process'
import { promisify } from 'util'

const execAsync = promisify(exec)

// ======= 工具定义（告诉 LLM 有哪些工具）=======
const TOOLS: Anthropic.Tool[] = [
  {
    name: 'read_file',
    description: '读取文件内容。使用此工具查看文件内容。',
    input_schema: {
      type: 'object',
      properties: {
        path: { type: 'string', description: '文件路径' },
      },
      required: ['path'],
    },
  },
  {
    name: 'bash',
    description: '执行 bash 命令并返回输出。',
    input_schema: {
      type: 'object',
      properties: {
        command: { type: 'string', description: '要执行的命令' },
      },
      required: ['command'],
    },
  },
]

// ======= 工具执行函数 =======
async function executeTool(
  toolName: string,
  toolInput: Record<string, unknown>,
): Promise<string> {
  if (toolName === 'read_file') {
    const path = toolInput.path as string
    try {
      return await fs.readFile(path, 'utf-8')
    } catch (e) {
      return `Error: ${e}`
    }
  }

  if (toolName === 'bash') {
    const command = toolInput.command as string
    try {
      const { stdout, stderr } = await execAsync(command, { timeout: 30000 })
      return stdout + (stderr ? '\nSTDERR: ' + stderr : '')
    } catch (e: any) {
      return `Error: ${e.message}\n${e.stderr || ''}`
    }
  }

  return `Unknown tool: ${toolName}`
}

// ======= 核心：带工具调用的对话循环 =======
const client = new Anthropic({ apiKey: process.env.ANTHROPIC_API_KEY })
const messages: Anthropic.MessageParam[] = []

async function ask(userInput: string): Promise<string> {
  messages.push({ role: 'user', content: userInput })

  // 工具调用可能发生多次（LLM 调用工具后，继续生成，可能再次调用工具）
  // 所以这里用一个内部循环
  while (true) {
    const response = await client.messages.create({
      model: 'claude-opus-4-6',
      max_tokens: 8096,
      tools: TOOLS,                  // 告诉 API 有哪些工具
      messages: messages,
    })

    if (response.stop_reason === 'end_turn') {
      // LLM 完成了回答，提取文本内容
      const text = response.content
        .filter(b => b.type === 'text')
        .map(b => b.text)
        .join('')

      messages.push({ role: 'assistant', content: response.content })
      return text
    }

    if (response.stop_reason === 'tool_use') {
      // LLM 请求调用工具
      // response.content 包含文本（可选）和工具调用请求

      // 1. 把 LLM 的这次"助手消息"（含工具调用请求）加入历史
      messages.push({ role: 'assistant', content: response.content })

      // 2. 找出所有工具调用请求（一次可以有多个）
      const toolUseBlocks = response.content.filter(b => b.type === 'tool_use')

      // 3. 执行所有工具，收集结果
      const toolResults: Anthropic.ToolResultBlockParam[] = []

      for (const block of toolUseBlocks) {
        if (block.type !== 'tool_use') continue

        console.log(`\n[Tool: ${block.name}]`, JSON.stringify(block.input))

        const result = await executeTool(
          block.name,
          block.input as Record<string, unknown>,
        )

        toolResults.push({
          type: 'tool_result',
          tool_use_id: block.id,   // 关键：要对应 LLM 的工具调用 ID
          content: result,
        })
      }

      // 4. 把工具结果作为"用户消息"加入历史
      // 注意：工具结果放在 role: 'user' 的消息里
      messages.push({
        role: 'user',
        content: toolResults,
      })

      // 5. 继续循环，让 LLM 基于工具结果继续生成
      // （LLM 可能会再次调用工具，所以是 while(true) 循环）
    }
  }
}

// ======= 主循环（与 step1 相同）=======
async function main() {
  const rl = readline.createInterface({ input: process.stdin, output: process.stdout })
  console.log('LLM CLI with tools ready.\n')

  for await (const line of rl) {
    if (!line.trim()) continue
    const answer = await ask(line)
    console.log('\nAssistant:', answer, '\n')
  }
}

main().catch(console.error)
```

## 关键理解：工具结果为什么放在 role: 'user' 里？

这是最容易困惑的地方。直觉上，工具是 Claude Code 执行的，应该放在 `role: 'assistant'` 里——但 Anthropic API 的设计不是这样。

**原因是角色的语义**：

- `role: 'assistant'`：Claude 说的话（包括 Claude 决定调用工具的请求）
- `role: 'user'`：用户（或系统）给 Claude 的反馈（包括工具执行结果）

工具结果是"外部世界返回给 Claude 的信息"，在语义上更接近"用户反馈"，所以放在 `user` 角色里。

对应 Claude Code 源码：

**文件：[src/core/queryLoop.ts](../src/core/queryLoop.ts)**（概念位置）

```typescript
// runTools() 执行完工具后，构建 tool_result 消息
const toolResultMessage: MessageParam = {
  role: 'user',
  content: toolResults.map(r => ({
    type: 'tool_result',
    tool_use_id: r.toolUseId,
    content: r.result,
  })),
}
messages.push(toolResultMessage)
```

与我们的实现完全一样的模式。

## 工具调用 ID：为什么重要

```typescript
toolResults.push({
  type: 'tool_result',
  tool_use_id: block.id,   // 必须对应 LLM 的工具调用 ID
  content: result,
})
```

`tool_use_id` 是 LLM 生成的唯一 ID，用来关联"工具调用请求"和"工具执行结果"。

当 LLM 发出多个并发工具调用时（比如同时调用 `read_file` 和 `bash`），结果需要明确归属到哪个调用。ID 确保了这种映射的准确性——即使你并行执行工具，乱序返回结果，LLM 也能正确理解哪个结果对应哪个工具调用。

## while(true) 循环：多轮工具调用

注意我们用了一个内部的 `while (true)` 循环：

```typescript
while (true) {
  const response = await client.messages.create(...)

  if (response.stop_reason === 'end_turn') {
    return text   // 退出循环
  }

  if (response.stop_reason === 'tool_use') {
    // 执行工具，加入历史
    // 继续循环，再次调用 API
  }
}
```

这对应 Claude Code 的 `queryLoop` 的外层循环结构。LLM 可以多次调用工具：

```
用户："统计 src/ 目录下有多少个 .ts 文件，然后读取最大的那个"

LLM 第一次 → 调用 bash: find src/ -name "*.ts" | wc -l
工具结果 → "42"
LLM 第二次 → 调用 bash: find src/ -name "*.ts" -exec ls -la {} \; | sort -k5 -rn | head -1
工具结果 → "src/core/queryLoop.ts 15423 bytes"
LLM 第三次 → 调用 read_file: { path: "src/core/queryLoop.ts" }
工具结果 → 文件内容...
LLM 最终 → end_turn，给出回答
```

没有 `while` 循环，第一轮工具调用后就停了，无法支持这种多步推理。

## 与 Claude Code 的对比

| 我们的实现 | Claude Code 的对应 |
|-----------|------------------|
| `TOOLS` 数组 | `assembleToolPool()` 注册的工具列表 |
| `executeTool()` | `tool.call()` 方法 |
| `while (stop_reason === 'tool_use')` | `queryLoop` 的主循环 |
| 直接 `console.log` 工具调用 | `printAgentAction()` 格式化输出 |
| 无权限检查 | `checkPermissions()` 七步流水线（第 11 章）|

我们的实现缺少权限检查——任何工具都直接执行。这在生产中是不可接受的，但现在先跳过，第 5 步（18-6）加权限系统。

## 本章小结

工具调用的核心模式：
1. 在 API 请求中附带工具定义
2. 检测 `stop_reason === 'tool_use'`
3. 执行工具，构建 `tool_result` 消息（`role: 'user'`）
4. 用工具结果继续调用 API
5. 循环直到 `stop_reason === 'end_turn'`
