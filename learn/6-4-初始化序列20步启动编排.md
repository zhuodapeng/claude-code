# 6-4 初始化序列：启动编排的工程决策

> **本节目标**：当你运行 `claude-haha` 直到 TUI 出现，中间经历了多少步？每一步为什么要在这里？为什么顺序是这样的？这节课的重点不是列清单——而是**理解并行、顺序、信任门控这三种调度策略，以及什么时候用哪种**。

---

## 启动的核心矛盾

一个理想的启动过程应该满足两个互相冲突的目标：

1. **快**：用户打开工具，应该立刻看到可交互的界面
2. **安全**：某些初始化（LSP 服务器、插件代码）不能在用户信任目录之前运行

这两个目标是矛盾的。要快，就要尽早并行做一切。要安全，就必须等信任对话框通过才能执行某些代码。

Claude Code 的启动编排，**本质上是在管理这个矛盾**。

---

## 三种调度模式

Claude Code 的启动代码里，有三种清晰可辨的调度模式：

```
模式 A：完全并行（void xxx()）
  - 扔出去，不等结果，稍后能用就行
  - 用于：检测 JetBrains、预连接 API、检测 Git 仓库

模式 B：等待但不阻塞 UI（Promise.all）
  - 几件事同时跑，完成后才能继续
  - 用于：setup() + getCommands() + getAgentDefs() 并行

模式 C：信任门控（必须在 showSetupScreens() 之后）
  - 只有用户确认信任目录后才能运行
  - 用于：LSP 服务器初始化、应用全部环境变量、执行插件代码
```

理解这三种模式，你就理解了整个启动流程的设计意图。

## 完整启动序列

下面是 `main.tsx` 的 `.action()` 处理函数里，从参数解析完毕到 TUI 渲染这段路程的关键步骤：

```
1. [preAction] init() - 核心初始化
        ↓
2. [action 入口] 解构 options 参数
        ↓
3. 验证互斥参数（--system-prompt vs --system-prompt-file）
        ↓
4. ┌──────────────────────────── 并行启动 ────────────────────────────┐
   │  setup()           getCommands()        getAgentDefinitions()     │
   │  ~28ms             读取 CLAUDE.md 等     读取 .claude/agents/      │
   └──────────────────────────────────────────────────────────────────┘
        ↓  await setup() 完成（commands 和 agents 继续跑）
5. initBuiltinPlugins() + initBundledSkills()（纯内存操作，<1ms）
        ↓
6. 启动 MCP 配置预取（并行，不阻塞）
        ↓
7. 启动 Session Start hooks（并行，不阻塞）
        ↓
8. [Interactive only] getRenderContext() + createRoot() 
        ↓
9. showSetupScreens() ← ████ 信任门控 ████
   （信任对话框、OAuth 登录、首次设置向导）
        ↓ 用户确认信任
10. 验证组织策略（企业用户）
        ↓
11. initializeLspServerManager() ← 必须在信任后
        ↓
12. 启动后台预取（quota、fastMode、bootstrap data）
        ↓
13. await Promise.all([commands, agentDefs]) ← 现在用结果了
        ↓
14. [根据启动模式分叉]
   ├── print 模式 → 走 print.ts 的无 UI 路径
   ├── resume 模式 → 加载历史消息
   └── 默认模式 → renderAndRun(root, <REPL .../>)
```

## 步骤详解

### 步骤 4：setup() 和 getCommands() 并行

这是启动优化里最重要的一个决策。

**文件：[src/main.tsx](../src/main.tsx#L1910-L1940)**

```typescript
// 同时启动三件事
const preSetupCwd = getCwd()
initBuiltinPlugins()
initBundledSkills()

// setup() 负责：确定工作目录、初始化文件权限系统、启动 UDS 消息 socket
const setupPromise = (async () => {
  const { setup } = await import('./setup.js')
  await setup(preSetupCwd, permissionMode, ...)
})()

// getCommands() 负责：读取 CLAUDE.md、发现 skills、加载子命令
const commandsPromise = worktreeEnabled ? null : getCommands(preSetupCwd)
const agentDefsPromise = worktreeEnabled ? null : getAgentDefinitionsWithOverrides(preSetupCwd)

// 错误抑制：在 await setupPromise 之前，如果 commands 先出错
// 会产生 unhandledRejection。这里提前 .catch 消音，不是忽略错误——
// 后面 Promise.all 仍然会捕获
commandsPromise?.catch(() => {})
agentDefsPromise?.catch(() => {})

// 等 setup 完成（commands 和 agents 继续在后台跑）
await setupPromise
```

注释里解释了为什么这样做：`setup()` 的 ~28ms 主要消耗在 `startUdsMessaging`（绑定 Unix domain socket）——是 IO 操作，不占 CPU。`getCommands()` 主要是读文件——也是 IO 操作。两件 IO 密集的事并行跑，等于"同时发出两个请求"。

**为什么 setup() 要先 await，commands 可以继续跑？**

因为 `setup()` 可能会调用 `process.chdir()`——改变工作目录。如果 `getCommands()` 在 setup 完成前跑完了，它可能用了旧的工作目录。

解法：`getCommands(preSetupCwd)` 传入明确的目录（不依赖 `cwd()`），所以实际上可以安全并行。

### 步骤 9：showSetupScreens()——信任门控的设计哲学

这是整个启动序列里最有意思的设计。

**文件：[src/interactiveHelpers.tsx](../src/interactiveHelpers.tsx#L104)**

`showSetupScreens()` 依次可能显示：
1. **信任对话框**：你信任这个目录的代码吗？
2. **OAuth 登录**：还没登录？
3. **首次使用向导**：第一次用？

**为什么 LSP 必须等 showSetupScreens() 之后？**

注释说得很清楚（`main.tsx` 第 2323 行）：

```
// Initialize LSP manager AFTER trust is established (or in non-interactive mode
// where trust is implicit). This prevents plugin LSP servers from executing
// code in untrusted directories before user consent.
```

LSP 服务器是插件提供的——它们会在用户的项目目录里执行代码。如果用户还没确认"信任这个目录"，就让插件代码跑起来，相当于攻击者可以通过在 `.claude/` 里放恶意插件来在信任对话框出现前执行代码。

**为什么 applyConfigEnvironmentVariables() 在非交互模式下提前调用，但在交互模式下等信任后？**

非交互模式（`-p`）：信任是隐式的——你把 `-p` flag 传进来，说明你是有意识地在这个目录里运行。
交互模式：用户需要明确点"信任"按钮才算建立信任。

这个区分很重要。同一段代码，在不同的信任语境下有不同的运行时机。

### 步骤 11：initializeLspServerManager()

**文件：[src/main.tsx](../src/main.tsx#L2327)**

```typescript
// Must be after inline plugins are set (if any) so --plugin-dir LSP servers are included.
initializeLspServerManager()
```

这里有另一个顺序要求：必须在 `setInlinePlugins()` 之后（`preAction` 里处理 `--plugin-dir` 参数）。

因为 LSP 管理器需要知道"所有插件的完整列表"才能初始化对应的 LSP 服务器。如果在插件列表确定前初始化，会漏掉 `--plugin-dir` 里的 LSP 服务器。

### 步骤 12：后台预取——"不阻塞首屏，但要在第一次请求前准备好"

**文件：[src/main.tsx](../src/main.tsx#L2353-L2381)**

```typescript
// 这些 API 调用不能阻塞 UI 渲染，但要在用户发第一条消息前完成
checkQuotaStatus().catch(error => logError(error))   // 检查配额
void fetchBootstrapData()                              // 从服务端拉配置
void prefetchPassesEligibility()                       // Claude.ai Pro 状态
void prefetchFastModeStatus()                          // Fast 模式是否可用
```

全部用 `void` 或 `.catch()` 包裹——不阻塞，自生自灭。

为什么可以这样？因为这些信息影响的是 UI 里某些状态的显示（比如 Fast 模式按钮是否灰显），不影响核心功能。即使预取失败，用户仍然可以正常使用。

### 步骤 14：renderAndRun()

**文件：[src/interactiveHelpers.tsx](../src/interactiveHelpers.tsx#L98-L103)**

```typescript
export async function renderAndRun(root: Root, element: React.ReactNode): Promise<void> {
  root.render(element)          // 把 React 组件挂到 Ink 的终端根节点
  startDeferredPrefetches()     // 启动另一批"更晚才需要"的预取
  await root.waitUntilExit()    // 等待用户退出（阻塞直到 Ink 卸载）
  await gracefulShutdown(0)     // 清理资源，正常退出
}
```

`root.render(element)` 就是这一切的终点——React 渲染器（Ink）接管，把 JSX 组件树渲染成终端 UI。

之后这个函数会一直阻塞（`await root.waitUntilExit()`），直到用户按 Ctrl+C 或退出。

## 两条不同的路：交互模式 vs 非交互模式

上面描述的是**交互模式**（默认启动 TUI）。

当用户加了 `-p` / `--print` 参数时，走另一条路：

**文件：[src/main.tsx](../src/main.tsx#L1958-L1996)**

```typescript
if (getIsNonInteractiveSession()) {
  // 非交互模式：信任是隐式的，提前应用全部环境变量
  applyConfigEnvironmentVariables()
  
  // 提前启动系统上下文获取（git status 等）——和后续加载并行
  void getSystemContext()
  void getUserContext()
  void ensureModelStringsInitialized()
  
  // 没有 showSetupScreens()！没有 createRoot()！没有 TUI！
}
```

非交互模式的核心差异：
- 没有信任对话框（隐式信任）
- 没有 Ink 渲染（没有 TUI）
- 提前启动更多并行预取（因为没有"等用户点击"的时间来并行）
- 最终走 `print.ts` 里的无界面路径，直接输出文本到 stdout

这两条路在 `main.tsx` 的 action handler 里分叉，但共享前面所有的初始化步骤。

## 为什么 setup() 是单独的文件？

你注意到了吗：`setup()` 是动态导入的：

```typescript
const { setup } = await import('./setup.js')
await setup(...)
```

为什么不直接静态导入？

因为 `setup.js` 是个很重的模块——它导入了文件系统操作、权限系统、worktree 逻辑……把它动态导入意味着**在非 setup 路径（如 `--daemon-worker`）里，这个模块根本不会被加载**。

这又是前两节讲的"动态导入作为按需加载"的实际应用。

## 本节小结

- Claude Code 的启动是三种调度模式的混合：完全并行（void）、等待完成（Promise.all）、信任门控（showSetupScreens() 之后）
- `setup()` 和 `getCommands()` 并行跑，因为都是 IO 操作；但 `setup()` 先 await，因为它可能改变工作目录
- LSP 服务器必须在信任确认后才初始化，防止恶意插件在用户同意前执行代码
- 非交互模式（`-p`）跳过 showSetupScreens，提前应用全部环境变量
- 最终 `renderAndRun()` 把 React 树挂到 Ink 根节点，TUI 出现
