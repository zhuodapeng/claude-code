# 5-2 流式响应（Streaming）是怎么工作的

> **本节目标**：理解 Streaming 的技术原理——SSE 协议、为什么要流式而不是等待完整响应、流式事件的类型，以及 Claude Code 如何消费这个流。这是理解 queryLoop 里"逐事件处理"的基础。

---

## 没有 Streaming 会是什么体验？

先想象没有流式响应的场景：

```
你：帮我分析这 500 行代码的架构

Claude 内部：（思考 30 秒，生成 2000 字分析）

你：（等待 30 秒，什么都看不到）

你：（又等了 10 秒，还是没有）

你：（开始怀疑是不是超时了）

突然：（2000 字的分析一次性蹦出来）
```

30-40 秒的空白等待，用户体验极差。

**有了 Streaming**：

```
你：帮我分析这 500 行代码的架构

Claude：这▋
Claude：这段▋
Claude：这段代码▋
Claude：这段代码的架构▋
...（文字像打字机一样逐渐出现）
```

**第一个字几乎立刻出现**，即使最终完成需要 30 秒，用户立刻看到 Claude 在"工作"，体验完全不同。

---

## 流式响应的底层技术：SSE

Streaming 使用的 HTTP 协议技术叫 **SSE（Server-Sent Events）**，中文叫"服务器发送事件"。

**普通 HTTP 请求**：
```
客户端请求 → 服务端处理完整 → 一次性响应
```

**SSE（流式）**：
```
客户端请求 → 服务端开始处理 → 分批发送数据片段 → 客户端逐片接收
```

SSE 是一个 HTTP 长连接：服务器在这个连接上持续写入数据，每写一片，客户端就能立即读到，不需要等所有数据准备好。

SSE 的数据格式非常简单：

```
HTTP/1.1 200 OK
Content-Type: text/event-stream

event: message_start
data: {"type":"message_start","message":{"id":"msg_xxx","type":"message",...}}

event: content_block_start
data: {"type":"content_block_start","index":0,"content_block":{"type":"text","text":""}}

event: content_block_delta
data: {"type":"content_block_delta","index":0,"delta":{"type":"text_delta","text":"这"}}

event: content_block_delta
data: {"type":"content_block_delta","index":0,"delta":{"type":"text_delta","text":"段"}}

... 更多 delta 事件 ...

event: message_stop
data: {"type":"message_stop"}
```

每个 `event: ...` 和 `data: ...` 是一个事件，双换行分隔。

---

## Anthropic API 的流式事件类型

Claude Code 需要处理多种流式事件：

```
message_start            ← 响应开始，包含 usage 信息
  │
  ├── content_block_start   ← 内容块开始（文本块或工具调用块）
  │   │
  │   └── content_block_delta  ← 内容增量（文字 delta 或工具输入 delta）
  │       │
  │       ├── text_delta        ← 文字增量："你好"、"，"、"世界"
  │       │
  │       └── input_json_delta  ← 工具调用参数增量（JSON 片段）
  │
  ├── content_block_stop    ← 内容块结束
  │
  └── message_delta         ← 消息级别的增量（stop_reason 等）

message_stop             ← 整个响应结束
```

这些事件类型在 `src/types/message.ts` 里有完整定义，在 [src/query.ts](../src/query.ts) 里逐一处理。

## 为什么工具调用参数也要流式传输？

这是一个有意思的设计问题。

Claude 在决定调用一个工具时，需要指定工具的参数（比如 `{"command": "ls -la", "timeout": 30}`）。这些参数是 JSON 格式的。

**问题**：如果等整个 JSON 完整了再传过来，那就不是流式了。但 JSON 必须完整才能解析啊？

**解法**：把 JSON 字符串的片段流式传输，客户端拼接完整后解析。

```
流式传入：
  '{'              ← input_json_delta
  '"command"'      ← input_json_delta
  ': "ls '        ← input_json_delta
  '-la",'          ← input_json_delta
  '"timeout"'      ← input_json_delta
  ': 30}'          ← input_json_delta（结束，现在可以 JSON.parse）
```

Claude Code 在流式处理阶段拼接这些片段，直到 `content_block_stop` 事件出现，再完整解析 JSON。

## Claude Code 如何消费流式响应

**文件：[src/services/api/claude.ts](../src/services/api/claude.ts#L752-L779)**

```typescript
export async function* queryModelWithStreaming({
  messages, systemPrompt, tools, signal, options,
}): AsyncGenerator<StreamEvent | AssistantMessage | SystemAPIErrorMessage, void> {
  return yield* withStreamingVCR(messages, async function* () {
    yield* queryModel(messages, systemPrompt, thinkingConfig, tools, signal, options)
  })
}
```

这个函数是 AsyncGenerator——每当 API 发来一个 SSE 事件，它就 `yield` 一个经过处理的事件对象给 `queryLoop`。

在 `queryLoop` 里处理这个流：

**文件：[src/query.ts](../src/query.ts#L337-L338)**（简化版）

```typescript
yield { type: 'stream_request_start' }  // 通知 TUI：开始请求了

// 逐事件处理流
for await (const event of callAPI(messages, tools, signal)) {
  if (event.type === 'text_delta') {
    yield event   // 转发给 TUI 显示
    
  } else if (event.type === 'tool_use') {
    toolUseBlocks.push(event)  // 收集工具调用
    
  } else if (event.type === 'message_stop') {
    break  // 本轮 API 调用完成
  }
}
```

## 流式响应 vs 非流式响应的 API 差异

**非流式请求**（等待完整响应）：

```typescript
// 等待完整响应
const message = await anthropic.messages.create({
  model: 'claude-sonnet-4-6',
  max_tokens: 1024,
  messages: [{ role: 'user', content: 'Hello' }],
})
console.log(message.content[0].text)  // 一次性拿到完整文字
```

**流式请求**：

```typescript
// 流式响应
const stream = anthropic.messages.stream({
  model: 'claude-sonnet-4-6',
  max_tokens: 1024,
  messages: [{ role: 'user', content: 'Hello' }],
})

for await (const event of stream) {
  if (event.type === 'content_block_delta' && event.delta.type === 'text_delta') {
    process.stdout.write(event.delta.text)  // 每个 token 立即打印
  }
}
```

Claude Code 使用流式请求，因为它需要把每个 token 实时展示给用户。

## 流式响应中的错误处理

流可能在中途中断——网络问题、API 限流、服务器错误。Claude Code 需要处理这些情况：

```
可能的错误：
- 连接超时（网络问题）
- 429 Too Many Requests（速率限制）
- 529 Service Overloaded（服务过载）
- AbortError（用户中断，3-6节讲过）
```

Claude Code 有重试机制，对于某些可恢复的错误（如 529）会自动重试。

---

## 本节小结

- Streaming（流式响应）让用户立刻看到第一个 token，而不是等待完整响应
- 底层使用 SSE（Server-Sent Events）协议：一个 HTTP 长连接，服务端持续写入
- Anthropic API 的流式事件类型：`message_start/stop`、`content_block_delta`（文字 delta）、`input_json_delta`（工具参数 delta）
- Claude Code 用 AsyncGenerator 消费这个流，每个事件 `yield` 给 `queryLoop` 处理
- 工具调用参数也是流式传输的，客户端拼接完整 JSON 再解析
