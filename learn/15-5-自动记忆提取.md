# 15-5 自动记忆提取：Claude 如何主动写记忆

前几节讲的都是记忆的"读"——如何存储、如何注入、如何召回。这一节讲"写"：每次对话结束后，Claude Code 会自动运行一个后台 Agent，分析这次对话，判断是否有值得保留的信息，如果有，将其写入磁盘。

这个系统的完整实现在 [src/services/extractMemories/extractMemories.ts](../src/services/extractMemories/extractMemories.ts)，一共 616 行，包含了设计巧妙的并发控制、互斥机制和缓存优化。

---

## 触发时机

记忆提取发生在每个 query loop 的**结束**时刻——就是 Claude 给出了最终回复（没有更多工具调用）的那一刻。在 `stopHooks.ts` 的 `handleStopHooks()` 里：

```
用户消息 → Claude 响应 → 工具调用 → Claude 响应 → ... → 最终响应
                                                              │
                                                              └── handleStopHooks()
                                                                      │
                                                                      ├── executeExtractMemories()   ← 记忆提取
                                                                      ├── confidenceRating()
                                                                      └── promptCoaching()
```

`executeExtractMemories()` 是**fire-and-forget**的——主线程不等待它完成就继续处理用户下一条消息。提取在后台进行，不阻塞交互。

## initExtractMemories()：闭包作用域的状态管理

[src/services/extractMemories/extractMemories.ts:296-587](../src/services/extractMemories/extractMemories.ts#L296-L587) 里的 `initExtractMemories()` 是整个系统的初始化函数，使用了闭包来隔离状态：

```typescript
export function initExtractMemories(): void {
  // 所有状态都是闭包私有的，不是模块级别的
  const inFlightExtractions = new Set<Promise<void>>()
  let lastMemoryMessageUuid: string | undefined   // 游标：上次处理到哪条消息
  let hasLoggedGateFailure = false                // 防止重复日志
  let inProgress = false                          // 互斥锁
  let turnsSinceLastExtraction = 0                // 节流计数器
  let pendingContext: ... | undefined             // trailing run 的暂存上下文

  // 内部提取逻辑
  async function runExtraction({ context, appendSystemMessage, isTrailingRun }) { ... }

  // 公开入口点（赋值给模块级变量）
  extractor = async (context, appendSystemMessage) => { ... }
  drainer = async (timeoutMs = 60_000) => { ... }
}
```

这个模式和第 14 章讲的 `confidenceRating.ts` 是一样的。**为什么用闭包而不是模块级变量？**

因为测试。如果状态是模块级别的，测试之间会相互污染：测试 A 的提取还在进行，测试 B 就开始了，它看到的 `inProgress = true` 是测试 A 留下的状态。用闭包，每个 `beforeEach` 调用 `initExtractMemories()` 就得到一个全新的状态，完全独立。

注释里说得很清楚：`Tests call initExtractMemories() in beforeEach to get a fresh closure.`

## 游标机制：只处理新消息

```typescript
let lastMemoryMessageUuid: string | undefined
```

每次提取后，游标被更新为最新消息的 UUID：
```typescript
const lastMessage = messages.at(-1)
if (lastMessage?.uuid) {
  lastMemoryMessageUuid = lastMessage.uuid
}
```

下次提取时，`countModelVisibleMessagesSince()` 从这个游标开始计数，只统计游标之后的新消息：

```typescript
const newMessageCount = countModelVisibleMessagesSince(
  messages,
  lastMemoryMessageUuid,
)
```

**为什么不每次提取全部消息？**

上下文窗口是有限的，而且每次都把所有对话历史发给提取 Agent 既昂贵又没必要。游标机制确保每次提取 Agent 只需要分析本次 query loop 里新增的内容。

**游标在 compaction（上下文压缩）后怎么办？**

`countModelVisibleMessagesSince()` 有这样的处理：

```typescript
// If sinceUuid was not found (e.g., removed by context compaction),
// fall back to counting all model-visible messages rather than returning 0
// which would permanently disable extraction for the rest of the session.
if (!foundStart) {
  return count(messages, isModelVisibleMessage)
}
```

如果 compaction 删除了游标 UUID 对应的消息，游标就"失效"了——此时退回到处理全部消息，而不是返回 0（否则会造成"永远没有新消息"的假象，永远不再提取）。

## 互斥锁：防止并发提取

```typescript
let inProgress = false
let pendingContext: { context, appendSystemMessage } | undefined
```

如果一次提取还在进行中，下一次 `executeExtractMemories()` 调用到来（用户发送了下一条消息并得到回复）：

```typescript
if (inProgress) {
  // 暂存最新的上下文，用于 trailing run
  pendingContext = { context, appendSystemMessage }
  return  // 不启动新的提取
}
await runExtraction({ context, appendSystemMessage })
```

`pendingContext` 只保留最新的一个——如果用户在一次提取进行时连续发送了 3 条消息，只有最新的上下文会被暂存。这是故意的：最新的上下文包含了所有历史消息，是最完整的。

在 `runExtraction()` 的 `finally` 块里处理 trailing run：

```typescript
finally {
  inProgress = false
  const trailing = pendingContext
  pendingContext = undefined
  if (trailing) {
    // 用最新的上下文，追加提取之前被跳过的消息
    await runExtraction({
      context: trailing.context,
      appendSystemMessage: trailing.appendSystemMessage,
      isTrailingRun: true,
    })
  }
}
```

这形成了一个"串行化"的机制：无论用户发送多快，提取始终串行执行，每次只分析自上次游标以来的新消息。

## 与主 Agent 的互斥：hasMemoryWritesSince()

记忆提取 Agent 和主 Agent 之间有一个重要的互斥机制：

```typescript
// src/services/extractMemories/extractMemories.ts:121-148
function hasMemoryWritesSince(
  messages: Message[],
  sinceUuid: string | undefined,
): boolean {
  // 扫描游标之后的所有消息
  // 查找 Write 或 Edit 工具调用，且目标路径在 memory 目录内
  for (const block of content) {
    const filePath = getWrittenFilePath(block)
    if (filePath !== undefined && isAutoMemPath(filePath)) {
      return true  // 主 Agent 已经写了记忆
    }
  }
  return false
}
```

```typescript
// 在 runExtraction 里
if (hasMemoryWritesSince(messages, lastMemoryMessageUuid)) {
  logForDebugging('[extractMemories] skipping — conversation already wrote to memory files')
  // 仅推进游标，不启动提取 Agent
  const lastMessage = messages.at(-1)
  if (lastMessage?.uuid) {
    lastMemoryMessageUuid = lastMessage.uuid
  }
  return
}
```

**这条互斥规则解决了什么问题？**

Claude Code 的 System Prompt 里有完整的记忆保存指南。如果用户在对话中明确告诉 Claude "记住这件事"，Claude 会直接用 Write 工具写记忆文件。这时候，后台提取 Agent 再跑一次会是多余的——它会分析同样的对话内容，可能写出重复的记忆，或者覆盖主 Agent 刚写的内容。

互斥规则让两条写入路径（主 Agent 主动写 vs 后台 Agent 自动提取）完全互斥，同一个时间段内只有一个路径会写记忆。

## runExtraction()：提取 Agent 的具体工作流程

```typescript
async function runExtraction({ context, appendSystemMessage, isTrailingRun }) {
  const { messages } = context
  const memoryDir = getAutoMemPath()
  const newMessageCount = countModelVisibleMessagesSince(messages, lastMemoryMessageUuid)

  // 1. 互斥检查：主 Agent 已写就跳过
  if (hasMemoryWritesSince(messages, lastMemoryMessageUuid)) { ... return }

  // 2. 节流：每 N 轮才提取一次（特性门控 tengu_bramble_lintel，默认 1）
  if (!isTrailingRun) {
    turnsSinceLastExtraction++
    if (turnsSinceLastExtraction < throttleN) { return }
  }
  turnsSinceLastExtraction = 0

  // 3. 预扫描现有记忆，生成 manifest（节省提取 Agent 的一轮 ls）
  const existingMemories = formatMemoryManifest(
    await scanMemoryFiles(memoryDir, createAbortController().signal),
  )

  // 4. 构建提取提示词（包含 newMessageCount + 现有记忆 manifest）
  const userPrompt = buildExtractAutoOnlyPrompt(
    newMessageCount,
    existingMemories,
    skipIndex,
  )

  // 5. 以 fork 模式运行提取 Agent（共享主 Agent 的 prompt cache）
  const result = await runForkedAgent({
    promptMessages: [createUserMessage({ content: userPrompt })],
    cacheSafeParams,
    canUseTool,
    querySource: 'extract_memories',
    forkLabel: 'extract_memories',
    skipTranscript: true,    // 不记录到会话转录
    maxTurns: 5,             // 最多 5 轮：read → write
  })

  // 6. 推进游标
  const lastMessage = messages.at(-1)
  if (lastMessage?.uuid) {
    lastMemoryMessageUuid = lastMessage.uuid
  }

  // 7. 提取写入的文件路径，通知用户
  const writtenPaths = extractWrittenPaths(result.messages)
  const memoryPaths = writtenPaths.filter(p => basename(p) !== ENTRYPOINT_NAME)
  if (memoryPaths.length > 0) {
    appendSystemMessage?.(createMemorySavedMessage(memoryPaths))
  }
}
```

几个关键设计：

**预注入现有记忆 manifest**：在调用提取 Agent 之前，先用 `scanMemoryFiles()` 扫描现有记忆，把 manifest 注入到提取提示词里。否则提取 Agent 的第一步会是 `ls` 或 `Glob` 来查看现有记忆，浪费一轮对话。注释里说：`Pre-inject the memory directory manifest so the agent doesn't spend a turn on ls.`

**maxTurns: 5**：提取 Agent 的最大对话轮数限制为 5。注释说：`Well-behaved extractions complete in 2-4 turns (read → write). A hard cap prevents verification rabbit-holes from burning turns.` 2-4 轮是正常情况（读现有记忆 → 判断 → 写新文件），偶尔需要额外验证也在 5 轮内完成。这个上限防止了提取 Agent 陷入无限循环。

**skipTranscript: true**：提取 Agent 的对话不记录到会话转录。否则用户的 `/context` 命令里会充斥着提取 Agent 的内部工作消息，严重影响用户体验。

## createCacheSafeParams()：共享主 Agent 的 Prompt Cache

提取 Agent 通过 `runForkedAgent()` 运行，它的 `cacheSafeParams` 是通过 `createCacheSafeParams(context)` 创建的——这个函数（在第 14 章讲过）从全局 slot 读取上一次完整对话的 System Prompt 和上下文参数，确保提取 Agent 的 API 请求前缀与主 Agent 的字节完全一致，从而共享 Anthropic 的 prompt cache。

这意味着：**提取 Agent 不需要额外支付 System Prompt 的 token 费用**。整个大段的 System Prompt（几千 token）已经在主 Agent 的响应里缓存了，提取 Agent 只需要支付自己新增内容（提取提示词 + 最终回复）的 token 费用。

## createAutoMemCanUseTool()：严格的权限限制

提取 Agent 的权限被严格限制，[src/services/extractMemories/extractMemories.ts:171-222](../src/services/extractMemories/extractMemories.ts#L171-L222)：

```typescript
export function createAutoMemCanUseTool(memoryDir: string): CanUseToolFn {
  return async (tool, input) => {
    // 允许 REPL（在 REPL 模式下，原始工具被隐藏）
    if (tool.name === REPL_TOOL_NAME) {
      return { behavior: 'allow', updatedInput: input }
    }
    // 允许 Read/Grep/Glob（只读，安全）
    if (tool.name === FILE_READ_TOOL_NAME || tool.name === GREP_TOOL_NAME || tool.name === GLOB_TOOL_NAME) {
      return { behavior: 'allow', updatedInput: input }
    }
    // 只允许只读的 Bash 命令（ls, find, grep, cat, stat, wc, head, tail）
    if (tool.name === BASH_TOOL_NAME) {
      const parsed = tool.inputSchema.safeParse(input)
      if (parsed.success && tool.isReadOnly(parsed.data)) {
        return { behavior: 'allow', updatedInput: input }
      }
      return denyAutoMemTool(tool, 'Only read-only shell commands are permitted ...')
    }
    // 允许 Edit/Write，但只能写 memory 目录
    if ((tool.name === FILE_EDIT_TOOL_NAME || tool.name === FILE_WRITE_TOOL_NAME) &&
        typeof input.file_path === 'string' && isAutoMemPath(input.file_path)) {
      return { behavior: 'allow', updatedInput: input }
    }
    return denyAutoMemTool(tool, `only ... within ${memoryDir} are allowed`)
  }
}
```

这是一个最小权限原则的实现：提取 Agent 只能读文件、执行只读命令、向 memory 目录写文件。它不能向项目代码写文件，不能执行任何有副作用的命令，不能访问网络。这确保了即使提取 Agent 的推理出了问题，最坏结果也只是"写了一条错误的记忆"，而不是"删了代码"或"执行了危险命令"。

## drainPendingExtraction()：程序退出前的等待

```typescript
drainer = async (timeoutMs = 60_000) => {
  if (inFlightExtractions.size === 0) return
  await Promise.race([
    Promise.all(inFlightExtractions).catch(() => {}),
    new Promise<void>(r => setTimeout(r, timeoutMs).unref()),
  ])
}
```

当用户退出 Claude Code 时，如果有正在进行的提取，`drainPendingExtraction()` 会等待最多 60 秒。`setTimeout(...).unref()` 意味着这个定时器不会阻止进程退出——如果超时触发，Node.js 进程会正常退出，即使提取还没完成。

在 `print.ts`（非交互模式）里，会在响应刷出之后调用这个函数：
```
响应输出完毕 → drainPendingExtraction() → 等待提取 → 进程退出
```

注释说：`Called by print.ts after the response is flushed but before gracefulShutdownSync, so the forked agent completes before the 5s shutdown failsafe kills it.`

## 节流机制：tengu_bramble_lintel

```typescript
turnsSinceLastExtraction++
if (
  turnsSinceLastExtraction <
  (getFeatureValue_CACHED_MAY_BE_STALE('tengu_bramble_lintel', null) ?? 1)
) {
  return  // 不到 N 轮，跳过本次提取
}
turnsSinceLastExtraction = 0
```

默认情况下（feature value 为 null，退回默认 1），每轮都提取（`turnsSinceLastExtraction < 1` 在第一次时不成立）。通过特性门控可以把这个值调到 2、3 等，实现每 N 轮才提取一次，减少 API 调用频率。

注意 trailing run（被暂存的上下文触发的二次提取）跳过节流检查：`if (!isTrailingRun)` 包裹了节流逻辑。因为 trailing run 本身是对已完成但未处理工作的补偿，不应该被再次延迟。

## 本章小结

自动记忆提取系统的核心设计：
1. **触发时机**：每次 query loop 结束时 fire-and-forget，不阻塞主线程
2. **游标机制**：`lastMemoryMessageUuid` 确保每次只处理增量新消息
3. **互斥控制**：`inProgress` 互斥锁 + `pendingContext` trailing run
4. **与主 Agent 互斥**：`hasMemoryWritesSince()` 防止双重写入
5. **最小权限**：`createAutoMemCanUseTool()` 限定只能读取 + 向 memory 目录写入
6. **Prompt Cache 共享**：`createCacheSafeParams()` 让提取 Agent 使用主 Agent 的缓存
7. **闭包隔离状态**：`initExtractMemories()` 每次返回新闭包，方便测试隔离
