# 9-8 循环的终止条件

## 本节要解决的问题

`queryLoop()` 是一个 `while(true)` 循环——它必须有退出条件，否则永远运行下去。

这些退出条件分散在代码各处，而且每个退出原因都有不同的语义。本节系统地整理所有终止路径。

**文件：[src/query.ts](../src/query.ts#L241-L1729)**

---

## Terminal 类型：退出原因的类型化

**文件：[src/query/transitions.ts](../src/query/transitions.ts)**（推断）

`queryLoop()` 的返回类型是 `Terminal`，包含 `reason` 字段：

```typescript
type Terminal =
  | { reason: 'completed' }
  | { reason: 'max_turns', turnCount: number }
  | { reason: 'blocking_limit' }
  | { reason: 'model_error', error: unknown }
  | { reason: 'image_error' }
  | { reason: 'prompt_too_long' }
  | { reason: 'aborted_streaming' }
  | { reason: 'aborted_tools' }
  | { reason: 'stop_hook_prevented' }
  | { reason: 'hook_stopped' }
```

## 终止路径分类

### 1. 正常完成：completed

**文件：[src/query.ts](../src/query.ts#L1062-L1357)**

```typescript
if (!needsFollowUp) {
  // 没有工具调用 → 进入终止检查
  
  // ... 错误恢复路径 ...
  
  const stopHookResult = yield* handleStopHooks(...)
  if (stopHookResult.preventContinuation) {
    return { reason: 'stop_hook_prevented' }
  }
  if (stopHookResult.blockingErrors.length > 0) {
    state = next; continue  // stop hook 要求重试
  }
  
  return { reason: 'completed' }  // ← 正常终止
}
```

条件：Claude 的响应不包含任何 `tool_use` block，且 stop hooks 没有拒绝这个响应。

这是最常见的终止原因——Claude 说完了，没有工具要执行，对话结束。

---

### 2. 达到最大轮次：max_turns

**文件：[src/query.ts](../src/query.ts#L1704-L1712)**

```typescript
if (maxTurns && nextTurnCount > maxTurns) {
  yield createAttachmentMessage({
    type: 'max_turns_reached',
    maxTurns,
    turnCount: nextTurnCount,
  })
  return { reason: 'max_turns', turnCount: nextTurnCount }
}
```

条件：`maxTurns` 参数被设置（SDK 模式），且工具执行轮次超过限制。

SDK 调用者通过 `maxTurns` 防止 Claude 无限循环。达到上限时，循环立即中断，即使还有工具要执行。

---

### 3. 上下文过长：blocking_limit

**文件：[src/query.ts](../src/query.ts#L636-L647)**

```typescript
const { isAtBlockingLimit } = calculateTokenWarningState(
  tokenCountWithEstimation(messagesForQuery),
  toolUseContext.options.mainLoopModel,
)
if (isAtBlockingLimit) {
  yield createAssistantAPIErrorMessage({
    content: PROMPT_TOO_LONG_ERROR_MESSAGE,
    error: 'invalid_request',
  })
  return { reason: 'blocking_limit' }
}
```

条件：消息历史已接近模型上下文窗口限制，且自动压缩（auto-compact）未启用。

这是**预防性终止**——在发出 API 请求之前就检测到"快撑满了"，主动停止而不是等 API 返回 413 错误。如果 auto-compact 开启，这个检查被跳过（compact 会自动压缩历史）。

---

### 4. 模型错误：model_error

**文件：[src/query.ts](../src/query.ts#L955-L996)**

```typescript
} catch (error) {
  yield* yieldMissingToolResultBlocks(assistantMessages, errorMessage)
  yield createAssistantAPIErrorMessage({ content: errorMessage })
  return { reason: 'model_error', error }
}
```

条件：`callModel()` 抛出了未被 `withRetry` 处理的错误。

注意：`withRetry` 已经处理了 529（过载）和认证错误。走到 `catch` 的是真正的、无法重试的错误：网络故障、格式错误的请求等。

---

### 5. 图片错误：image_error

```typescript
if (error instanceof ImageSizeError || error instanceof ImageResizeError) {
  yield createAssistantAPIErrorMessage({ content: error.message })
  return { reason: 'image_error' }
}
```

条件：图片超出 API 限制或无法 resize。单独成一类，因为这不是模型错误，而是输入错误。

---

### 6. Context 太长（prompt-too-long）：prompt_too_long

**文件：[src/query.ts](../src/query.ts#L1168-L1182)**

```typescript
if ((isWithheld413 || isWithheldMedia) && reactiveCompact) {
  const compacted = await reactiveCompact.tryReactiveCompact(...)
  if (compacted) {
    // 压缩成功 → 继续（state = next; continue）
  }
  // 压缩失败 → yield 错误并退出
  yield lastMessage
  return { reason: 'prompt_too_long' }
}
```

条件：API 返回了 413/prompt-too-long 错误，reactive compact 尝试压缩但失败了（比如消息历史太小，压缩没有效果）。

---

### 7. 用户中断（流式阶段）：aborted_streaming

**文件：[src/query.ts](../src/query.ts#L1011-L1052)**

```typescript
if (toolUseContext.abortController.signal.aborted) {
  if (streamingToolExecutor) {
    // 生成中断的工具的占位结果
    for await (const update of streamingToolExecutor.getRemainingResults()) { ... }
  } else {
    yield* yieldMissingToolResultBlocks(assistantMessages, 'Interrupted by user')
  }
  
  if (abortController.signal.reason !== 'interrupt') {
    yield createUserInterruptionMessage({ toolUse: false })
  }
  return { reason: 'aborted_streaming' }
}
```

条件：用户在 Claude 流式输出期间按了 Ctrl+C（或 SDK 调用者调用了 `abortController.abort()`）。

**为什么还要生成 `yieldMissingToolResultBlocks`？**

如果 Claude 已经发出了 `tool_use` block（但 streaming 被中断，工具还没执行），消息历史里会有一个"孤儿" `tool_use`。下次重试或 `--resume` 时，历史里的 `tool_use` 没有对应的 `tool_result`，API 会报错。

`yieldMissingToolResultBlocks` 为每个未完成的 `tool_use` 生成一个错误 `tool_result`，保持消息历史的 `tool_use/tool_result` 配对完整。

**注意 `reason !== 'interrupt'`**：这是"submit-interrupt"的特殊逻辑。当用户输入新消息并提交（interrupt），系统不显示"interrupted by user"消息，因为新消息本身就是上下文。只有按 Ctrl+C（reason 不是 'interrupt'）才显示中断提示。

---

### 8. 用户中断（工具执行阶段）：aborted_tools

**文件：[src/query.ts](../src/query.ts#L1484-L1515)**

```typescript
if (toolUseContext.abortController.signal.aborted) {
  // ...Computer Use 清理...
  if (abortController.signal.reason !== 'interrupt') {
    yield createUserInterruptionMessage({ toolUse: true })
  }
  return { reason: 'aborted_tools' }
}
```

条件：用户在工具执行期间（`runTools` 之后）中断。

与 `aborted_streaming` 的区别：此时工具已经有了结果（可能部分完成），不需要补 missing tool results。

---

### 9. Stop Hook 阻止：stop_hook_prevented

**文件：[src/query.ts](../src/query.ts#L1278-L1280)**

```typescript
if (stopHookResult.preventContinuation) {
  return { reason: 'stop_hook_prevented' }
}
```

条件：用户配置了 PostToolUse hook，hook 脚本返回了"阻止继续"的信号。

这允许用户通过脚本实现自定义的安全检查——如果 Claude 执行了某个不期望的操作，hook 可以阻止 Claude 继续行动。

---

### 10. Hook 停止（PreToolUse）：hook_stopped

**文件：[src/query.ts](../src/query.ts#L1519-L1521)**

```typescript
if (shouldPreventContinuation) {
  return { reason: 'hook_stopped' }
}
```

条件：工具执行结果中有一个 `hook_stopped_continuation` 类型的附件消息（PreToolUse hook 返回了 `stop` 指令）。

## 终止路径总览

```
queryLoop() 的退出路径

"正常退出"
    completed                ← 最常见：Claude 说完了，没有工具调用
    max_turns                ← SDK 限制轮次
    stop_hook_prevented      ← PostToolUse hook 阻止
    hook_stopped             ← PreToolUse hook 停止

"错误退出"
    blocking_limit           ← 预防性：上下文快满了
    model_error              ← API/网络错误
    image_error              ← 图片限制
    prompt_too_long          ← API 413，自动压缩失败

"用户干预"
    aborted_streaming        ← Ctrl+C（流式阶段）
    aborted_tools            ← Ctrl+C（工具执行阶段）
```

---

## 退出后发生什么

`queryLoop()` 的 `Terminal` 被传给 `query()` 函数，然后传给 `submitMessage()` 的 `for await` 循环。`submitMessage()` 在循环结束后构建最终的 `result` 消息：

```typescript
yield {
  type: 'result',
  subtype: 'success',
  stop_reason: terminal.reason,
  ...
}
```

SDK 调用者通过 `stop_reason` 知道为什么停止，可以做出相应处理（比如 `max_turns` → 提示用户；`model_error` → 显示错误）。

---

## 本节小结

- `queryLoop()` 有 10 种退出原因，分为"正常完成"、"错误退出"和"用户干预"三类
- 最常见的是 `completed`：`!needsFollowUp` 且 stop hooks 通过
- 用户中断（aborted）有两个时机：流式阶段和工具执行阶段，处理方式略有不同
- 中断时需要补全 `tool_use/tool_result` 配对，保证消息历史的合法性
- 退出原因以 `stop_reason` 字段暴露给 SDK 调用者
