# 17-3 MCP 工具转换

## 问题：Claude 看到的工具和 MCP 服务器给的工具是同一个东西吗？

当 MCP 服务器连接成功后，Claude Code 调用 `tools/list` RPC 获取工具列表。服务器返回的是 MCP 格式的工具定义。但 Claude Code 内部有自己的 `Tool` 接口——权限检查、执行、结果格式化都基于这个接口。

两者不是同一件事，中间需要一个转换层。

这节我们详细看这个转换：工具名怎么规范化、inputSchema 怎么处理、call() 怎么委托给 MCP 服务器、权限检查怎么接入。

核心文件：[src/services/mcp/client.ts](../src/services/mcp/client.ts#L1739) 的 `fetchToolsForClient()` 函数。

---

## fetchToolsForClient()：带 LRU 缓存的工具列表获取

**文件：[src/services/mcp/client.ts](../src/services/mcp/client.ts#L1739)**

```typescript
export const fetchToolsForClient = memoizeLRU(
  async (
    client: ConnectedMCPServer,
    ...
  ): Promise<Tool[]> => {
    // 1. 调用 MCP RPC 获取工具列表
    const { tools: mcpTools } = await client.client.listTools()

    // 2. 将每个 MCP 工具转换为内部 Tool 对象
    return mcpTools.map(mcpTool => convertMcpToolToTool(mcpTool, client))
  },
  20,                          // LRU 容量：最多缓存 20 个不同 client 的工具列表
  client => client.name,       // 缓存 key：用 client 的名字
)
```

这里用的是 LRU（Least Recently Used）缓存，容量 20。

为什么是 LRU 而不是普通 memoize？

- 普通 memoize 永不淘汰——如果用户在一次会话里接入了 100 个 MCP 服务器（不现实，但可能），内存会一直增长
- LRU-20 意味着：如果同时活跃的 MCP 服务器超过 20 个，最久没用的那个工具列表会被释放
- 实践中，用户很少同时使用超过 20 个 MCP 服务器，LRU-20 基本等同于无限缓存

## 工具名规范化：buildMcpToolName()

MCP 服务器里工具叫什么名字不重要，因为 Claude Code 会给它重新命名。

**文件：[src/services/mcp/mcpStringUtils.ts](../src/services/mcp/mcpStringUtils.ts#L1)**

```typescript
export function buildMcpToolName(serverName: string, toolName: string): string {
  return `${getMcpPrefix(serverName)}${normalizeNameForMCP(toolName)}`
}

function getMcpPrefix(serverName: string): string {
  return `mcp__${normalizeNameForMCP(serverName)}__`
}

function normalizeNameForMCP(name: string): string {
  // 把所有非字母数字下划线的字符替换为下划线
  return name.replace(/[^a-zA-Z0-9_]/g, '_')
}
```

举例：
- 服务器名 `"my-weather-server"`，工具名 `"get-weather"`
- 规范化后服务器名：`"my_weather_server"`
- 规范化后工具名：`"get_weather"`
- 最终工具名：`"mcp__my_weather_server__get_weather"`

这个命名约定的作用：

1. **防止冲突**：内置工具（Bash、Read……）不带 `mcp__` 前缀，永远不会与 MCP 工具同名
2. **便于识别**：看到 `mcp__` 前缀就知道这是 MCP 工具，知道它来自哪个服务器
3. **权限规则匹配**：权限规则可以用 `mcp__my_server__*` 匹配某个服务器的所有工具

`normalizeNameForMCP` 把连字符、空格、点等字符全换成下划线，因为 Claude 的工具名规范只允许字母数字和下划线。

## mcpInfoFromString()：从工具名解析回服务器和工具

有时需要从规范化后的工具名反推原始信息：

**文件：[src/services/mcp/mcpStringUtils.ts](../src/services/mcp/mcpStringUtils.ts#L1)**

```typescript
export function mcpInfoFromString(toolName: string): {
  serverName: string
  toolName: string
} | null {
  if (!toolName.startsWith('mcp__')) return null

  // 按 __ 分割：['mcp', serverName, ...toolNameParts]
  const parts = toolName.split('__')
  if (parts.length < 3) return null

  return {
    serverName: parts[1],
    // 工具名可能包含 __，所以从第三段开始 join
    toolName: parts.slice(2).join('__'),
  }
}
```

注意这个已知 limitation：**如果服务器名本身包含 `__`，解析会出错**。源码注释里承认了这个问题。这是一个为了简单性做出的取舍。

## MCPTool：工具转换的基础模板

在看转换逻辑之前，先看一个辅助类：

**文件：[src/tools/MCPTool/MCPTool.ts](../src/tools/MCPTool/MCPTool.ts#L1)**

```typescript
export class MCPTool implements Tool {
  // inputSchema 是 passthrough——接受任意结构的输入
  // 因为每个 MCP 工具都有自己的 schema，这里只是占位
  inputSchema = z.object({}).passthrough()

  // 结果最大 10 万字符
  maxResultSizeChars = 100_000

  // 以下方法都是存根，在 fetchToolsForClient 里被覆盖
  name = ''
  description = ''

  async call(_input: unknown): Promise<ToolResult> {
    throw new Error('MCPTool.call() should be overridden')
  }

  async checkPermissions(_input: unknown): Promise<PermissionResult> {
    throw new Error('MCPTool.checkPermissions() should be overridden')
  }
}
```

`z.object({}).passthrough()` 是关键设计：`.passthrough()` 允许对象包含 schema 未声明的字段。因为每个 MCP 工具的输入 schema 都不同，在 MCPTool 这个基类里无法预知，所以用 passthrough 接受任意输入，再把运行时校验交给 MCP 服务器本身。

## convertMcpToolToTool()：核心转换逻辑

**文件：[src/services/mcp/client.ts](../src/services/mcp/client.ts#L1762)**

```typescript
function convertMcpToolToTool(
  mcpTool: MCPTool_SDK,           // MCP SDK 的工具类型
  server: ConnectedMCPServer,
): Tool {
  // 构建规范化的工具名
  const toolName = buildMcpToolName(server.name, mcpTool.name)

  // 截断过长的描述
  const description = mcpTool.description?.length > MAX_MCP_DESCRIPTION_LENGTH
    ? mcpTool.description.slice(0, MAX_MCP_DESCRIPTION_LENGTH) + '...'
    : mcpTool.description ?? ''

  return {
    // 基础属性：从 MCP 工具继承，覆盖名字和描述
    ...new MCPTool(),
    name: toolName,
    description,

    // inputSchema：把 MCP 的 JSON Schema 转换为 Zod schema
    // 使用 zodToJsonSchema 的逆过程（或保持为 JSON Schema 透传）
    inputSchema: mcpToolInputSchemaToZod(mcpTool.inputSchema),

    // 注解：来自 MCP 工具的 annotations（readOnly、destructive 等）
    annotations: mcpTool.annotations,

    // 覆盖 call()：委托给 MCP 服务器执行
    async call(input: unknown): Promise<ToolResult> {
      return callMcpTool(server, mcpTool.name, input)
    },

    // 覆盖 checkPermissions()：走内部权限系统
    async checkPermissions(input: unknown): Promise<PermissionResult> {
      return checkMcpToolPermissions(toolName, input)
    },
  }
}
```

转换做了四件事：
1. **名字规范化**：`get-weather` → `mcp__my_server__get_weather`
2. **描述截断**：超过 2048 字符的描述截断
3. **call() 代理**：调用时委托给 `callMcpTool()` 走 MCP RPC
4. **权限检查接入**：`checkPermissions()` 使用内部权限系统，不是 MCP 自己的

## call() 的执行路径：ensureConnected + 重试

**文件：[src/services/mcp/client.ts](../src/services/mcp/client.ts#L1829)**

```typescript
async function callMcpTool(
  server: ConnectedMCPServer,
  toolName: string,
  input: unknown,
): Promise<ToolResult> {
  // 第一步：确保连接仍然有效（MCP 连接可能已断开）
  const client = await ensureConnectedClient(server)

  // 第二步：调用 MCP 工具，带重试逻辑
  return callMCPToolWithUrlElicitationRetry(client, toolName, input)
}
```

**ensureConnectedClient()**：检查 server 的连接状态，如果已断开则尝试重连。这处理了这种情况：用户在工具调用中间 MCP 服务器崩溃了，Claude Code 自动重连而不是直接报错。

**callMCPToolWithUrlElicitationRetry()**：这个名字很长，但含义清晰——带 URL elicitation 重试的工具调用。

```typescript
async function callMCPToolWithUrlElicitationRetry(
  client: Client,
  toolName: string,
  input: unknown,
): Promise<ToolResult> {
  try {
    const result = await client.callTool({
      name: toolName,
      arguments: input as Record<string, unknown>,
    })
    return formatMcpResult(result)
  } catch (error) {
    // 如果是会话过期错误（McpSessionExpiredError），重连后重试一次
    if (error instanceof McpSessionExpiredError) {
      await reconnectServer(client)
      const result = await client.callTool({ name: toolName, arguments: input as Record<string, unknown> })
      return formatMcpResult(result)
    }
    throw error
  }
}
```

`McpSessionExpiredError` 是 SSE/WebSocket 连接中常见的情况：HTTP 长连接有空闲超时，服务器可能在 Claude 思考过程中悄悄关闭了连接。重连后重试一次，对用户透明。

## checkPermissions()：工具名决定权限规则

**文件：[src/services/mcp/mcpStringUtils.ts](../src/services/mcp/mcpStringUtils.ts#L1)**

```typescript
export function getToolNameForPermissionCheck(tool: Tool): string {
  // MCP 工具使用完整规范化名（mcp__server__tool）做权限检查
  // 这防止了规则如 "deny: bash" 错误匹配到 MCP 工具名里含 "bash" 的情况
  if (isMcpTool(tool)) {
    return tool.name  // 已经是规范化的全名
  }
  return tool.name
}
```

权限规则（allow/deny）匹配时，MCP 工具用完整的 `mcp__server__tool` 名字，这意味着：

- 规则 `deny: bash` 不会误匹配 `mcp__myserver__bash_exec`
- 可以写规则 `allow: mcp__trusted_server__*` 放行某个服务器的所有工具
- 可以写规则 `deny: mcp__dangerous_server__*` 禁止某个服务器的所有工具

这种设计让权限规则对内置工具和 MCP 工具都有统一语义，不需要两套规则体系。

## 工具结果格式化

MCP 工具调用返回的结果需要转换为 Claude Code 内部的 ToolResult 格式：

**文件：[src/services/mcp/client.ts](../src/services/mcp/client.ts#L1)**

```typescript
function formatMcpResult(mcpResult: CallToolResult): ToolResult {
  // MCP 结果可能包含 text、image、resource 等多种内容块
  const content = mcpResult.content.map(block => {
    if (block.type === 'text') {
      return { type: 'text', text: block.text }
    } else if (block.type === 'image') {
      return {
        type: 'image',
        source: { type: 'base64', media_type: block.mimeType, data: block.data },
      }
    }
    // resource 等其他类型...
    return { type: 'text', text: JSON.stringify(block) }
  })

  return {
    content,
    isError: mcpResult.isError ?? false,
  }
}
```

MCP 工具可以返回图片（比如截图工具）、文本或资源引用。这些都会被规范化为 Claude 的 content block 格式，最终作为 `tool_result` 消息发回给 Claude。

## 整个 MCP 工具系统的调用链

```
MCP 服务器连接建立
        │
        ▼
fetchToolsForClient(connectedServer)    ← client.ts#L1739, LRU-20 缓存
        │ 调用 client.listTools() RPC
        ▼
[{ name: "get_weather", description: "...", inputSchema: {...} }, ...]
        │ 每个工具经过 convertMcpToolToTool()
        ▼
[{ name: "mcp__my_server__get_weather", call: () => ..., checkPermissions: () => ... }, ...]
        │
        ▼
工具注册到 assembleToolPool()           ← 与内置工具合并，Claude 可以使用
        │
Claude 决定调用 mcp__my_server__get_weather
        │
        ▼
tool.call({ city: "Beijing" })
        │ → callMcpTool()
        │ → ensureConnectedClient() 检查连接
        │ → client.callTool({ name: "get_weather", arguments: { city: "Beijing" } })
        │ → MCP RPC → 服务器执行
        │ → 结果格式化
        ▼
ToolResult → 作为 tool_result 消息发给 Claude
```

## 本章小结

MCP 工具转换的核心设计：

1. **工具名规范化**：`mcp__server__tool` 格式，防冲突、易识别
2. **LRU-20 缓存**：避免每次 session 都重新列举工具
3. **call() 代理**：转换后的工具调用委托给 MCP RPC，透明
4. **MCPSessionExpiredError 重试**：连接超时自动重连，用户无感
5. **权限检查接入**：使用同一套权限系统，不需要 MCP 专用规则
