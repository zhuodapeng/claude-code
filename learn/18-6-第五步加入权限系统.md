# 18-6 第五步：加入权限系统

## 目标

在执行危险工具（bash 命令、写文件）之前，向用户确认。让 LLM 不能在没有授权的情况下执行破坏性操作。

---

## 权限系统的核心问题

简单的想法：在每次工具调用前都问用户"要不要执行？"

但这会导致频繁打断——读一个文件也要问，太烦了。

更好的设计：**按风险级别分类工具**：

- **只读工具**（read_file、list_dir）：自动批准，不打断
- **危险工具**（bash、write_file）：每次都问
- **记住授权**：用户说"这次允许"，还是"总是允许"？

这对应 Claude Code 权限系统第 11 章讲的设计。我们实现一个简化版。

## 权限决策类型

```typescript
type PermissionDecision =

  | 'ask-user'         // 需要用户确认
  | 'auto-deny'        // 自动拒绝（黑名单操作）
```

```typescript
const DEFAULT_RULES: PermissionRule[] = [
```

## 权限检查的完整实现

```typescript
// permissions.ts

export type PermissionDecision = 'auto-approve' | 'ask-user' | 'auto-deny'

export interface PermissionResult {
  decision: PermissionDecision
  reason?: string
}

// 工具风险分类
const TOOL_RISK_MAP: Record<string, PermissionDecision> = {
  'read_file': 'auto-approve',
  'list_dir': 'auto-approve',
  'bash': 'ask-user',
  'write_file': 'ask-user',
  'delete_file': 'ask-user',
}

export function checkPermission(
  toolName: string,
  input: unknown,
): PermissionResult {
  const decision = TOOL_RISK_MAP[toolName] ?? 'ask-user'  // 未知工具默认要问

  // 对于 bash：检查命令内容，升级危险命令为 ask-user
  if (toolName === 'bash' && typeof input === 'object' && input !== null) {
    const command = (input as Record<string, string>).command ?? ''

    // 特别危险的命令模式
    const dangerousPatterns = [
      /rm\s+-rf?\s+\//, // rm -rf / 或类似
      /mkfs/,           // 格式化分区
      /dd\s+if=/,       // dd 命令
      /chmod\s+777/,    // 开放所有权限
    ]

    if (dangerousPatterns.some(p => p.test(command))) {
      return {
        decision: 'ask-user',
        reason: `⚠️  高危命令：${command.slice(0, 100)}`,
      }
    }
  }

  return { decision, reason: decision === 'ask-user' ? `执行 ${toolName}` : undefined }
}
```

## 在 Ink TUI 里展示权限对话框

这是最有趣的部分：工具执行变成了一个**需要等待用户响应的异步操作**。

我们需要：
1. 工具执行中，停下来展示确认对话框
2. 等用户按 y/n
3. 根据结果继续或跳过

关键实现：用 Promise 把用户的按键转换为 resolve/reject：

```typescript
// 在 App 组件里
const [pendingPermission, setPendingPermission] = useState<{
  toolName: string
  input: string
  resolve: (approved: boolean) => void
} | null>(null)

// 当需要权限时调用这个函数
const requestPermission = useCallback((toolName: string, input: unknown): Promise<boolean> => {
  return new Promise(resolve => {
    setPendingPermission({
      toolName,
      input: JSON.stringify(input),
      resolve,
    })
  })
}, [])

// 键盘处理：如果有待确认的权限请求，优先处理
useInput((char, key) => {
  if (pendingPermission) {
    if (char === 'y' || char === 'Y') {
      pendingPermission.resolve(true)
      setPendingPermission(null)
    } else if (char === 'n' || char === 'N') {
      pendingPermission.resolve(false)
      setPendingPermission(null)
    }
    return
  }

  // 普通输入处理...
})
```

权限对话框组件：

```typescript
// 渲染权限对话框
{pendingPermission && (
  <Box
    borderStyle="double"
    borderColor="yellow"
    flexDirection="column"
    padding={1}
    marginTop={1}
  >
    <Text bold color="yellow">Permission Required</Text>
    <Text>Tool: <Text bold>{pendingPermission.toolName}</Text></Text>
    <Text>Input: {pendingPermission.input.slice(0, 200)}</Text>
    <Text></Text>
    <Text>Press <Text bold color="green">y</Text> to allow, <Text bold color="red">n</Text> to deny</Text>
  </Box>
)}
```

## 集成到工具执行流程

```typescript
// 在 runLoop() 里，工具执行前加权限检查
for (const block of toolUseBlocks) {
  const { decision, reason } = checkPermission(block.name, block.input)

  if (decision === 'auto-deny') {
    toolResults.push({
      type: 'tool_result',
      tool_use_id: block.id,
      content: `Permission denied: ${reason}`,
      is_error: true,
    })
    continue
  }

  if (decision === 'ask-user') {
    // 展示对话框，等待用户响应
    const approved = await requestPermission(block.name, block.input)

    if (!approved) {
      toolResults.push({
        type: 'tool_result',
        tool_use_id: block.id,
        content: 'User denied permission',
        is_error: true,
      })
      continue
    }
  }

  // auto-approve 或用户批准：执行工具
  const result = await executeTool(block.name, block.input as Record<string, unknown>)
  toolResults.push({ type: 'tool_result', tool_use_id: block.id, content: result })
}
```

## 与 Claude Code 的对比

Claude Code 的权限系统（第 11 章）比这复杂得多，但核心思想相同：

| 我们的实现 | Claude Code 对应 |
|-----------|----------------|
| `checkPermission()` | `src/permissions/checkPermissions.ts` 七步流水线 |
| `TOOL_RISK_MAP` | 权限规则配置（allow/deny/ask） |
| `requestPermission()` Promise | `Promise.race([userResponse, autoApproveTimeout])` |
| `useInput()` 处理 y/n | `PermissionDialog` 组件 |

Claude Code 的权限系统增加了：
- 规则持久化（记住"总是允许"的决策）
- 会话级规则 vs 全局规则
- 正则表达式模式匹配（`bash:/usr/bin/*`）
- bypassPermissions 模式（单元测试或 CI 环境）

## "记住我的选择"功能

Claude Code 里有"本次会话允许"和"总是允许"两种选项。我们的简化版只有"允许"和"拒绝"，但可以用一个 Set 来实现"本次会话记住"：

```typescript
// 本次会话记住的授权（简化版）
const sessionApprovals = new Set<string>()

function checkPermission(toolName: string, input: unknown): PermissionResult {
  // 如果本次会话已批准过这个工具，自动批准
  if (sessionApprovals.has(toolName)) {
    return { decision: 'auto-approve' }
  }
  // ...
}

// 用户批准时，可以选择记住
const approved = await requestPermission(block.name, block.input)
if (approved && userSaidRemember) {
  sessionApprovals.add(block.name)
}
```

## 本章小结

权限系统的核心设计：
1. **风险分类**：按工具类型预定义风险级别，只读操作自动批准
2. **Promise 化异步等待**：`requestPermission()` 返回 Promise，等用户按键 resolve
3. **UI 和逻辑解耦**：权限对话框只是一个 React 组件状态，业务逻辑通过 Promise 等待结果
