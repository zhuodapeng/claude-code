# 15-4 记忆的上限：截断策略的设计取舍

记忆系统是为了积累知识，但积累带来了容量问题：如果你用 Claude Code 几个月，记忆文件可能有几十个，MEMORY.md 可能增长到几百行。这些都不能无限制地塞进 System Prompt。这一节深入截断机制的具体实现，以及 AI 驱动的记忆召回设计。

---

## MEMORY.md 的两层截断

[src/memdir/memdir.ts:57-103](../src/memdir/memdir.ts#L57-L103) 里的 `truncateEntrypointContent()` 实现了一个两层截断机制：

```typescript
export const MAX_ENTRYPOINT_LINES = 200
export const MAX_ENTRYPOINT_BYTES = 25_000  // ~25KB

export function truncateEntrypointContent(raw: string): EntrypointTruncation {
  const trimmed = raw.trim()
  const contentLines = trimmed.split('\n')
  const lineCount = contentLines.length
  const byteCount = trimmed.length

  const wasLineTruncated = lineCount > MAX_ENTRYPOINT_LINES
  // 注意：字节数检查用的是截断前的原始大小
  // 目的是检测"长行"这个具体失败场景
  const wasByteTruncated = byteCount > MAX_ENTRYPOINT_BYTES

  if (!wasLineTruncated && !wasByteTruncated) {
    return { content: trimmed, lineCount, byteCount, wasLineTruncated, wasByteTruncated }
  }

  // 第一层：先按行截断（更自然的边界）
  let truncated = wasLineTruncated
    ? contentLines.slice(0, MAX_ENTRYPOINT_LINES).join('\n')
    : trimmed

  // 第二层：行截断后仍超过字节上限，再按字节截断
  // 在最后一个换行符处切断，不在行中间截断
  if (truncated.length > MAX_ENTRYPOINT_BYTES) {
    const cutAt = truncated.lastIndexOf('\n', MAX_ENTRYPOINT_BYTES)
    truncated = truncated.slice(0, cutAt > 0 ? cutAt : MAX_ENTRYPOINT_BYTES)
  }

  // 生成具体的警告消息，说明触发了哪个上限
  const reason =
    wasByteTruncated && !wasLineTruncated
      ? `${formatFileSize(byteCount)} (limit: ${formatFileSize(MAX_ENTRYPOINT_BYTES)}) — index entries are too long`
      : wasLineTruncated && !wasByteTruncated
        ? `${lineCount} lines (limit: ${MAX_ENTRYPOINT_LINES})`
        : `${lineCount} lines and ${formatFileSize(byteCount)}`

  return {
    content:
      truncated +
      `\n\n> WARNING: ${ENTRYPOINT_NAME} is ${reason}. Only part of it was loaded. Keep index entries to one line under ~200 chars; move detail into topic files.`,
    lineCount, byteCount, wasLineTruncated, wasByteTruncated,
  }
}
```

**为什么是两层，而不是只用字节限制？**

行截断更符合 Markdown 的语义边界——它不会在一个链接的中间截断（`[Some Very Long Title With Extra Description That Keeps Going](file.md) — a`），而字节截断即使有 `lastIndexOf('\n')` 兜底，也依赖 MEMORY.md 的格式足够规范。

**为什么字节数检查用截断前的原始大小，而不是截断后的大小？**

注释里说得很清楚：`post-line-truncation size would understate the warning`。字节上限的目标场景是"每条索引行太长了"（p97 的失败案例是 197KB 但只有 200 行以内——说明每行非常长）。如果先行截断再检查字节，可能得到"200 行但每行只有 100 字节"的结论，看不出"每行 1KB"这个真正的问题。

**截断警告的价值**：

截断时附带的警告不只是给用户看的，也是给 Claude 看的：
```
> WARNING: MEMORY.md is 47KB (limit: 25KB) — index entries are too long. Only part of it was loaded. 
> Keep index entries to one line under ~200 chars; move detail into topic files.
```

这个警告直接指导 Claude 如何修复：把内容从索引行移到主题文件里，保持索引行简洁。这是一个让 Claude 自我维护记忆质量的反馈机制。

## MAX_MEMORY_FILES：主题文件的数量上限

[src/memdir/memoryScan.ts:21-22](../src/memdir/memoryScan.ts#L21-L22)：

```typescript
const MAX_MEMORY_FILES = 200
const FRONTMATTER_MAX_LINES = 30
```

`scanMemoryFiles()` 返回的是按修改时间**降序排列**后截取的前 200 个文件。这意味着：
- 如果你有 300 个记忆文件，最旧的 100 个会被 `scanMemoryFiles()` 的结果忽略
- 这些被忽略的文件**不会参与 AI 召回**
- 隐含的假设：最旧的记忆往往也是最不相关的

200 这个数字本身是一个工程妥协——足够覆盖大多数用户的记忆量，但不会在每次召回时产生过多的 frontmatter 读取开销（即使是 FRONTMATTER_MAX_LINES=30 的优化，200 个文件也是 200 次 I/O）。

## findRelevantMemories()：用 AI 来召回相关记忆

MEMORY.md 索引只能覆盖 200 行，而且每次都全量加载。当用户有很多记忆主题文件时，大部分对当前对话是无关的。`findRelevantMemories()` 解决的是"从海量主题文件里找到当前对话真正需要的那几条"的问题。

[src/memdir/findRelevantMemories.ts:39-75](../src/memdir/findRelevantMemories.ts#L39-L75)：

```typescript
export async function findRelevantMemories(
  query: string,
  memoryDir: string,
  signal: AbortSignal,
  recentTools: readonly string[] = [],
  alreadySurfaced: ReadonlySet<string> = new Set(),
): Promise<RelevantMemory[]> {
  const memories = (await scanMemoryFiles(memoryDir, signal)).filter(
    m => !alreadySurfaced.has(m.filePath),
  )
  if (memories.length === 0) return []

  const selectedFilenames = await selectRelevantMemories(
    query,
    memories,
    signal,
    recentTools,
  )
  const byFilename = new Map(memories.map(m => [m.filename, m]))
  const selected = selectedFilenames
    .map(filename => byFilename.get(filename))
    .filter((m): m is MemoryHeader => m !== undefined)

  // 即使选出 0 条，也记录遥测（分母用于计算召回率）
  if (feature('MEMORY_SHAPE_TELEMETRY')) {
    logMemoryRecallShape(memories, selected)
  }

  return selected.map(m => ({ path: m.filePath, mtimeMs: m.mtimeMs }))
}
```

`alreadySurfaced` 是一个去重集合——如果同一条记忆在这次会话里已经被展示给 Claude 了，不要在 selector 的 5 个名额里再选它。这确保了每次召回都是新鲜的、增量的。

核心的 `selectRelevantMemories()` 函数调用了一个 Sonnet 模型：

```typescript
async function selectRelevantMemories(
  query: string,
  memories: MemoryHeader[],
  signal: AbortSignal,
  recentTools: readonly string[],
): Promise<string[]> {
  const manifest = formatMemoryManifest(memories)
  // 最近使用的工具列表，避免把"工具参考文档"误召回为相关
  const toolsSection =
    recentTools.length > 0
      ? `\n\nRecently used tools: ${recentTools.join(', ')}`
      : ''

  const result = await sideQuery({
    model: getDefaultSonnetModel(),
    system: SELECT_MEMORIES_SYSTEM_PROMPT,
    messages: [{
      role: 'user',
      content: `Query: ${query}\n\nAvailable memories:\n${manifest}${toolsSection}`,
    }],
    max_tokens: 256,
    output_format: {
      type: 'json_schema',
      schema: {
        type: 'object',
        properties: {
          selected_memories: { type: 'array', items: { type: 'string' } },
        },
        required: ['selected_memories'],
        additionalProperties: false,
      },
    },
    querySource: 'memdir_relevance',
  })
  // ...
}
```

Sonnet 收到的输入是：用户的 query + 所有记忆文件的 manifest（格式化的 frontmatter 列表），输出是应该加载的文件名列表（最多 5 个）。

System Prompt 里有一条重要的过滤规则：

```typescript
const SELECT_MEMORIES_SYSTEM_PROMPT = `...
- If a list of recently-used tools is provided, do not select memories that are usage 
  reference or API documentation for those tools (Claude Code is already exercising them). 
  DO still select memories containing warnings, gotchas, or known issues about those tools 
  — active use is exactly when those matter.
`
```

这条规则解决了一个真实观察到的误召回场景：当 Claude 正在使用某个 MCP 工具（比如 `mcp__database__query`），它的记忆里如果有这个工具的使用参考文档，keyword 相似性会导致 selector 把这个文档选进来——但这是噪音，Claude 正在用这个工具，不需要文档了。然而，如果记忆里有这个工具的已知 bug 或使用陷阱（gotcha），那恰恰是此时最需要知道的。所以规则是：过滤参考文档，但保留警告类记忆。

**`formatMemoryManifest()` 的输出格式**：

```typescript
// src/memdir/memoryScan.ts:84-94
export function formatMemoryManifest(memories: MemoryHeader[]): string {
  return memories
    .map(m => {
      const tag = m.type ? `[${m.type}] ` : ''
      const ts = new Date(m.mtimeMs).toISOString()
      return m.description
        ? `- ${tag}${m.filename} (${ts}): ${m.description}`
        : `- ${tag}${m.filename} (${ts})`
    })
    .join('\n')
}
```

格式化后的 manifest 看起来像：
```
- [user] user_role.md (2025-11-15T10:23:11.000Z): senior Go dev, new to React, prefers explicit types
- [feedback] feedback_testing.md (2025-12-01T08:00:00.000Z): no mock db — hit real db in integration tests
- [project] project_auth_rewrite.md (2025-12-10T15:30:00.000Z): auth rewrite driven by legal compliance
```

包含了 `[type]`、时间戳（供 selector 判断新鲜度）和 description。

## 记忆新鲜度警告：memoryFreshnessNote()

当主题文件被召回并展示给 Claude 时，如果这条记忆超过 1 天，会自动附加新鲜度警告：

```typescript
// src/memdir/memoryAge.ts:49-53
export function memoryFreshnessNote(mtimeMs: number): string {
  const text = memoryFreshnessText(mtimeMs)
  if (!text) return ''
  return `<system-reminder>${text}</system-reminder>\n`
}
```

被包裹在 `<system-reminder>` 标签里的文本：
```
This memory is 47 days old. Memories are point-in-time observations, not live state — 
claims about code behavior or file:line citations may be outdated. 
Verify against current code before asserting as fact.
```

为什么要用 "47 days ago" 而不是 "since 2025-11-15T..."？

注释给出了来自工程实践的观察：`Models are poor at date arithmetic — a raw ISO timestamp doesn't trigger staleness reasoning the way "47 days ago" does.`

这是一条经验性的发现：给 Claude 一个 ISO 时间戳，它通常不会主动推算"这是 47 天前的数据，可能已经过期了"；但给它"47 days ago"，它会触发"可能需要验证"的推理链。所以这里选择了对模型更友好的表达形式，而不是更机器友好的时间戳格式。

## 截断与召回的系统设计图

综合来看，记忆系统在容量管理上的整体设计：

```
磁盘上的记忆文件（可能有几百个）
        │
        ▼
scanMemoryFiles()
   → 按 mtime 降序排列
   → 最多 200 个文件
   → 每个文件只读前 30 行（frontmatter）
        │
        ├── formatMemoryManifest()
        │       → [type] filename (timestamp): description
        │       → 作为 Sonnet selector 的输入
        │
        └── Sonnet selectRelevantMemories()
                → 分析当前用户 query
                → 过滤已展示过的文件
                → 过滤当前活跃工具的参考文档
                → 选出最多 5 条相关记忆文件名
                │
                ▼
            实际读取这 5 个文件的完整内容
                │
                ▼
            附加 memoryFreshnessNote()（如果超过 1 天）
                │
                ▼
            注入到 Claude 的 user context（作为 attachment）

另一条路径：
MEMORY.md 索引（始终加载）
   → truncateEntrypointContent()
       → 最多 200 行 / 25KB
       → 超出时附加警告
   → 注入到 System Prompt 的一部分
```

这个设计的核心取舍是：**用一次 Sonnet 调用的开销（约 $0.000x）换来精准的记忆召回，避免把大量无关记忆塞进主 Claude 的上下文**。对于大量使用记忆的用户，这个开销是值得的。对于只有几条记忆的新用户，这个调用的成本也很低（manifest 很小，Sonnet 很快返回）。

---

## 设计取舍总结

| 机制 | 上限 | 超出后 | 原因 |
|------|------|--------|------|
| MEMORY.md 行数 | 200 行 | 截断 + 警告 | System Prompt 大小限制 |
| MEMORY.md 字节 | 25KB | 截断（保留到最后换行） | 防止长行 memory 绕过行数限制 |
| 记忆文件数量 | 200 个 | 忽略最旧的 | I/O 和 Sonnet 输入大小 |
| 每次召回 | 5 条主题文件 | 只加载最相关的 | 避免 context 膨胀 |
| 新鲜度警告 | > 1 天 | `<system-reminder>` 警告 | 防止过期信息被当事实断言 |

---

## 本章小结

记忆系统的上限设计有多个层次：
1. MEMORY.md 索引：200 行 / 25KB 双重截断，超出附带指导性警告
2. 主题文件扫描：最多 200 个，按 mtime 降序优先最新
3. AI 召回：Sonnet 根据当前 query 选出最多 5 条相关文件
4. 新鲜度：超过 1 天的召回记忆自动附加过期警告

核心哲学：用一次廉价的 Sonnet 调用做精准过滤，而不是把所有记忆一股脑塞进 Claude 的上下文。
